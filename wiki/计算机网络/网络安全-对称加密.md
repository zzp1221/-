## 核心定义

对称加密（Symmetric Encryption）是最基本的密码学技术，其核心特征是加密和解密使用相同的密钥（即共享密钥/秘密密钥）。发送方使用密钥K将明文P加密为密文C = E_K(P)，接收方使用同一个密钥K解密还原明文P = D_K(C)。对称加密算法分为两大流派：（1）分组密码（Block Cipher）——将明文分为固定长度的分组逐块加密，典型算法包括DES（64位分组，56位密钥，已不再安全）、3DES（三次DES加密的增强版）和AES（高级加密标准，128/192/256位密钥，128位分组，当前使用范围最广）；（2）流密码（Stream Cipher）——将密钥扩展为与明文等长的密钥流，逐位或逐字节加密，典型代表为RC4（因安全性问题已不推荐）和ChaCha20（在现代协议中得到广泛采用）。对称加密的高效率使其适用于大数据量的机密性保护（如TLS握手后使用会话密钥进行加速的对称加密传输）。对称加密面临的核心挑战是密钥分发问题：通信双方如何在安全的信道上预先共享秘密密钥。

## 关键结论

- AES（Advanced Encryption Standard）是当前最广泛使用的对称加密标准，支持128/192/256位密钥，使用SPN（替换-置换网络）结构，通过多轮（10/12/14轮）的字节替换、行移位、列混淆和轮密钥加操作实现高强度加密
- 分组密码的常见加密模式：ECB（电码本模式，每个分组独立加密，相同明文产生相同密文—安全性弱，不推荐）、CBC（密码分组链接模式，前一个密文分组与当前明文分组XOR后再加密，需要随机IV初始化向量）、CTR（计数器模式，将分组密码转为流密码，支持并行加解密和随机访问）、GCM（Galois/Counter Mode，加密和完整性认证一体化模式，TLS 1.3的主导选择）
- 对称加密的最主要优势是加密速度快（AES-NI硬件指令集加速下可达GB/s级吞吐量），适合对海量数据进行实时加密。主要痛点：密钥管理——对于n个通信方的网络，需要管理n(n-1)/2个共享密钥。这一缺陷由非对称加密的密码分发方案（如DH密钥交换+数字证书）来弥补
- 密钥长度与安全性的关系：虽然AES-256理论上比AES-128强，但在大多数实际场景中AES-128已足够安全（目前已知最佳攻击为Biclique攻击，复杂度约2^126.1，仍远高于实际可行）。AES-192/256主要用于对安全裕度有极高要求的政府/军事场景
- 对称加密与完整性保护的分离：AES本身只提供机密性，不保证完整性（密文在传输中可能被篡改）。需要与消息认证码（MAC/HMAC）结合使用，或直接使用认证加密模式（AEAD，如AES-GCM、ChaCha20-Poly1305）同时提供加密和完整性保护

## 易错点

1. **AES密钥长度不是分组大小**：AES分组大小固定128位（16字节），但密钥长度可以是128/192/256位。"AES-128"指的是128位密钥（而非128位分组）。不同密钥长度的加密轮数不同（分别为10/12/14轮）。

2. **ECB模式的安全缺陷不等于AES的缺陷**：许多人因为看到ECB模式下相同明文块产生相同的密文（企鹅图像轮廓透过加密可视）而误以为AES本身有问题。实际上ECB模式不安全，但AES算法本身是安全的——需要使用CBC/CTR/GCM等安全的操作模式。

3. **密钥≠口令**：对称加密的密钥是需要具有高熵的随机比特序列。用户输入的口令（password）通常长度短、熵低，不能直接用作加密密钥。需要用PBKDF2/bcrypt/scrypt/Argon2等密钥派生函数（KDF）将口令转化为高熵密钥。

4. **对称加密不解决"你是谁"的问题**：对称加密仅保护数据机密性，不提供身份认证（需要数字签名/证书）、不可抵赖性（数字签名）和完整性保护（MAC）。完整的安全方案需要结合多种密码学工具。

## 例题

**例题1**：AES-128-CBC加密模式下，第一个分组解密的计算公式是什么？CBC模式的IV为什么必须随机且不可预测？

**解答**：在CBC模式解密中，第i个密文分组C_i解密后的中间结果P'_i = AES_Decrypt(K, C_i)，最终明文P_i = P'_i XOR C_{i-1}（第一个分组时C_0 = IV）。解密公式：P_1 = AES_Decrypt(K, C_1) XOR IV。IV必须随机且不可预测的原因：假设攻击者知道下一个要加密的消息内容（如某个已知HTTP请求头），如果IV可预测，攻击者可以通过精心构造数据和猜测的明文进行选择明文攻击，推导出密钥信息或绕过加密保护。例如前向安全需要IV的不可预测性。

**例题2**：为什么TLS 1.3放弃了CBC模式而全面使用AEAD（AES-GCM和ChaCha20-Poly1305）？从安全性角度说明。

**解答思路**：CBC模式存在两大安全缺陷：（1）不提供完整性保护——攻击者可以篡改密文，通过CBC模式的密文比特翻转攻击来有目的地修改解密后的明文（padding oracle攻击利用服务器对不同类型错误的不同响应逐步解密整个报文——如Lucky13攻击）；（2）CBC加密是串行操作，无法并行加密。AEAD模式（AES-GCM、ChaCha20-Poly1305）同时提供加密和完整性认证，且GCM和ChaCha20都基于CTR模式支持并行加解密，性能更优。TLS 1.3移除了所有不安全/过时的加密算法，仅保留认证加密模式。

## 代码示例

```python
from cryptography.hazmat.primitives.ciphers import Cipher, algorithms, modes
from cryptography.hazmat.backends import default_backend
import os

# AES-GCM加密示例（认证加密模式）
def aes_gcm_encrypt(key, plaintext, associated_data=b''):
    """AES-GCM加密，同时提供机密性和完整性"""
    nonce = os.urandom(12)  # 96位随机Nonce
    encryptor = Cipher(
        algorithms.AES(key),
        modes.GCM(nonce),
        backend=default_backend()
    ).encryptor()
    encryptor.authenticate_additional_data(associated_data)
    ciphertext = encryptor.update(plaintext) + encryptor.finalize()
    return nonce + ciphertext + encryptor.tag  # 返回 nonce|密文|认证标签

def aes_gcm_decrypt(key, encrypted_data, associated_data=b''):
    """AES-GCM解密"""
    nonce = encrypted_data[:12]
    tag = encrypted_data[-16:]        # GCM Tag 128位
    ciphertext = encrypted_data[12:-16]
    decryptor = Cipher(
        algorithms.AES(key),
        modes.GCM(nonce, tag),
        backend=default_backend()
    ).decryptor()
    decryptor.authenticate_additional_data(associated_data)
    return decryptor.update(ciphertext) + decryptor.finalize()

# 生成随机AES密钥（256位）
aes_key = os.urandom(32)
print(f"AES-256密钥: {aes_key.hex()}")
```
