## 核心定义

停止-等待协议（Stop-and-Wait Protocol）是数据链路层最基础的流量控制和差错控制协议，属于自动重传请求（ARQ，Automatic Repeat reQuest）机制的一种。其工作原理非常简单：发送方每发送一个数据帧后就停止发送，等待接收方的确认帧（ACK）；接收方收到数据帧后进行CRC校验，如果正确则发送ACK确认，如果检测到错误则发送NAK否认或直接丢弃等待超时；发送方只有在收到ACK后才可以发送下一个数据帧，若超时未收到ACK则重传上一个数据帧。这种"发送一帧、停下等待、收到确认、再发下一帧"的交替方式实现了可靠传输，但由于信道的利用率极低，不适合高速长距离的通信场景。

## 关键结论

- 停止-等待协议的三种可靠性保证机制：（1）确认与超时重传——解决帧丢失问题；（2）帧序号（0和1交替）——解决重复帧问题；（3）CRC校验——解决帧损坏问题
- 使用1比特帧序号即可区分新旧帧：发送方发送编号为0的帧后等待ACK0，超时重传的仍是编号0的帧；接收方收到正确编号0的帧后发送ACK0，如果再次收到编号0的帧（重传），就丢弃并重发ACK0，这样避免了重复交付给上层
- 停止-等待协议的信道利用率U = (发送时延TD) / (发送时延TD + RTT + 确认帧发送时延TA)。当RTT远大于TD时（如卫星信道），利用率趋近于0
- 发送方的超时定时器设置是关键：太短会导致不必要的重传，太长会导致信道长时间空闲。一般设置为略大于正常情况下从发送完毕到收到确认的最长时间

## 易错点

1. **误认为ACK丢失和ACK延迟是同一个问题**：ACK丢失时发送方超时重传，接收方再次收到相同编号的帧后丢弃并重发ACK。如果发送方收到延迟到达的ACK（上一个帧的ACK迟到了），会错误地将其当作当前帧的ACK，此时1比特的帧序号恰好能解决这个问题——收到重复编号的ACK直接丢弃即可，因为已经处理过了。

2. **帧序号只需要1比特（0/1）**：初学者常质疑1比特是否足够。实际上停止-等待协议在任意时刻只有一个帧在传输中，帧序号的作用仅仅是让接收方区分"当前帧"和"重传帧"。0和1交替使用完全足够——发送方不会连续发送两个相同序号的不同帧，因为必须等上一个ACK才能发下一个。

3. **ACK本身不需要序号标注**：早期的停等协议中ACK不需要序号，但实际实现中ACK通常也携带所确认的帧序号（如ACK0/ACK1），这样可以明确告知发送方确认的是哪一帧，增强了协议的鲁棒性。

4. **超时重传后的帧可能不是上一帧**：在停等协议中，超时后重传一定是上一帧（因为没收到ACK不能发新帧）。但在滑动窗口协议（GBN/SR）中，超时重传的帧与"上一帧"的关系则更加复杂。

## 例题

**例题1**：在一个采用停止-等待ARQ的信道上，数据帧长度为1000字节，信道速率为1Mbps，RTT为20ms，确认帧长度为40字节。计算信道利用率和有效吞吐量。

**解答**：发送时延TD = (1000×8)/(1×10^6) = 8ms；确认帧发送时延TA = (40×8)/(1×10^6) = 0.32ms；信道利用率U = TD/(TD+RTT+TA) = 8/(8+20+0.32) = 8/28.32 ≈ 0.282 = 28.2%。有效吞吐量 = 1Mbps × 28.2% = 0.282Mbps。可见即使在局域网环境中（RTT=20ms较理想），停等协议也只能利用不到30%的信道容量。

**例题2**：在停止-等待ARQ中，如果发送方设置的超时时间小于RTT，会产生什么后果？如果大于RTT又会怎样？

**解答思路**：超时时间小于RTT时，发送方在ACK还在传输途中时就判定超时，触发不必要的重传。接收方会收到重复的帧（通过序号识别为重复），丢弃该帧并重发ACK。虽然不会导致数据错误，但浪费了信道资源。超时时间大于RTT时，前一个帧的ACK已经到达，但发送方不会立即发送新帧（因为定时器还在走），直到定时器超时才发现"超时"，造成信道空闲。理想情况下，超时定时器应设置为恰好略大于(RTT+TA+处理时间)。

## 代码示例

```python
import time
import random

class StopAndWaitSender:
    def __init__(self, timeout=0.5):
        self.seq_num = 0      # 当前帧序号（0/1交替）
        self.timeout = timeout
    
    def send(self, frame, channel):
        """发送一个帧并等待确认"""
        print(f"[SENDER] 发送帧#{self.seq_num}: {frame}")
        channel.transmit(frame, self.seq_num)
        start_time = time.time()
        
        while time.time() - start_time < self.timeout:
            ack = channel.receive_ack()
            if ack is not None:
                expected_ack = 1 - self.seq_num  # ACK序号应等于1-self.seq_num
                if ack == expected_ack:
                    print(f"[SENDER] 收到正确的ACK#{ack}")
                    self.seq_num = 1 - self.seq_num
                    return True
                else:
                    print(f"[SENDER] 收到重复ACK#{ack}，丢弃")
        
        print(f"[SENDER] 超时，重传帧#{self.seq_num}")
        return self.send(data, channel)  # 递归重传

# 说明：实际代码中会使用网络socket和select实现超时机制
# 此处为简化演示版本，重点在于协议逻辑
```
