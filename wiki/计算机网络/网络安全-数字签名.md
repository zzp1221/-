## 核心定义

数字签名（Digital Signature）是利用非对称密码学实现的消息认证技术，用于验证消息的完整性（消息在传输过程中未被篡改）和身份真实性（消息确实来自声称的发送者），并提供不可抵赖性（发送者不能事后否认曾经发送过该消息）。数字签名的基本流程为：发送方首先对原始消息运行密码学哈希函数（如SHA-256）生成消息摘要（Message Digest），然后使用自己的私钥对该摘要进行加密（或用私钥对摘要执行签名算法的Sign操作），生成数字签名。接收方收到消息和签名后，用发送方的公钥对签名进行解密/验证（Verify操作），得到摘要并重新计算消息的哈希值，两者比较——若匹配则验证通过：消息完整且确实由私钥持有者签署。数字签名的核心性质：（1）任何拥有发送方公钥的人都能验证签名；（2）只有私钥持有者才能生成有效签名；（3）签名与消息内容绑定——消息的任意微小改变都会导致签名验证失败。常见的数字签名算法包括基于RSA的签名（RSASSA-PKCS1-v1_5和RSASSA-PSS）、DSA（Digital Signature Algorithm）和ECDSA（椭圆曲线数字签名算法）。

## 关键结论

- 签名流程实际是先哈希再签名，不是直接对原始数据做签名运算：raw_signature = Sign(private_key, Hash(message))。这样做的原因是直接对原始数据签名有长度限制且计算开销大——哈希将任意长度的消息压缩为固定长度的摘要（如SHA-256输出256位），签名算法只需对固定长度的摘要进行操作
- ECDSA（椭圆曲线数字签名算法）是当前最推荐的非对称签名算法，基于椭圆曲线离散对数问题的困难性。相比RSA签名，ECDSA提供等效安全强度时密钥长度更短（如ECC P-256 ≈ 3072位RSA），且签名速度快、签名长度短。Ed25519（基于Curve25519椭圆曲线）是更新的高性能签名方案
- 数字签名的三大安全目标：（1）完整性——任何对消息的修改都会导致签名验证失败；（2）身份验证——只有对应私钥持有者才能生成合法签名；（3）不可抵赖性——因为有只有所有者知道的私钥的存在，送者无法否认签名有效消息是自己签署的
- 数字签名与消息认证码（MAC/HMAC）的区别：MAC使用共享对称密钥生成和验证，不能提供不可抵赖性（因为通信双方共享密钥，任何一方都可以生成相同的MAC）。数字签名使用非对称密钥对，私钥仅签名者持有，因此可以提供不可抵赖性
- 数字签名的典型应用：软件分发（发布者签名二进制安装包，用户验证发布者真实性）、TLS/HTTPS证书验证（CA用私钥签名证书，浏览器用CA公钥验证）、电子邮件（DKIM签名防伪造发件人）、区块链交易认证和电子合同/协议的不可抵赖性

## 易错点

1. **数字签名不等于将私钥加密和公钥解密的顺序反过来**：这种理解在RSA中确实近似正确（RSA的Sign = M^d mod n，Verify = S^e mod n），但ECDSA不是简单的"公钥解密验证签名"——ECDSA的签名和验证是完全不同的算法操作（无对等的"用公钥加密"等价操作）。不应将数字签名普遍理解为"用私钥加密后用公钥解密对比"。

2. **数字签名不提供机密性**：签名只保证消息来源的真实性、完整性和不可抵赖性，不能防止窃听。签名可以与加密结合使用（先签名再加密，或先加密再签名），但签名本身不提供保密功能。

3. **PSS比PKCS1v1.5更安全**：RSASSA-PSS（概率签名方案）在标准中比PKCS1-v1.5更推荐，因为它提供了在随机预言机模型下的可证明安全性。PKCS1-v1.5签名存在Bleichenbacher类型攻击风险（已在实践中被利用窃取TLS RSA密钥交换）。

4. **哈希算法替代问题**：数字签名的整体安全性依赖于其底层的哈希函数——如果SHA-1哈希被破坏（存在碰撞攻击，如SHAttered攻击），使用SHA-1的数字签名就失去了完整性保证。这就是为什么需要从SHA-1迁移到SHA-256/384/512。

## 例题

**例题1**：Alice向Bob发送消息M和签名S = Sign(Alice私钥, Hash(M))。Eve截获了M和S后，将M篡改为M'并发送给Bob。Bob用Alice公钥验证S和M'时会发生什么？Eve能否为M'创建新的签名？

**解答**：Bob用Alice公钥验证S时，会从S中恢复出原始哈希值H = Hash(M)。然后Bob对收到的M'计算Hash(M')。由于M' ≠ M，且哈希函数具有抗碰撞性（对不同的消息产生不同的哈希值），Hash(M') ≠ Hash(M)。验证失败，Bob得出结论：消息在传输中被篡改或签名与消息不匹配。Eve无法为M'创建新签名——她没有Alice的私钥，而只有私钥才能生成签名。公钥体制保证了伪造签名的计算不可行性。

**例题2**：论述ECDSA相比RSA数字签名的优势，并解释为什么现代TLS 1.3倾向于使用ECDSA证书。

**解答思路**：ECDSA优势：（1）密钥和签名长度更短（ECDSA P-256签名=512位，等强安全RSA签名=3072位）——更小的证书和握手数据传输量；（2）签名速度更快——ECDSA签名操作比RSA快数倍；（3）验证速度也优于RSA（使服务器能处理更多TLS连接）；（4）密钥生成速度快得多（即时生成vs RSA中找大素数的慢生成）。TLS 1.3倾向ECDSA的原因：缩短握手时间、减少带宽（移动和IoT设备）、被选为TLS 1.3的必要算法之一（RSA-PSS数字签名算法改进后也被支持但不如ECDSA流行）。

## 代码示例

```python
from cryptography.hazmat.primitives import hashes
from cryptography.hazmat.primitives.asymmetric import rsa, padding, ec, utils
from cryptography.exceptions import InvalidSignature

# RSA数字签名示例
private_key = rsa.generate_private_key(65537, 2048)
public_key = private_key.public_key()
message = b"Important document"

# 签名
signature = private_key.sign(
    message,
    padding.PSS(mgf=padding.MGF1(hashes.SHA256()),
                salt_length=padding.PSS.MAX_LENGTH),
    hashes.SHA256()
)

# 验证
try:
    public_key.verify(
        signature, message,
        padding.PSS(mgf=padding.MGF1(hashes.SHA256()),
                    salt_length=padding.PSS.MAX_LENGTH),
        hashes.SHA256()
    )
    print("RSA签名验证通过：消息完整且来源可信")
except InvalidSignature:
    print("验证失败：签名-消息不匹配")

# ECDSA数字签名示例
ec_private_key = ec.generate_private_key(ec.SECP256R1())
ec_public_key = ec_private_key.public_key()
ec_signature = ec_private_key.sign(message, ec.ECDSA(hashes.SHA256()))

try:
    ec_public_key.verify(ec_signature, message, ec.ECDSA(hashes.SHA256()))
    print("ECDSA签名验证通过")
except InvalidSignature:
    print("验证失败")
```
