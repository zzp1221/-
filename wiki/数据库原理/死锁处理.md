## 核心定义

死锁（Deadlock）是多个事务并发执行时，因每个事务都在等待对方释放已持有的锁，导致所有相关事务永远无法继续执行的现象。死锁的产生需要同时满足四个必要条件：互斥条件（资源必须互斥使用）、持有并等待（事务已持有至少一个资源，又在等待其他事务持有的资源）、不剥夺条件（资源只能由持有者自愿释放，不能被强制剥夺）和循环等待条件（存在一个事务的循环等待链，T1等T2、T2等T3……Tn等T1）。数据库处理死锁的策略分为三类：（1）死锁预防（Deadlock Prevention）——在事务启动前破坏循环等待（如一次性申请全部所需资源或定序申请资源）或通过抢占机制（事务间让权）破坏死锁条件；（2）死锁避免（Deadlock Avoidance）——系统在每次授予锁前判断是否可能导致死锁，如使用银行家算法确保始终存在安全序列可执行完毕；（3）死锁检测与恢复（Deadlock Detection & Recovery）——系统允许死锁发生，但周期性地（或当锁等待超时时）构建等待图（Wait-For Graph），通过检测有向环发现死锁，并回滚一个或多个事务来打破环状死锁关系。InnoDB使用死锁检测策略——通过后台检查锁等待有向图，当发现循环等待时选择一个代价最小的事务（undo log最少的）作为牺牲者回滚。

## 关键结论

- 死锁检测机制的实现：维护锁等待图（有向图——每个当事务是节点，有向边T_i→T_j表示T_i正在等待T_j持有的锁。算法用DFS搜索回环检测死锁。InnoDB在等待锁时运行检测——如果发现了死锁直接回滚请求锁的事务或等待链中的牺牲者
- 死锁超时机制（Timeout Lock机制）：设置每个锁等待的固定最长时限（如60秒）——超时后自动回滚超时的锁等待事务。这是一个简单但会导致长延迟经验的预防手段——在复杂事务中不推荐用超时作为唯一防死锁方式
- 死锁预防的非抢占方式——一次性申请全部锁：事务必须在开始执行前一次性锁定它将用到的所有数据。若不能获取全部锁则立即放弃——这可以打破"持有并等待"条件。但会浪费锁资源（加了很多最终不使用的数据锁）且无法预知动态SQL需要的全部锁
- 按序加锁（Ordered Locking）：所有事务按照全局一致的对象顺序进行锁定（如按主键ID递增顺序逐列锁定）来打破循环等待——因为这样一来两个事务不会相互等且永远只存在单向等待链（不会形成环）
- InnoDB死锁检测与报告：show engine innodb status 包含最近一次死锁信息（时间、事务SQL、持有的锁和正在等待的锁）。日志中出现deadlock时表示磁盘锁上的检测检测-日志中的LATEST DETECTED DEADLOCK帮助分析原因

## 易错点

1. **死锁检测的开销不是零的**：在高并发许多小事务争用共同资源——检测图需要检查有向环（对每个锁等待都要搜索路径），当并发非常高时死锁检测本身会消耗相当的CPU。InnoDB通过innodb_deadlock_detect参数可关闭死锁检测并将等待锁定控制权重交给超时控制。

2. **回滚事务不一定回滚死锁检测中断的事务**：InnoDB死锁检测会选择回滚一个事务（最小化工作量）——通常是持有最少行锁或最少UNDO日志的事务，不一定是发起最后锁请求的事务。检测到回滚后需应用程序捕获死锁错误并重试。

3. **"锁等超时会自动解锁"是常见误解**：lock_wait_timeout超时后系统将事务回滚并释放相关的锁——但这是一种强制中断解决方式，并不是自动友好解锁并让等的一方拿到锁。被中断会导致事务失败。

4. **死锁更多发生在交互应用中而非纯批处理**：因为交互应用中事务持续期更长、锁持有期更长更容易形成循环等待，而且交互的多用户随机路径增加无法预测冲突模式。批处理由于有序预定义加锁顺序死锁较少。

## 例题

**例题1**：分析银行转账事务死锁：T1转账A→B：UPDATE A...; UPDATE B... T2转账B→A：UPDATE B...; UPDATE A...。这两个转账过程同时运行出现什么后果？如何设计避免？

**解答**：T1锁A等B，T2锁B等A——死锁。防止方案：定义全局的锁顺序(SELECT ... FOR UPDATE ORDER BY id)按照账户ID升序锁定——两个事务都先锁ID较小的再锁较大的。如果A<B，T1先锁A→再锁B；T2也先锁A（已被T1锁定，T2等A释放）。T1锁A和B完成→提交→释放锁→T2获得A和B锁顺利进行。按统一顺序锁资源消除了循环等→无死锁。

**例题2**：如何阅读 InnoDB 死锁日志，举例说明等待图中每个事务的"持有锁"和"请求锁"如何对应检测循环？

**解答思路**：（1)MySQL死锁日志列出冲突的事务TRANSACTION 1和2；(2)每个TRANSACTION列出持有的锁记录数量（如2 row lock(s), undo log entries 1）；(3)标明正等待的锁类型: RECORD LOCKS space id n page no m index PRIMARY of table t; lock_mode X——表示在等待X锁获取的主键行。(4)等待图可视化——T1持PRIMARY INDEX on id=1的X锁，等待id=2的X锁；T2持id=2的X锁等id=1的X锁→形成T1→T2→T1回环。MySQL日志还会说明回滚哪个事务。
