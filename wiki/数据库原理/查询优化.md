## 核心定义

查询优化（Query Optimization）是DBMS中最核心的技术之一，目标是为给定的SQL查询找到执行效率最高（通常是估计代价最小）的执行计划。查询优化分为两大类型：基于规则的启发式优化（Heuristic Optimization）——使用关系代数的等价变换规则对查询树进行重写优化（不依赖于数据统计信息，仅基于逻辑等价规则转换）；基于代价的优化（Cost-Based Optimization, CBO）——利用数据字典的统计信息（表行数、列不同值基数、数据分布柱状图、物理存储页面数和索引层级等）估算每个候选执行计划的IO和CPU代价，选择代价最小的执行计划。现代DBMS主要采用CBO方式（如MySQL 8.0优化器、PostgreSQL优化器、Oracle优化器）。查询优化的核心操作包括：（1）谓词下推（Predicate Pushdown）——将WHERE过滤条件尽可能推向执行树的底层以减少上层操作的数据量；（2）连接顺序选择（Join Ordering）——确定多表连接的最优次序以减少中间结果大小；（3）选择最优的物理算子实现——对同一种逻辑操作选择合适的物理算法实现（如JOIN选择嵌套循环/哈希/排序合并、INDEX选择用哪个索引等）。

## 关键结论

- 关系代数等价变换规则是优化的逻辑基础：(a)选择的级联σ_F1(σ_F2(E)) = σ_F1∧F2(E)，允许将多个选择合并；(b)选择对投影的分配——σ_F(π_A(E))在条件F仅涉及A时等价于π_A(σ_F(E))其他地方不等价（因为缺失要过滤的列信息）；(c)选择的交换律σ_F1(σ_F2(E)) = σ_F2(σ_F1(E))；(d)选择对连接的自然分配σ_F(E1⋈E2)按F的不同部分可能等价且提前过滤每个表（谓词下推的关键）：σ_{F1∧F2∧F3}(E1⋈E2) → σ_{F1}(E1) ⋈ σ_{F2}(E2) （其中F1仅涉及E1, F2仅涉及E2, F3涉及两表联接键）
- 谓词下推（Predicate Pushdown）是最重要的优化启发式规则：WHERE条件尽量推到尽可能初级的表（在表的扫描阶段就过滤行），而不是在JOIN之后全局过滤。这么做可以大幅减少参与JOIN运算的行数、降低中间结果大小
- 连接顺序选择——穷举搜索vs动态规划：n个表的连接顺序排列数为Catalan数(C_n ∝ (2n)!/((n+1)!n!))以表数量指数增长。系统通常使用动态规划（DP）将搜索空间缩小——保留连接子集的最优子计划逐步构建更大表集的连接决策——最终得到整表集的最优计划
- 统计信息对CBO准确性的关键作用：信息包括表总行数N、索引不同值基数（NDV/唯一值数量，影响索引过滤后预计行数）、数据分布直方图(均衡柱状图/频率分布柱状图)——频度高的值可能导致错误的索引选择。统计信息过期可通过ANALYZE TABLE刷新
- 代价公式模型（简化的）：成本 ≈ (IO读取页数 × IO代价系数) + (CPU处理行数 ×  CPU代价系数)。各DBMS内部有精细的代价常数并按平台存储介质（SSD vs HDD）调整代价模型

## 易错点

1. **查询优化器不一定总能找到最优计划**：因为搜索空间巨大，动态规划在某些过多Join的情况下截断搜索（MySQL 8.0的join_optimizer的search_depth控制搜索二叉树深度）——最终计划是"有界搜索的局部最优"而非"全局最优"。理解这一点有助于手动引导优化器。

2. **等价的谓词条件写法下优化器反应不同**：WHERE col1 IN (SELECT ...)可能被重写为半连接Semijoin策略而WHERE col1 = (subquery)用标量比较（两者底层选择不同逻辑的物理计划）。优化器的改写模块(Query Rewrite)先语义优化匹配再CBO选择，写法不同→最终计划不同。

3. **Hint/FORCE INDEX不是永久方案**：因为数据分布会变化(从读写10/20变成了80/20)，当初有用的强制方案在新的统计信息下已不再适用。维护硬绑定索引的SQL随着数据增长和变化会劣化。不如依靠及时更新的统计信息让CBO自动决策。

4. **优化器也会"犯错"并使用"计划基线"修复**：某些DBMS(如Oracle SQL Plan Baseline, SQL Server Plan Guide)允给特定查询固定基线计划——如果自动（新）计划性能倒退早固定回已证明高效的计划。这是实践中的加固措施(adaptive plan)。没有基线的DBMS通常需HINT固定。

## 例题

**例题1**：对查询树应用等价变换优化——原始查询：SELECT name FROM Emp WHERE dept='IT' AND salary>8000。配合表Emp(emp_id, name, dept, salary, INDEX idx_dept_sal(dept,salary))，通过谓词下推及索引选择生成高效的计划。

**解答**：逻辑查询树在原层是σ_{dept='IT'∧salary>8000}(π_{name}(Emp))。优化通过交换投影和选择：π_{name}(σ_{dept='IT'∧salary>8000}(Emp))—投影下推但由于name依赖于全行需要完整过滤后才能投影。物理索引选择：联合索引idx(dept,sal)涵盖dept='IT'等值+salary>8000范围—匹配最左前缀(等值dept→+2列范围salary)。物理计划: 索引扫描→索引条件过滤→投影name。索引覆盖所有需要的条件(dept,salary)+最终SELECT的name(?)——如果索引用(dept,salary,name)则是全面覆盖查询无回表。只用(dept,salary)的话name不在就需要回表。

**例题2**：三表JOIN查询优化——描述优化器如何利用动态规划选出最佳连接顺序。

**解答思路**：假设表A,B,C。动态规划构建所有子集组合的最优计划：大小为1时各自全表扫描/索引选择为初始起点(1表最优)。大小为2子集：AB和AC和BC——每个子集找最优的JOIN方法和驱动顺序(前者用旧记录子计划最优,选择Hash/嵌套循环/合并)。大小为3全组合：ABC = AB结果+C连接(把C最优连接放在最后)。对整个ABC组合选出最小总代价的顺序。右深树和左深树处理分别对应不同计划拓扑。最终输出总代价最低的物理连接树。
