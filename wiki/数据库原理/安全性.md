## 核心定义

数据库安全性（Database Security）是指保护数据库免受非法用户访问、恶意篡改和数据泄露的一组技术手段和策略。数据库的安全体系涵盖多个层次：（1）身份认证（Authentication）——验证尝试连接数据库的用户身份（用户名/密码、证书、LDAP/Kerberos集中认证）；（2）访问控制（Authorization/Access Control）——通过DCL（GRANT/REVOKE）授予用户对特定数据库对象的操作权限（自主访问控制DAC），或基于安全标签和密级的强制访问控制（MAC）限制数据访问；（3）数据加密——传输层加密(TLS/SSL)和存储加密（透明数据加密TDE/列级加密）防范窃听和数据文件失窃；（4）审计（Audit）——记录所有敏感操作的日志（谁、何时、做了何种操作）以供事后追溯和合规检查；（5）SQL注入防护——应用程序通过参数化查询/预编译语句防范攻击者通过拼接恶意SQL窃取/破坏数据库。数据库安全是层层防御体系，不能仅靠单一措施。

## 关键结论

- 自主访问控制（DAC）：数据所有者决定哪些用户有权限访问其数据，并通过GRANT和REVOKE管理权限的传播。这是大多数DBMS的基础安全模型
- 强制访问控制（MAC）：每个数据对象和用户都有一个安全等级标签（如绝密/机密/公开），DBMS强制执行标签比对规则——用户只能读取同级或更低级别对象（read down）、只能写入同级或更高级别对象（write up—防止数据向下泄露）
- 审计功能：通过数据库审计插件或内置审计日志，跟踪SELECT/INSERT/UPDATE/DELETE/DDL等操作，在发生安全事故后可以回溯操作者和时间。审计开销对性能有影响（审计策略需要精细设计避免记录太多无用日志）
- 传输加密：客户端与数据库服务器之间的TCP连接通过SSL/TLS加密，防止中间人攻击和网络窃听——数据库证书的配置在客户端和服务端的正确设置
- 存储加密（TDE透明数据加密）：数据库数据文件（数据和日志文件）在磁盘上以加密形式存储，数据在读入内存中自动解密写回加密——即使物理磁盘被盗取也无法复原数据（需密钥管理安全存储）。也支持应用级的列加密（应用存密文）提供更强的保护

## 易错点

1. **访问控制不等于应用安全**：数据库的GRANT/REVOKE仅管理DBMS内的对象权限。如果应用程序对用户使用了数据库相同的DB用户连接，则绕过了数据库用户细分的权限管理。应该采用每个真实用户在应用层隔离，DB用户根据角色管理（如app_readonly/app_readwrite/app_admin）。

2. **加密不等同于本身绝对安全**：加密数据的访问需要密钥——密钥管理和轮替是安全的更艰难的挑战。TDE将密钥也存在服务器磁盘上（即使是HSM保护），服务器能自解密——如果攻击者获得服务器操作权（OS root级别），TDE不能阻止其读数据（因为内存中已解密或在OS层面数据已可访问）。仅防护物理盗窃场景。

3. **SQL注入的根源——将用户输入拼接到SQL字符串**：所有的防御措施归结为使用预编译语句（参数化语句）和存储过程。对用户输入做转义/加反斜杠是辅助手段（不同DBMS转义不统一——应彻底由参数化语句切断定义代码和参数的混排）。任何试图通过在前端清洗和黑名单过滤来阻止注入最终是会失败的。

4. **RBAC（角色权限管理）与强制访问控制的区分**：RBAC本质是DAC的一种有组织的快捷分配机制（将权限绑定角色→将用户绑定对应角色），并非强制访问控制MAC——MAC有绝对强制遵循的安全标签/密级矩阵不依赖用户自主选择共享。

## 例题

**例题1**：设计一个银行数据库的安全方案——有三类用户：柜员(只能查询账户余额和交易明细)、操作员(可办理转账等事务)、经理(能看报表和账户全息)。用角色实现权限管控并说明各级限制方法。

**解答**：
```sql
CREATE ROLE teller, operator, manager;
-- 柜员：仅查询
GRANT SELECT ON account TO teller;
GRANT SELECT ON transaction TO teller;
-- 操作员：增加转账事务
GRANT SELECT,UPDATE ON account TO operator;
GRANT INSERT ON transaction TO operator;
CREATE VIEW Report AS SELECT ... --经理能看全视图
GRANT SELECT ON Report TO manager;
-- 权限分配
GRANT teller TO user1;
GRANT operator TO user2;
GRANT manager TO user3;
```

**例题2**：何为审计并举例审计日志如何帮助识别和追踪数据泄露。

**解答思路**：数据库审计记录操作日志：包括操作时间、操作者(DB用户+来源IP)、对象名称、操作类型(SELECT/UPDATE/...)、操作前的旧值和操作后的新值。一旦发现某用户的敏感数据大规模下游异常（暗网数据泄露），回顾审计日志——发现哪些用户在近期批量查询了敏感客户表(全表SELECT)——锁定可能的泄露源和时间，可进一步分析其IP/操作模式。审计日志是事后追溯类安全合规的必要手段。
