## 核心定义

数据库（Database）是长期存储在计算机内、有组织、可共享的大量数据的集合。数据库管理系统（DBMS，Database Management System）是位于用户和操作系统之间的一层数据管理软件，负责数据的定义、操纵、组织、存储、管理和维护。数据库系统（Database System，DBS）由数据库、数据库管理系统（及其开发工具）、应用程序和数据库管理员（DBA）组成的完整系统。数据管理技术的发展经历了三个阶段：（1）人工管理阶段（1950年代以前）——数据由应用程序自行管理，不共享、无独立性、无专门的管理软件；（2）文件系统阶段（1950年代末至1960年代中期）——数据以文件形式存储在磁盘上，由操作系统的文件系统统一管理，一定程度上实现了数据共享但数据冗余大、一致性差、数据和程序之间联系紧密（缺乏独立性）；（3）数据库系统阶段（1960年代末至今）——出现了专门的数据库管理系统，具有数据结构化、数据共享性高且冗余度低、数据独立性（物理独立性和逻辑独立性）高、由DBMS统一管理和控制的特征。

## 关键结论

- 数据库系统的核心特征：数据结构化（数据按数据模型组织，不仅描述数据本身还描述数据间的联系）、高共享低冗余（面向整个企业或应用系统共享，减少冗余，但有时允许必要的冗余以提高效率）、高数据独立性（应用程序不受物理存储和逻辑结构变化的影响——通过三级模式结构/两层映像实现）、统一的数据控制（安全性控制、完整性控制、并发控制、数据库恢复）
- 数据库系统的三级模式结构（ANSI/SPARC体系结构）：模式（Schema，也称逻辑模式，是数据库中全体数据的逻辑结构和特征的描述）、外模式（External Schema，也称子模式/用户模式，是数据库用户能看见和使用的局部数据逻辑结构的描述）、内模式（Internal Schema，存储模式，是数据物理结构和存储方式的描述）
- 两层映像：外模式/模式映像（定义了外模式与逻辑模式之间的对应关系，保证了逻辑独立性）和模式/内模式映像（定义了逻辑模式与存储模式之间的对应关系，保证了物理独立性）
- 数据模型是数据库系统的核心和基础，它是对现实世界数据特征的抽象，分为三个层次：概念模型（如E-R模型，信息世界的建模，与DBMS无关）、逻辑模型（如关系模型、层次模型、网状模型，由DBMS支持，进行数据组织的逻辑建模）、物理模型（数据在存储介质上的具体存储结构和存取方法）

## 易错点

1. **数据库和DBMS不是同一个概念**：数据库特指数据集合本身，DBMS指管理数据库的软件系统。日常语言中常将MySQL、Oracle称为数据库，严格来说是DBMS。在正式场合需明确区分两个概念。

2. **文件系统与数据库系统的结构区别**：文件系统中文件与应用程序直接对应，一个文件通常只服务于一个应用程序，数据定义被嵌入在应用程序代码中。数据库系统中数据由DBMS统一管理，多个应用程序可以共享同一数据，数据结构定义在DBMS中且与应用程序分离。

3. **物理独立性与逻辑独立性的区别**：物理独立性是指当数据的存储结构（内模式）改变时，通过调整模式/内模式映像，逻辑模式和应用程序不受影响。逻辑独立性是指当数据的逻辑结构（模式）改变时（如增加新字段），通过调整外模式/模式映像，外模式以及由此外模式支持的应用程序不受到影响。

4. **DBMS不是自动解决所有数据问题**：虽然DBMS提供了事务、恢复、并发控制等功能，但如果数据库设计不当（如不好的范式化程度、不合理的索引策略等），DBMS无法自动优化——数据库设计者/DBA仍需深入理解业务需求和DBMS特征才能高效运用数据库系统。

## 例题

**例题1**：简述数据库系统阶段相对于文件系统阶段的主要优势，并解释"数据独立性"的含义和重要性。

**解答**：优势：（1）数据结构化——数据和数据之间的联系由DBMS维护而非分散在应用程序中；（2）数据共享——多个用户/应用并发访问同一数据而DBMS保证了数据一致性和并发安全；（3）数据独立性——应用程序与数据结构和物理存储解耦；（4）数据控制——DBMS有安全性、完整性、并发和恢复机制。数据独立性：外模式/模式映像保证逻辑独立性，模式修改不影响已有应用程序；模式/内模式映像保证物理独立性，物理存储方式和存取路径的改变不影响逻辑结构和应用程序。数据独立性降低了系统维护成本，允许数据库随需求演变而应用程序代码不需要全面重写。

**例题2**：三级模式结构中，一个数据库可以有多个外模式但只有一个模式和一个内模式。同一数据的多种外模式如何协作？用SQL视图(View)说明外模式的体现方式。

**解答思路**：一个数据库只有一个逻辑模式（全体数据逻辑结构），但不同用户群体看到的可以是不相同的部分数据——这是通过定义外模式实现的。在关系数据库中，外模式通常以SQL视图(View)的方式呈现。例如，人力资源部的外模式只展示员工信息视图：CREATE VIEW HR_View AS SELECT emp_id, name, department, salary FROM Employee WHERE department='HR'；财务部的外模式可能展示员工薪资详情。二者通过GRANT权限限制各自只能访问允许的视图/外模式。当模式变化（如Employee表加了新列）时，只需调整相关外模式视图定义，不修改外模式的应用程序无需改动——这就是逻辑独立性。

## 代码示例

```sql
-- 展示三级模式结构在SQL中的体现
-- 1. 模式（逻辑模式）—— CREATE TABLE定义全体数据逻辑结构
CREATE TABLE Employee (
    emp_id   INT PRIMARY KEY,
    name     VARCHAR(100),
    dept_id  INT,
    salary   DECIMAL(10,2),
    hire_date DATE,
    FOREIGN KEY (dept_id) REFERENCES Department(dept_id)
);

-- 2. 外模式—— CREATE VIEW定义特定用户可看到的局部数据
CREATE VIEW HR_Employee_View AS
SELECT emp_id, name, dept_id, hire_date
FROM Employee;

-- 3. 授予用户只能访问外模式的权限
GRANT SELECT ON HR_Employee_View TO hr_user;
```
