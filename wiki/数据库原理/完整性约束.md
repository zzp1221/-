## 核心定义

关系完整性约束（Integrity Constraints）是关系模型中保证数据正确性和相容性的一组规则，分为三大类。（1）实体完整性（Entity Integrity）：关系的主码（主键）中的每个属性值不能为空（NULL），且元组的主码值必须唯一。这是因为主码是用来唯一标识每个元组的，如果主码存在空值或重复则失去了标识功能。实体完整性通过PRIMARY KEY声明和DBMS自动创建的唯一索引来保证。（2）参照完整性（Referential Integrity）：外码的值要么为空（表示当前没有关联实体），要么必须等于其参照的关系中某个元组的候选码值。参照完整性保证了实体间引用关系的正确性——不允许出现指向不存在的实体的"悬挂引用"。在SQL中通过FOREIGN KEY声明和UPDATE/DELETE时的级联或拒绝操作来维护。（3）用户定义完整性（User-Defined Integrity）：针对具体应用中数据的语义要求而定义的约束条件，反映现实世界的业务规则。例如成绩取值范围0-100、性别取值只能是男或女、身份证号的格式校验等。在SQL中通过CHECK(CONDITION)约束、域(DOMAIN)的定义、规则(RULE)和触发器(TRIGGER)等手段强制实现。

## 关键结论

- 实体完整性规则确保了数据库中每一个实体都有确定的身份标识——每行都是不同的且可被寻址的。这是关系数据库操作的最基本前提（没有主键的实体是不能被有效引用的）
- 参照完整性的维护策略（级联操作）：ON DELETE/UPDATE CASCADE（级联执行，当被参照行删除/更新时自动同步删除/更新外键行）、SET NULL（将被参照行删除/更新时外键置为空）、SET DEFAULT、RESTRICT/NO ACTION（如果外键有依赖关联则拒绝删除被参照行）。这些策略决定了数据操作的安全行为
- INSERT操作只破坏实体完整性和用户定义完整性（新增了不满足约束的行），不直接破坏参照完整性（因为只有外键被建立了才需要关联被动的行，插入的行本身是否引用存在的被参照表是需要检查的，这是检查参照完整性）
- DELETE操作可能破坏参照完整性（如果被删除的行被其他表的外键引用，级联删除/置空/拒绝规则需要被妥善定义）。UPDATE操作可能破坏所有三种完整性（更新主键影响实体完整性、更新外键或主键影响参照完整性、更新任意字段可能违反CHECK约束）
- 完整性约束检查的执行时机：对于IMMEDIATE约束，每条DML语句执行后立即检查；对于DEFERRABLE约束，可在整个事务提交前才检查（如导入大数据时暂时关闭参照完整性检查到commit前统一验证以提升性能）

## 易错点

1. **实体完整性中主键不能为空≠每个属性都不能有空值**：组合主键的情况下，主键的每个属性都不能为NULL（所有组件都不为空）。但主键不必是单列的——组合键中任意一分量为NULL都会导致该行的主键值为NULL，违反了实体完整性。

2. **NULL值在参照完整性中是"允许"而非"适用"**：外键允许NULL但当外键不为NULL时，它的值必须匹配参照表的候选码。空外键的意思是该元组"尚未关联任何参照实体"——这体现了"关联是可选的（Optional）"。如果外键不允许NULL（NOT NULL），就必须确保每个外键值都匹配参照表。

3. **数据库会"自动"的约束不代表应用层可以不做检查**：虽然DBMS在存储层强制实体/参照/域约束，但应用程序仍应进行预检——因为DBMS报错的错误信息不友好，用户体验差。合理的做法是由应用层在提交DML查询前先做验证（但应用层的验证不能代替数据库约束保证）。

4. **用户定义完整性和业务规则的关系**：用户定义完整性约束可以在数据库层面上表达许多类型的规则，但过于复杂的业务规则通过CHECK约束和触发器实现会导致复杂性和性能问题。复杂业务规则通常放在应用层实现，数据库保留基本的约束作为"最后防线"。

## 例题

**例题1**：考虑三个关系：Student(sno, sname, sdept)、Course(cno, cname, credit)、SC(sno, cno, grade)。分析执行以下各操作时，可能违反哪种完整性约束：
(1)向Student表插入('2021001', NULL, 'CS')
(2)删除Course表中cno='C01'的记录，且SC中有该课程的选课记录
(3)在SC中插入('2021001', 'C99', 95)，其中C99在Course表中不存在

**解答**：（1）违反用户定义完整性——sname被定义为NOT NULL但插入NULL。不违反实体完整性（sno=2021001唯一）。实际上如果sname为NOT NULL，插入会被立即拒绝。（2）可能违反参照完整性——SC中仍有记录引用C01为外键。若无级联删除策略，删除Course中的C01行会违反参照完整性（SC中存有指向之的外键），操作被拒绝。若有ON DELETE CASCADE，删除C01自动删除SC中所有cno='C01'的行。（3）违反参照完整性——外键cno='C99'在Course表中不存在，不能建立有效的引用。

**例题2**：说明CHECK约束的作用（举两个SQL示例说明它涵盖的规则类型），CHECK约束是否有其局限性（例如交叉行/交叉表的约束）？

**解答思路**：CHECK约束可以验证单列或行级的用户定义完整性——如CHECK(grade BETWEEN 0 AND 100)确保成绩在有效范围、CHECK(ssex IN ('男', '女'))确保性别合法。局限性： CHECK不能涉及其他的行（不能表达聚合约束如"班级平均成绩不低于70%"，这需要断言ASSERTION或触发器）、不能跨表引用（不能检查外表中对应值的状态，如"选课人数不能超过教室容量"，这需要触发器或应用层检查）、在某些DBMS中（如MySQL 5.7及以下）CHECK只做语法解析不实际强制（MySQL 8.0+ 完全支持）。

## 代码示例

```sql
-- 完整性约束的完整SQL定义示例
CREATE TABLE Student (
    sno   CHAR(8) PRIMARY KEY,                     -- 实体完整性
    sname VARCHAR(20) NOT NULL,                     -- 用户定义完整性
    ssex  CHAR(2) CHECK (ssex IN ('男', '女')),     -- 用户定义完整性
    sage  INT CHECK (sage BETWEEN 10 AND 100),
    sdept VARCHAR(20)
);

CREATE TABLE SC (
    sno   CHAR(8),
    cno   CHAR(4),
    grade DECIMAL(4,1) CHECK (grade BETWEEN 0 AND 100),
    PRIMARY KEY (sno, cno),                        -- 实体完整性
    FOREIGN KEY (sno) REFERENCES Student(sno)      -- 参照完整性
        ON DELETE CASCADE ON UPDATE CASCADE,
    FOREIGN KEY (cno) REFERENCES Course(cno)
        ON DELETE RESTRICT ON UPDATE CASCADE
);

-- 创建触发器实现复杂完整性约束
-- 例如：限制每个课程选课人数不超过100
CREATE TRIGGER check_enrollment_limit
BEFORE INSERT ON SC
FOR EACH ROW
BEGIN
    DECLARE cnt INT;
    SELECT COUNT(*) INTO cnt FROM SC WHERE cno = NEW.cno;
    IF cnt >= 100 THEN
        SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = '选课人数已达上限';
    END IF;
END;
```
