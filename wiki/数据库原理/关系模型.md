## 核心定义

关系模型（Relational Model）由IBM研究员E.F.Codd于1970年提出，是目前数据库领域的主流数据模型。关系模型建立在严格的数学理论之上——以集合论中的关系和谓词逻辑为基础。其核心概念包括：（1）域（Domain）——一组具有相同数据类型的值的集合（如整数的集合、长度为10的字符串集合）；（2）笛卡尔积（Cartesian Product）——给定一组域D1,D2,...,Dn，所有这些域的笛卡尔积是所有可能的n元序组的集合：D1×D2×...×Dn = {(d1,d2,...,dn) | di∈Di, i=1,2,...,n}，其中基数为|D1|×|D2|×...×|Dn|；（3）关系（Relation）——笛卡尔积的一个有限子集被称为关系，直观上就是一张二维表。关系中的每一行称为元组（Tuple），每一列称为属性（Attribute），属性的取值范围即对应的域。关系模式（Relation Schema）是关系的逻辑结构描述，形式化为R(U, D, DOM, F)，其中U是属性名集合，D是域的集合，DOM是属性到域的映射，F是属性间数据依赖关系的集合。关系模型通过外键实现不同关系之间的联系，通过关系代数/关系演算提供严格的数据操作语言基础。

## 关键结论

- 关系的三个基本性质：（1）关系中的每一列（属性）是不可再分的数据项（满足原子性——1NF要求）；（2）同一关系中任意两行不能完全相同（关系是元组的集合，集合中不允许重复元素，所有关系都必须有候选码）；（3）行列的顺序无关紧要（关系是无序的集合），但在实际数据库中物理存储可能有顺序以支持索引优化
- 关系模式vs关系实例：关系模式是关系结构的描述（类似类型的定义），是相对静态、稳定的；关系实例是某一时刻关系中元组的集合（类似变量的当前值），随时间随数据操作变化。模式定义了关系的结构约束，实例必须符合模式规定
- 候选码、主码和外码：候选码（Candidate Key）是能唯一标识每个元组的最小属性集（真子集不能再做码）；从候选码中选一个作为主码（Primary Key）；外码（Foreign Key）是属性或属性集，其取值要么为空，要么等于另一个关系的某个候选码的值，用来表示关系间的引用联系
- 关系的三种类型：基本表（实际存储的关系）、查询表（查询结果对应的临时关系）和视图（从基本表导出的虚表，不实际存储数据只有逻辑定义）

## 易错点

1. **域和属性的区别**：域是值的集合（值空间），属性是列的逻辑名称。多个属性可以定义在同一个域上，但它们代表不同的语义。例如属性"发货地址"和"账单地址"都定义在Address域上，但它们是两个不同的属性（可以取不同值，虽然值在同一值空间）。

2. **笛卡尔积的子集不都满足完整性约束**：形式上笛卡尔积的任意子集都是"关系"，但实际数据库系统要求关系必须满足完整性约束（实体完整性、参照完整性、用户定义完整性）。反过来，不满足约束的笛卡尔子集合虽然数学上是"关系"但数据库内不接受。

3. **"关系模式"不等同于"SQL的CREATE TABLE"**：关系模式是一个逻辑结构描述的五元组，CREATE TABLE只是在这个描述中创建一个具体实现的方式（将模式名称、属性名称和类型以及其他约束定义在DBMS中）。一个关系模式可以有不同数据库平台上对应的创建语法。

4. **关系的外码必须指向主码吗？** 外码可以指向被参照表的任意候选码，不一定是主码（虽然实践中绝大多数外键指向主键）。参照完整性要求被参照属性在被参照表中必须满足唯一性（即是候选码），并和参照表中的值相等或存在NULL值。

## 例题

**例题1**：设域D1={张三，李四}，D2={数学，物理}。写出D1和D2的笛卡尔积和笛卡尔积的基数。说明所有笛卡尔积有哪些是现实世界中有意义的关系？

**解答**：笛卡尔积D1×D2 = {(张三,数学), (张三,物理), (李四,数学), (李四,物理)}。基数 = 2×2 = 4个元组。这些笛卡尔积表达了"任意学生都可能学任意课程"的所有可能组合。在实际"选课"关系中，可能只有部分组合有意义（张三只选了数学，李四选了数学和物理），所以有意义的选课关系应是这个笛卡尔积的某个特定子集。例如，选课关系SC可以只包含3个元组：(张三,数学), (李四,数学), (李四,物理)，它就是笛卡尔积的一个子集。

**例题2**：解释关系模型中"关系"与一般意义上的"表格"的区别。为什么关系数据库要求关系是规范化的？

**解答思路**：日常所说的表格可以有合并单元格、嵌套的子表、不统一的数据类型等，但关系模型中的"关系"是严格定义的形式结构：每个分量必须是原子值（不可再分的标量）——这就是1NF规范化要求。关系不能有嵌套表是因为关系代数的操作（选择、投影、连接等）都是在平坦的元组之上定义的——如果某个属性本身又是一个子表，就无法用标准的集合论操作套用扁平化关系代数。规范化要求保证了对数据库一致操作的理论基础。

## 代码示例

```sql
-- 关系模式定义和关系实例
-- 学生关系模式：Student(sno, sname, ssex, sage, sdept)
CREATE TABLE Student (
    sno   CHAR(8) PRIMARY KEY,
    sname VARCHAR(20) NOT NULL,
    ssex  CHAR(2) CHECK (ssex IN ('男', '女')),
    sage  INT,
    sdept VARCHAR(20)
);

-- 选课关系（外键参照）
CREATE TABLE SC (
    sno   CHAR(8),
    cno   CHAR(4),
    grade DECIMAL(4,1),
    PRIMARY KEY (sno, cno),
    FOREIGN KEY (sno) REFERENCES Student(sno),
    FOREIGN KEY (cno) REFERENCES Course(cno)
);

-- 关系实例（插入数据）
INSERT INTO Student VALUES ('20210001', '张三', '男', 20, '计算机系');
INSERT INTO SC VALUES ('20210001', 'C001', 85.5);
```
