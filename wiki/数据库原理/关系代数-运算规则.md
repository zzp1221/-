## 核心定义

关系代数运算规则（Relational Algebra Equivalence Rules / Transformation Laws）是指关系代数表达式中，多个运算符之间可以等价替换的一组代数定律。在数据库查询优化中，优化器（Query Optimizer）利用这些等价规则将一个关系代数表达式（逻辑查询计划）转换成另一个执行效率更高但结果相同的等价表达式（等价物理计划）。这些规则是查询改写（Query Rewrite）和逻辑优化的理论基础，涵盖选择、投影、连接、笛卡尔积和集合运算的交换律、结合律、分配律等基本变换。核心规则包括：（1）选择下推（Predicate Pushdown）——先把选择条件σ尽量下推到查询树的叶节点靠近数据扫描端，因为选择操作可以大量减少需要处理的元组数，是最有效的优化手段；（2）投影下推（Projection Pushdown）——尽早移除不需要的属性列以缩减中间元组宽度和内存开销；（3）连接顺序重排（Join Reordering）——利用连接的交换律和结合律，调整多表连接顺序使得产生的中间结果最小化（通常选用选择性高、结果集小的两表先做连接）。关系代数的等价变换规则是查询优化器生成候选执行计划的语法工具——不同的变换序列产生不同的物理计划。

## 关键结论

- 选择下推（Predicate Pushdown）是最重要的优化规则：σ_{p1∧p2}(R) ≡ σ_{p1}(σ_{p2}(R))（合取条件下的选择分解），σ_p(R ⋈ S) ≡ σ_p(R) ⋈ S（当p只涉及R的属性时选择可以下推到连接之内减少参与连接的行数），这种"先过滤再参与连接"的策略极大减少从磁盘加载和网络传输的量
- 选择与集合运算的分配规则：σ_p(R ∪ S) ≡ σ_p(R) ∪ σ_p(S)，σ_p(R - S) ≡ σ_p(R) - S（R差S——过滤可以下推到做差的两个数据集上处理；但不能直接下推到交，需保留等价语义）
- 投影下推规则：Π_A(R ⋈ S) ≡ Π_A(Π_{A∩R}(R) ⋈ Π_{A∩S}(S))（在连接前先投影出连接需要的属性和最终需要的属性，丢弃其他列——"先裁减列再连接"比全列连接高效）
- 连接等价变换：R ⋈ S ≡ S ⋈ R（交换律），(R ⋈ S) ⋈ T ≡ R ⋈ (S ⋈ T)（结合律——是改变连接顺序的基础），R × S（笛卡尔积）是连接的特例
- 选择与笛卡尔积等价→转换为连接：σ_p(R × S) ≡ R ⋈_p S（将笛卡尔积+选择合起来变为等值连接/θ连接——减少临时笛卡尔积的爆炸性开销，是最常见改写）
- 大小表连接的启发式规则：当σ(R)或σ(S)能大幅缩小结果时（高选择性），先完成那条选择并物化缩小后的关系，再与另一表连接——这是执行"尽可能减少在连接关系中出现的行"的方式

## 易错点

1. **等价规则的先决条件关注不足**：很多规则应用的先决条件决定可否正确应用——例如投影下推规则要求Σ的实际属性列在归约后的投影Π中包含——如果你丢弃了连接条件中的列，正确性失效（投影过渡会抛出错误）。在没有完备统计信息的情况下关系代数变换可能生成错误计划。

2. **语法等价 ≠ 语义等价**：由于修饰限制/空值/重复等因素——某些查询原始架构的等价但数据结果不同。尤其在包含集差值（差、并→重复值问题）与外连接（不能肆意交换/移项——半连接和反连接规则不同）下，优化的关系代数等价性必须保持确切的语义不变。

3. **连接结合律在真实数据库中受数据偏斜影响大**：虽然R ⋈ (S ⋈ T)和(R ⋈ S) ⋈ T在关系代数上是等价的——但若中间连接产生大量膨胀而后T过滤掉多数（类似"连接后过滤"劣势），相反另一变换顺序产生最小的中间结果——调用成本模型而不仅是代数等价性来选择调优。

4. **"选择下推永远有利"有例外**：比如σ_p对索引无帮助的过滤（不是索引键列或索引次序不满足对p的评估）因扫描全表的开销大于先连接再用小结果筛选——这种情况将选择上提到连接之上反而更差（选择p对表自身没有访问路径优化），下推使得优化错失了在连接树更高处合并过滤条件的机会。

## 例题

**例题1**：给定图书馆数据库的三表：Books(bid, title, author, pub_year)，Borrowers(uid, uname, city)，Loans(bid, uid, loan_date)。查询"2024年后借出的北京读者所借书的书名"。写原始关系代数并做选择下推和投影下推变换。

**解答**——原始：
Π_title( σ_{pub_year > 2024 ∧ city='Beijing' ∧ loan_date > '2024-01-01'} (Books ⋈ Loans ⋈ Borrowers) )

选择分解下推：
(1) σ_{city='Beijing'}(Borrowers) ⋈ Loans → 中间结果Borrow_L
(2) Π_{bid,title}(σ_{pub_year > 2024}(Books)) → 先过滤并投影列
(3) Π_{uid,bid}(σ_{loan_date > '2024-01-01'}(Loans)) → 过滤日期后只保留主外键
再嵌套Borrow_L ⋈ Book_filter ⋈ Loan_filter → 最后投影title

优化后大幅减少参与连接的行数和列宽。

**例题2**：验证选择下推等价性：σ_{p1}(σ_{p2}(R)) ≡ σ_{p1∧p2}(R)。

**解答思路**：(左侧)先对表R根据条件p2筛选→结果集中再基于条件p1筛选——相当于根据p1和p2的交集作为总筛选条件（只选定同时满足两个条件的行），等价于一次性筛选条件p1∧p2——因为筛选操作作用于元组独立的布尔判定，合取顺序不影响最终结果。

## 代码示例

在pgAdmin/psql中通过EXPLAIN观察选择下推和条件应用位置：

```sql
-- 创建测试数据
CREATE TABLE orders (oid INT PRIMARY KEY, cid INT, order_date DATE);
CREATE TABLE customers (cid INT PRIMARY KEY, city VARCHAR(50));
-- 等价代数写入SQL（对应原始写法）
SELECT o.oid FROM orders o
JOIN customers c ON o.cid = c.cid
WHERE c.city = 'Beijing' AND o.order_date > '2024-01-01';
-- EXPLAIN查看优化器的"扫描+过滤"的顺序和条件位置
EXPLAIN (ANALYZE, VERBOSE)
SELECT o.oid FROM orders o
JOIN customers c ON o.cid = c.cid
WHERE c.city = 'Beijing' AND o.order_date > '2024-01-01';
-- 观察JOIN顺序与过滤下推关联
```
