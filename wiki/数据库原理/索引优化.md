## 核心定义

索引优化（Index Optimization / Index Tuning）是指根据查询负载特征和数据库的运行统计信息，设计、选择和调整索引以最大化查询性能并最小化维护开销的系统性工作。索引优化的核心工具是EXPLAIN——分析SQL语句的执行计划，了解DBMS优化器如何选择和使用索引。索引优化的原则包括：（1）在查询的选择度高（过滤后返回少量行）的列上建立索引；（2）优先匹配高频查询的WHERE、JOIN ON、ORDER BY、GROUP BY子句中的列建立多列联合索引覆盖多个条件；（3）避免在低基数（Low Cardinality）列上单独建立索引（如性别仅男女→索引几乎无用）；（4）监控索引的使用情况，定期删除未被使用冗余索引（多余索引浪费存储、降低DML性能）；（5）在生产中避免索引失效的情况——函数操作、隐式类型转换、前导通配符、OR条件导致全扫描等。索引优化是数据库运维中最重要的持续性能调优活动。

## 关键结论

- EXPLAIN执行计划的解读（MySQL）：常用字段——id（查询序号）、select_type（查询类型: SIMPLE简单查询, SUBQUERY子查询, DERIVED派生表, UNION联合）、type（访问类型好坏排序: system>const>eq_ref>ref>range>index>ALL）、possible_keys（可选索引）、key（实际使用的索引）、rows（估计扫描行数）、Extra（额外信息: Using index覆盖索引, Using where过滤, Using filesort文件排序, Using temporary临时表）。type=ALL全扫描通常是最需要优化的点
- 索引失效的常见原因：(a)对索引列使用函数或表达式——WHERE YEAR(date_col)=2023→索引失效，应写为date_col BETWEEN '2023-01-01' AND '2023-12-31'；(b)隐式类型转换——索引列是VARCHAR但查询写了WHERE id=123(数字)会转换为VARCHAR→索引可能失效；(c)LIKE '%abc'以通配符开头→索引失效；(d)联合索引未匹配最左前列→索引不可用；(e)OR条件中任一方的列无索引→合并为全扫描
- 索引维护的开销：新增索引加速读取但减慢写入（INSERT/UPDATE/DELETE需同步更新索引B+树的每个插入更改）。一个表有5-8个以上索引很常见会出现写困扰——应评估写入负载和读取负载的权衡
- 定期统计信息更新：优化器依赖索引基数/数据分布统计信息做出索引选择决策，统计信息过期后会发生"应走索引而优化器选择了全表扫描"的判断。通过ANALYZE TABLE更新统计信息保持优化器决策正确
- 索引分区和局部索引：在大数据量的表上使用分区表——可设置局部索引（按分区键分别建索引而非全局大索引）减小索引大小和维护开销

## 易错点

1. **创建了索引≠数据库就一定会用它**：优化器根据统计信息决定成本最低的访问路径。如果统计数据不准确（未及时ANALYZE），优化器可能错误估算行数而导致弃索引选全表扫描——此时应更新统计信息而非强制索引。MySQL的FORCE INDEX应慎用。

2. **索引越多越好**：多个索引在读取时提供额外路径选择，但每个索引维护都会消耗DML的插入/更新/删除时间（B+树分裂/合并）和磁盘空间。建索引需衡量查询频率vs更新频率的平衡并删除无用的索引。

3. **小表不需要索引**：行数很少的表（几百到几千行）全扫描速度远快于索引查找（因为索引查找涉及两次读取：索引页面+数据页面），优化器会自动选择全表扫描而忽略索引。手动建索引对小表无性能收益。

4. **复合索引的列长度影响索引大小和效率**：每个索引页存放的键条目数更多，页分裂更少。在复合索引中使用小类型(INT>VARCHAR)排前，使用短前缀索引（INDEX idx_email(email(10))仅索引email的前10字符）可能牺牲部分精确性换取大幅削减索引占用——需根据选择性判断。

## 例题

**例题1**：SQL查询SELECT * FROM orders WHERE status = 'pending' AND YEAR(create_time)=2023 AND amount > 100。分析此查询可能存在的索引使用问题和优化方案。

**解答**：主要问题：YEAR(create_time)函数使create_time上的索引失效。优化方案：改写为WHERE status='pending' AND create_time BETWEEN '2023-01-01' AND '2023-12-31' AND amount>100。设计联合索引(status, create_time, amount)——利用status等值在最前→用create_time范围过滤→amount作为范围条件或ICP下推。

**例题2**：MySQL EXPLAIN Extra 显示 "Using index; Using temporary; Using filesort" 分别表示什么问题？如何解决Using temporary和Using filesort？

**解答思路**：Using index——覆盖索引（好指标）。Using temporary——需要临时表存放中间结果(常见于GROUP BY和DISTINCT与索引排序不匹配)。Using filesort——无法利用索引排序需独自排序（文件排序——额外开销）。解决：Using temporary常因GROUP BY列与索引列序不匹配产生——让GROUP BY的列顺序和索引最左前缀一致消除。Using filesort常出现在ORDER BY列无最左前缀索引匹配——加上符合排序序的索引即可自动获得有序结果消除排序步。
