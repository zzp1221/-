## 核心定义

**软件演化**（Software Evolution）是软件在其生命周期中持续变化和适应的过程。Lehman 的软件演化定律（Laws of Software Evolution）是对大型软件系统变化规律的经典总结：(1) **持续变化**（Continuing Change）：一个有用的软件必须不断适应真实世界的变化，否则将逐渐丧失有用性；(2) **复杂度增加**（Increasing Complexity）：若不主动降低复杂度，软件的结构会随着每次修改而逐渐退化和复杂化；(3) **自调节**（Self Regulation）：软件演化过程具有自调节特性——产品增量的分布和变化率趋向于稳定；(4) **组织稳定性守恒**（Conservation of Organizational Stability）：在软件的整个生命周期中，开发团队的有效全球活动率（产出率）是近似恒定的；(5) **熟悉度守恒**（Conservation of Familiarity）：每次发布的增量内容（新增和修改的功能量）维持稳定——否则团队难以保持对系统的熟悉度；(6) **持续增长**（Continuing Growth）：软件的功能内容必须持续增长以满足用户需求；(7) **质量衰减**（Declining Quality）：若不主动维护和适应，用户感知的软件质量将持续下降；(8) **反馈系统**（Feedback System）：软件演化过程是一个多层次、多回路、多代理的反馈系统。

**过程改进**（Process Improvement）是系统化评估和优化软件开发过程的持续活动。PDCA 循环（Plan-Do-Check-Act，也称戴明环）是过程改进的通用方法论：Plan（分析现状、设定目标）→ Do（小范围实施变更）→ Check（评估变更效果）→ Act（成功的变更标准化、失败的总结经验，开启新一轮循环）。软件过程改进的经典框架：(a) CMMI 成熟度模型（见质量保证）；(b) ISO/IEC 15504（SPICE, Software Process Improvement and Capability Determination）；(c) 敏捷 Retrospective（回顾会议）——Scrum 中的代次优化反馈环。遗留系统的处理策略：(a) 继续维护（高昂成本但风险最低）；(b) 包装/封装（Wrapping，通过 API 封装暴露服务）；(c) 迁移/替换（逐步替换为新系统）；(d) 逆向工程 + 再工程（Reverse Engineering → Restructuring → Forward Engineering）。

## 关键结论

- Lehman 定律揭示了大型软件系统的根本属性——变化是不可避免的，"冻结"软件的尝试注定失败
- 复杂度增加的对抗手段是持续重构（Continuous Refactoring）
- 过程改进不是一次性评审——它是嵌入日常工作节奏的持续实践
- 遗留系统不是"单纯的垃圾"——它承载了多年积累的、往往未文档化的业务规则（"将系统运行了 15 年的知识"）
- Retrospective 是 Scrum 中最容易被跳过的仪式——但却是过程改进最强的杠杆

## 易错点

1. 遗留系统"大爆炸重写"：抛弃旧系统从零重写看似诱人，但往往低估旧系统暗含的业务规则（边界情况处理）——风险极高
2. 过程改进只关注引入新工具新流程——忽视了人的培训和文化的转变（过程改进 = 人 + 技术 + 流程的三位一体）
3. Retrospective 只讨论问题不产出行动项——须在回顾结束时至少产出 2-3 个可行动改进项并分配负责人

## 例题

**例题1**：一家公司有一个 12 年前用 Visual Basic 6 开发的订单管理系统。业务方希望将其现代化。推荐什么策略？

**解答**：推荐**绞杀者模式**（Strangler Fig Pattern）。(1) 保留旧系统继续运行；(2) 新建现代化服务（如 Java/Spring Boot 微服务），通过 API 网关路由；(3) 逐步将旧系统功能"绞杀"替换——每次移出一块功能到新服务。新老系统并存期间通过集成层通信；(4) 当所有功能全部迁移完毕，退役旧系统。避免大爆炸重写的风险，同时逐步现代化。

**例题2**：敏捷 Retrospective 的最佳实践有哪些？

**解答**：(1) 设定安全氛围——"无论发生了什么，都是团队共同承担"；(2) 数据驱动——回顾 Burndown Chart、缺陷数量、Cycle Time 等实际数据；(3) 聚焦少数关键改进项——产出 2-3 个可行动项（而非 15 个愿望）；(4) 跟踪跟进——Sprint 开始时回顾上次的 Action Items 完成情况；(5) 变换形式——不只问"什么好/什么不好"（容易产生惯性回答），使用 Sailboat、4Ls 等多样化格式。
