## 核心定义

软件架构评估（Software Architecture Evaluation）是在软件开发的早期阶段对架构设计方案进行评审和验证的方法，旨在在编码之前发现架构可能无法满足质量属性需求的风险。核心思想由 SEI（软件工程研究所）提出：架构决策影响深远，架构层面的缺陷修复成本远高于需求和设计层面的缺陷。

**场景和架构驱动因素**：在评估前首先明确系统的关键质量属性（Quality Attributes）：性能（吞吐量/延迟/可扩展性）、可用性（故障检测与恢复）、安全性（授权/加密/审计）、可修改性（新功能添加成本）、可测试性、可用性（Usability）、可互操作性。每个质量属性通过具体**场景**（Scenario）来描述——场景包含六要素：刺激源（Source）、刺激（Stimulus）、环境（Environment）、制品（Artifact）、响应（Response）、响应度量（Response Measure）。例如"当数据库连接失败（刺激）时，系统（制品）在正常操作模式（环境）下应在 3 秒内（响应度量）切换到备用数据库并通知运维（响应）"。

**ATAM**（Architecture Tradeoff Analysis Method，架构权衡分析法）是 SEI 开发的最成熟的架构评估方法。ATAM 不仅评估架构是否满足质量属性——更关键的是揭示质量属性之间的**权衡点**（Tradeoff Points）和**敏感点**（Sensitivity Points）。其步骤如下（分 4 个阶段 9 个步骤）：

**阶段一：展示和架构介绍**（Presentation）
1. 介绍 ATAM：向干系人解释 ATAM 流程
2. 业务驱动因素介绍：项目经理阐述业务目标和主要需求
3. 架构介绍：架构师展示架构方案（含视图和关键设计决策）

**阶段二：调查与分析**（Investigation and Analysis）
4. 识别架构方案/方法：提取架构选用的模式、策略和框架
5. 生成质量属性效用树（Utility Tree）：将质量属性细化到场景层级并排定优先级。效用树的根是"效用"（系统整体有用性），第二层为各质量属性（性能/可修改性/安全性...），第三层为具体属性（如性能→数据延迟），第四层为可量化的场景。对每个场景标注（重要性 H/M/L, 实现难度 H/M/L），高重要性 + 高难度的场景是评估重点
6. 针对高优先级场景分析架构方案：对每个重点场景讨论架构方案如何响应，识别风险决策（Risk）、非风险（Non-Risk）、敏感点和权衡点

**阶段三：测试**（Testing）
7. 头脑风暴与场景优先级排序：与全部干系人确认最重要场景
8. 再次分析：评估架构对头脑风暴场景的覆盖

**阶段四：报告**（Reporting）
9. 输出最终评估报告，包含：记录在案的架构方案、ATAM 场景、效用树、风险集、非风险集、敏感点和权衡点

输出产物：**(a) 风险集**：可能导致不良后果的架构决策，如"数据库连接池大小假设为单应用实例而不支持水平扩展"；(b) **非风险**：已通过标准实践良好应对的合理决策；(c) **敏感点**：某个架构参数的变化显著影响特定质量属性，如"缓存 TTL 对数据新鲜度（可修改性）敏感"；(d) **权衡点**：一个架构决策改善某个质量属性却损害另一个质量属性——如"加密提高了安全性但降低了性能"，这是 ATAM 的核心关注。

**SAAM**（Software Architecture Analysis Method，架构分析方法）是 ATAM 的前身，更关注可修改性。SAAM 使用场景评估架构对预期变更的适应性——将场景映射到架构组件上，统计受变更影响的组件数量（修改影响范围）。

**ARID**（Active Reviews for Intermediate Designs）：针对中间设计的轻量评估，适用于设计阶段而非整体架构层。

**轻量级架构评估**（Lightweight ATAM）：适用于中小项目，压缩为 1-2 天，聚焦核心场景而非全量步骤。

## 关键结论

- ATAM 的核心价值不是"找出架构对不对"而是"发现架构决策之间的冲突"——权衡信息是最宝贵的
- 效用树是把模糊的质量属性转化为可评估的具体场景的核心工具——无法度量的质量属性无法评估
- 架构评估的时间越早越好——在编码前识别风险的成本远低于编码后重构
- 评估不需要外部顾问——可以是团队内部的同行架构审查，关键是结构化方法和可追溯的结论
- 架构决策记录（ADR）应记录评估中发现的权衡点——供未来参考"为什么选了方案A而放弃了方案B"

## 易错点

1. ATAM 评估走过场——邀请大量干系人但只讨论"架构很清晰"等空洞评价，没有产生具体风险清单
2. 效用树中的场景不可量化——"系统要快"不是场景，"500 并发下 API 响应时间 p95 < 200ms"才是有效场景
3. 忽视权衡点中的反面——认识到加密降低性能，但没有量化"降低多少"以及"是否可接受"
4. 评估结论没有记录——数月后团队反问"为什么架构当时这么选"却无人记得

## 例题

**例题1**：电商系统架构评估中发现以下架构决策：(a) 每个微服务都有独立数据库（数据库专有模式）；(b) 使用同步 HTTP 调用做服务间通信；(c) 商品搜索使用 Elasticsearch 而非数据库直接查询。请指出哪个是敏感点，哪个是权衡点，哪个是非风险。

**解答**：
(a) 数据库专有模式是**权衡点**：提高服务独立性和可扩展性（正面），但跨服务的 join 查询困难且数据一致性问题显著（负面）。需要额外设计 CQRS/Saga 模式补偿。
(b) 同步 HTTP 是**敏感点**：当服务数量增长时延迟叠加效应显著——某个依赖服务变慢可导致整个调用链雪崩。对服务数量和延迟高度敏感。解决方案：引入异步消息、断路器。
(c) Elasticsearch 是**非风险**：搜索功能使用专用搜索引擎是成熟实践，提升搜索性能和关联度，无显著负面影响。

**例题2**：为一个社交 App 后端架构的质量属性"性能"部分绘制效用树（至少 6 个场景）。

**解答**：
性能 → 响应时间 → (H,H) 用户在 500 万条数据下搜索好友，API 响应时间 p99 < 200ms
性能 → 吞吐量 → (H,M) Timeline 接口在 1 万 QPS 下错误率 < 0.1%
性能 → 启动时间 → (M,L) App 冷启动 3 秒内首页内容渲染完毕
性能 → 资源利用 → (M,M) 在 100 万 DAU 下服务器 CPU < 70%, 内存 < 80%
性能 → 缓存效果 → (H,H) 热门内容命中缓存概率 > 95%
性能 → 并发性 → (H,H) 1 万并发用户同时评论时不出现数据竞态
