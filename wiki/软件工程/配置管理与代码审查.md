## 核心定义

**软件配置管理**（SCM, Software Configuration Management）是跟踪和控制软件变更的工程规程，贯穿整个软件生命周期。SCM 的核心活动包括：(a) **配置标识**（Configuration Identification）：定义配置项（CI, Configuration Item）——被版本控制的独立单元（如源文件、文档、构建脚本、Dockerfile）；(b) **变更控制**（Change Control）：通过变更请求（CR / RFC）流程评估、批准/拒绝、记录所有变更；(c) **配置状态报告**（Configuration Status Accounting）：记录和报告 CI 的当前状态和变更历史；(d) **配置审计**（Configuration Audit）：验证 CI 的完整性和一致性——确保实际状态与文档记录一致。

关键概念——**基线**（Baseline）是一个经过正式评审和批准的配置状态里程碑，作为后续开发的参照点。例如：需求基线、设计基线、代码基线（如 v1.0.0 发布）。基线建立后，任何对基线 CI 的变更必须通过正式变更控制流程。

**代码审查**（Code Review）是同行评审的一种形式——由除编写者之外的开发者检查代码变更，目的是：(a) 发现逻辑错误和潜在的 bug；(b) 确保代码风格和规范一致；(c) 知识分享——审查者和编写者互相学习；(d) 团队代码集体所有权意识。现代代码审查实践依托 Pull Request / Merge Request 流程，平台为 GitHub/GitLab/Bitbucket。

代码审查最佳实践：(a) 每次审查的代码量应小（< 400 行），否则审查质量显著下降；(b) 审查清单（Checklist）：命名是否清晰、是否有足够的测试、是否处理了异常、安全性考虑等；(c) 自动化前置——Lint、格式化、测试由 CI 自动检查，人工审查聚焦于设计和逻辑；(d) 建设性评论：批评代码而非作者，提出改进建议而非仅指出问题。

## 关键结论

- SCM 是"变更管理"的工程化——不仅是版本控制工具，更是制度和流程
- 基线是"冻结状态"——只有通过审批的变更才能修改基线 CI
- 代码审查的投入回报比极高：据统计可发现 60-90% 的缺陷（远高于测试的发现率）
- 审查者最多检查 200-400 行代码——超过此阈值的 PR 应拆分
- Git 本身是配置管理的基础工具，SMC 在此之上增加流程层的治理
- 轻量级代码审查（如异步 PR 评论）比正式的 Fagan 审查更高效且同样有效

## 易错点

1. 将 SCM 等同于 Git 版本控制——版本控制是 SCM 的一部分，SCM 还包括变更审批、基线管理、审计等
2. 代码审查变成了"找语法错误"——这些应由自动化工具（Linter, IDE）完成，人工审查应聚焦于设计、逻辑和安全性
3. 审查者只检查不批准：长时间不回复 PR 会阻塞开发流程——团队应约定 PR 的 SLA（如 24 小时内响应）

## 例题

**例题1**：某项目发布 v1.0 后，客户报告了一个严重 bug。描述 SCM 中如何管理这个修复的变更。

**解答**：(1) 以 v1.0 基线为基础创建 hotfix 分支；(2) 开发者修复 bug 并提交，附上关联的 Bug ID；(3) 通过代码审查；(4) CI 测试通过后合并到 master 和 develop；(5) 创建新基线 v1.0.1（Tag）；(6) 配置状态报告记录变更：v1.0 → v1.0.1，修复了哪个Bug、谁修复的、何时发布的；(7) 配置审计验证 v1.0.1 的 CI 完整（代码、文档、构建脚本）。

**例题2**：设计一个代码审查清单（至少 5 项）。

**解答**：
1. 逻辑正确性：代码是否实现了需求/修复了 Bug？边界条件是否处理？
2. 测试覆盖：是否为新代码添加了足够的单元测试？
3. 错误处理：异常路径是否被恰当处理？错误信息是否有意义？
4. 安全性：是否存在 SQL 注入、XSS、敏感信息泄露等安全问题？
5. 可读性：变量/函数命名是否清晰？是否需要额外的注释解释"为什么"？
