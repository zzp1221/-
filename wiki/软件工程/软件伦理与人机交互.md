## 核心定义

**软件伦理**（Software Ethics）是软件工程师在专业实践中应遵循的道德准则和责任。ACM/IEEE 联合发布的《软件工程伦理规范与职业实践》提出了八项基本原则：(1) **公众利益**（Public Interest）：软件工程师的行动应以公众利益为首要考虑；(2) **客户和雇主**：为符合公众利益的客户和雇主提供最佳服务；(3) **产品**：确保产品符合最高的专业标准；(4) **判断**：保持专业判断的完整性和独立性；(5) **管理**：软件工程管理者应倡导和遵循道德的开发管理方法；(6) **职业**：提升软件工程职业的诚信和声誉；(7) **同事**：公平对待和支持同事；(8) **自我**：终身学习提升专业能力。

软件伦理的具体议题包括：(a) 隐私与数据保护（GDPR、用户数据的合法收集和使用）；(b) 算法偏见和公平性（AI 系统中的歧视是否由训练数据偏差导致）；(c) 透明性和可解释性（算法决策尤其是影响用户权益时应可解释）；(d) 安全性责任（工程师是否有责任拒绝开发不安全的系统）；(e) 知识产权（尊重许可证和版权）。

**人机交互**（HCI, Human-Computer Interaction）是研究人与计算机系统之间交互的设计、评估和实现的学科。核心原则：(a) **以用户为中心的设计**（UCD, User-Centered Design）：设计过程始终围绕用户需求、能力和限制；(b) **可用性**（Usability）的五项质量组件：可学习性（Learnability）、效率（Efficiency）、可记忆性（Memorability）、错误（Errors）、满意度（Satisfaction）；(c) **可访问性**（Accessibility / a11y）：使残疾用户也能有效使用软件（WCAG 无障碍指南）——如为屏幕阅读器提供语义 HTML、足够的色彩对比度、键盘可操作性。

用户体验（UX, User Experience）超越可用性，涵盖用户使用产品的所有情感和感知——愉悦感、美观、品牌信任。UX 设计方法：用户研究（用户画像 Persona、用户旅程地图 Journey Map）、原型设计（Wireframe → Mockup → Prototype）、可用性测试、A/B 测试。

## 关键结论

- 软件工程师的专业责任不仅仅是对雇主——对社会和公众有更高责任（如报告危害公共安全的软件缺陷）
- 可访问性不是"附加功能"——它是基本人权，Web 内容的无障碍指南（WCAG 2.1）在许多国家具有法律效力
- 可用性测试不需要昂贵实验室——5 个用户就能发现 85% 的可用性问题（Nielsen 启发式评估）
- 好的 UX 不等于好看的 UI——核心是"用户能高效无挫地完成其目标"
- 算法偏差常在数据中而非代码中——训练数据的代表性和偏差评估至关重要

## 易错点

1. 将"伦理"视为边缘或额外话题——事实上它涉及日常决策（"数据处理稍微超出授权范围一点应该没事吧"）
2. 可用性测试仅找"非技术用户"——应测试不同背景、能力水平的真实代表性用户
3. UX 设计在开发完成后"美化"——UX 应从需求阶段介入，而非事后油漆

## 例题

**例题1**：你发现你正在开发的健康 App 在特定情况下可能泄露用户的病历数据。项目经理想快速发布稍后修复。你的伦理责任是什么？

**解答**：依据 ACM 准则第 1 条"公众利益"和第 3 条"产品"，数据泄露对用户造成严重隐私伤害，属于不可接受的发布风险。软件工程师有责任：向项目经理明确阐述风险后果、向上级或安全负责人直接报告、如内部解决无效可考虑外部举报（Whistleblowing）——公众利益高于雇主的短期商业利益。同时提供替代方案（如延迟发布修复后的版本）。

**例题2**：设计一个登录表单的可用性改进方案（不少于 3 点）。

**解答**：
1. 密码可见切换：提供"显示密码"的眼睛图标（用户在输入时可检查密码正确性）。
2. 有意义的错误提示：不提示"用户名或密码错误（不区分）"而提示"此邮箱未注册"或"密码错误，还剩 3 次尝试"。避免过于细化以防枚举攻击——平衡安全与可用性。
3. 回车键支持：输入完成后按 Enter 直接提交，无需鼠标点击按钮。
4. 输入焦距自动聚焦：页面加载后光标自动定位到用户名输入框。
