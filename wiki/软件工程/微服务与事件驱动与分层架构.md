## 核心定义

**分层架构**（Layered Architecture）是最经典和最广泛使用的架构风格，将系统分解为若干水平层次，每层承担特定职责，上层依赖下层。典型的四层架构：(1) **表示层**（Presentation Layer：UI、Controller）；(2) **业务逻辑层**（Business / Service Layer：领域模型、业务规则）；(3) **数据访问层**（Data Access / Persistence Layer：DAO、Repository）；(4) **数据库层**（Database）。关键约束：跨层调用只能从上到下（不能反向调用），通常也不允许跨层调用。分层架构的优点：结构清晰、各层可独立测试和替换、标准化。缺点：层间调用有性能开销、严格的分层有时与需求不吻合（如简单 CRUD 不需要过重分层）。

**微服务架构**（Microservices Architecture）是近年来最热门的新一代架构风格：将应用拆分为一组小而自治的服务，每个服务围绕业务能力构建，独立开发、独立部署、独立扩展。每个微服务有独立的数据库（Database per Service）——这是与单体架构的本质区别。微服务之间通过轻量级通信协议（REST/gRPC/消息队列）协作。核心基础设施：服务注册与发现（Eureka, Consul）、API 网关（Gateway）、配置中心、负载均衡、断路器（Circuit Breaker, Hystrix/Resilience4j）、分布式追踪（Zipkin/Jaeger）。

**事件驱动架构**（Event-Driven Architecture, EDA）围绕事件的产生、检测和响应来组织系统。事件是"系统中已经发生的事实"（过去式命名），如 OrderPlaced、PaymentCompleted。EDA 的两种主要拓扑：(a) **中介拓扑**（Mediator）：事件通过中央 Event Router/Message Broker 路由；(b) **代理拓扑**（Broker）：事件链式传播，无中央协调。CQRS（命令查询职责分离, Command Query Responsibility Segregation）和 Event Sourcing（事件溯源，所有状态变更以事件存储）是 EDA 的高级变体。

## 关键结论

- 分层架构适合中小型或中等复杂度的项目，微服务适用于大型分布式系统，EDA 适合异步、高吞吐场景
- 微服务的"去中心化数据管理"引入了分布式事务问题——Saga 模式替代传统 ACID 事务
- 微服务的"与单体相比"：微服务赢得可扩展性和技术多样性，但付出分布式复杂性、运维成本的代价
- 事件驱动与微服务天然契合——事件作为微服务之间的异步通信手段
- CQRS 将读模型和写模型分离，适合读写负载差异大的场景
- 领域驱动设计（DDD）的"限界上下文"（Bounded Context）概念是划定微服务边界的理论基础

## 易错点

1. 盲目拆分微服务：微服务不是银弹——对简单应用而言，微服务带来的分布式复杂性远超收益（Monolith First 原则）
2. 分布式事务的简单处理：跨多个微服务的一致性不是靠传统的 2PC（两阶段提交），而是用 Saga（补偿事务）或最终一致性
3. 分层架构中"开放分层"与"封闭分层"的混淆：封闭分层不允许跨层调用，开放分层允许——但封闭分层保持更好的模块性

## 例题

**例题1**：某电商系统使用微服务架构，设计订单创建和支付流程中各服务的协作。

**解答**：
- Order Service 创建订单（状态 = PENDING）
- 发布事件 `OrderCreated` → Payment Service 收到事件 → 执行扣款
- 支付成功 → 发布 `PaymentCompleted` → Order Service 更新状态为 CONFIRMED → Inventory Service 扣减库存 → 发布 `InventoryReserved`
- 若支付失败 → 发布 `PaymentFailed` → Order Service 标记订单为 CANCELLED
各阶段通过事件异步驱动，使用 Saga 编排（Orchestration 方式有中央 Saga Coordinator）。

**例题2**：单体应用拆分微服务时应考虑哪些原则？

**解答**：(1) 单一职责：每个服务只做一套紧密关联的事情；(2) 数据隔离：每个服务拥有自己的数据库，不允许跨服务直接访问数据库；(3) 限界上下文对齐：按 DDD 的限界上下文分割服务边界；(4) 逐步拆分：先拆分外围模块（非核心、变更频繁），再逐步深入核心；(5) 不要过早拆分：在单体中先建立清晰模块边界（Modular Monolith），再考虑物理拆分。
