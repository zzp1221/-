## 核心定义

软件知识产权（Software Intellectual Property）是保护软件创造者智力成果的法律权利体系，主要包括：(a) **版权**（Copyright）：保护代码的"表达形式"（源代码、二进制），不保护"思想"（算法本身）；(b) **专利**（Patent）：保护技术方案的新颖性和创造性发明（如 Google 的 PageRank 算法专利），需经过审查授予，有效期通常 20 年；(c) **商标**（Trademark）：保护软件产品的名称、标志和品牌标识；(d) **商业秘密**（Trade Secret）：保护未公开的算法、商业模式、客户数据等（如推荐算法的参数权重）。软件作为同时受版权和专利保护的独特对象，其法律地位在国际上多依据 TRIPS 协议（WTO 与贸易有关的知识产权协定）界定。

**开源许可**（Open Source License）是在开放源代码的同时对使用、修改、分发代码施加特定法律约束的许可协议。OSI（Open Source Initiative）审批的许可证满足十项标准（自由再分发、源码可得、允许衍生作品等）。主要许可证类型：

**宽松许可**（Permissive License）：
1. **MIT License**：最简单的许可证，允许任意使用、复制、修改、合并、出版、分发、再许可和/或销售软件副本，只要保留版权声明和免责声明。代表性项目：React、Node.js、Ruby on Rails、jQuery。
2. **Apache License 2.0**：与 MIT 类似但额外：(a) 明确授予专利权（贡献者向用户授予其专利许可）；(b) 要求标注修改（修改处说明）；(c) 包含专利报复条款（若被许可人起诉贡献者专利侵权则失去许可）。代表性项目：Kubernetes、Android、Apache HTTP Server。
3. **BSD License**（2-Clause / 3-Clause）：2-Clause BSD 与 MIT 几乎等价；3-Clause BSD 额外增加"禁止使用版权持有人名称做广告"条款。

**强保护（Copyleft）许可**：
4. **GNU GPL v2/v3**（General Public License）：最具代表性的 Copyleft 许可证，要求任何分发该软件的修改版本也必须以 GPL 兼容许可发布（"感染性" / Viral License）。GPL v3 新增：(a) 反 Tivoization 条款（硬件上的软件不可因数字签名限制而阻止修改后版本运行）；(b) 专利报复条款；(c) 与 Apache 2.0 兼容。代表性项目：Linux Kernel (v2)、Git、GCC。
5. **GNU AGPL v3**（Affero GPL）：GPL 的加强版——即使不"分发"软件仅"通过网络交互"（SaaS）、也必须公开修改后的源代码。代表性项目：MongoDB (旧版)、Grafana。
6. **GNU LGPL**（Lesser GPL）：允许非 GPL 的专有软件链接 LGPL 保护的库——折中方案，保护库本身但允许商业封装。代表性项目：Qt (双许可)、FFmpeg。

**非商业限制许可**（Source-Available）：
7. **BSL**（Business Source License）：MariaDB 公司创建——软件可见可用，但一定时间后（通常 4 年）自动转为真正的开源许可（GPL/Apache）。
8. **SSPL**（Server Side Public License）：MongoDB 创建，AGPL 变体——要求提供服务的整个技术栈软件（不仅是MongoDB本身）都必须以 SSPL 许可公开（过于激进，未被 OSI 批准）。

**开源合规**（Open Source Compliance）：企业使用开源软件的法律义务：(a) 建立开源使用审批清单；(b) 使用 Software Composition Analysis (SCA) 工具自动扫描依赖的许可证（如 FOSSA、Black Duck、Snyk）；(c) 遵守许可证"归因/Notices"义务——在产品的 About 页面列出所有第三方开源组件及其许可；(d) Copyleft 许可证的衍生作品边界界定——动态链接是否构成"衍生作品"是法律灰色地带。

## 关键结论

- 软件自动受版权保护（无需注册）——写代码即创建版权作品
- MIT/Apache 2.0 是商业友好的"无脑选择"——既能享受开源生态红利又不影响闭源产品的商业模式
- Copyleft ≠ 自由软件（Stallman 区分 Free Software 和 Open Source）——GNU 运动强调用户自由的道德意义，OSI 侧重开源的实用商业价值
- 许可证不兼容是一个真实问题——例如 Apache 2.0 与 GPL v2 不兼容（前者有专利条款后者无），混合使用需要特别的法律分析
- 软件专利的"新颖性"门槛各国差异大——美国历史上曾批准过明显算法（如 Amazon 的一键购买专利），欧洲则排除"计算机程序本身"为可专利客体

## 易错点

1. 代码没有许可证不等于"自由使用"——默认是"全权保留"（All Rights Reserved），作者未授权他人复制修改
2. 将"开源"和"免费"等同——Copyleft 许可证要求开放衍生代码（GPL），可能与企业商业模式冲突
3. GPL v2 与 v3 不兼容——使用 "GPL v2 only" 的代码不能与 "GPL v3 only" 代码合并
4. 修改 MIT 项目但删除了原有的版权声明——MIT 许可允许修改但必须保留原版权声明（即使只保留一行 `Copyright (c) 2022 Original Author` 也算合规）

## 例题

**例题1**：你正在开发一个商业 SaaS 产品，后端使用了以下开源组件：(a) React (MIT) 做前端；(b) PostgreSQL (PostgreSQL License, 与 MIT 类似) 做数据库；(c) 修改了 GPL v3 授权的某 ORM 库做数据访问。分析许可合规风险。

**解答**：(a) React (MIT) 无风险——只须在文档中保留版权声明；(b) PostgreSQL 无风险——允许商业使用；(c) 修改 GPL v3 库有**重大风险**——若你的后端服务"分发"该修改版（即使是内部的部署也可能构成分发），则整个后端服务可能被视为 GPL v3 衍生作品，有义务公开全部后端源代码。且 SaaS 模式下提供网络服务不涉及"分发"——但 GPL v3 的 Affero 条款可能有影响——本场景中 GPL v3 (不是 AGPL) 不要求因网络交互公开源码。建议：改用 MIT/Apache 许可的 ORM 方案（如 Prisma/TypeORM），或将自定义修改以单独库的形式按 GPL 发布（隔离感染）。

**例题2**：比较 MIT、Apache 2.0、GPL v3、AGPL v3 四个许可证对商业软件公司的影响梯度。

**解答**：
- MIT：最友好——可随意使用、闭源、商业化，仅需保留版权声明
- Apache 2.0：友好但须注意专利条款——若有专利争议则可能失去此许可下的专利授权
- GPL v3：限制大——若分发包含 GPL 代码的软件须公开完整的衍生源代码，与大多数商业闭源模式冲突。可作为服务器内部分离进程独立使用（不被视为衍生作品），但风险由法律解释决定
- AGPL v3：限制最大——除 GPL v3 全部要求外，即使不"分发"仅提供网络服务交互也必须公开代码，对 SaaS 公司直接不兼容。大多数公司和云服务商内部禁止使用 AGPL 组件
