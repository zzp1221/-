## 核心定义

软件度量（Software Metrics）是对软件产品、过程和项目进行量化测量的学科。度量为管理决策提供数据支撑——"你无法管理你无法测量的东西"（DeMarco）。核心度量分为三类：**产品度量**（衡量软件产品的复杂度、质量、规模）、**过程度量**（衡量开发过程的效率，如开发速度、缺陷率）、**项目度量**（衡量项目进度和资源消耗）。

**McCabe 圈复杂度**（Cyclomatic Complexity）由 Thomas McCabe 于 1976 年提出，度量程序控制流的复杂度。定义为 V(G) = E − N + 2P，其中 E 是控制流图的边数，N 是节点数，P 是连通分量数（通常为 1，故 V(G) = E − N + 2）。等价于判定节点数 d 加 1：V(G) = d + 1，或简单的："代码中 if/while/for/case 等条件分支的数量加 1"。圈复杂度度量了基本路径（独立执行路径）的数量。行业指导值：1-10 为简单（低风险），11-20 为中等复杂，21-50 为复杂（高风险），>50 为不可测试（必须重构）。

**Halstead 软件科学度量**由 Maurice Halstead 于 1977 年提出，基于程序的操作符和操作数计数的度量族。核心指标：(a) n₁=不同操作符数，n₂=不同操作数数，N₁=操作符总出现次数，N₂=操作数总出现次数；(b) 程序词汇量 n = n₁ + n₂；(c) 程序长度 N = N₁ + N₂；(d) 程序容量 V = N × log₂(n)，表示写出程序所需的"信息量"（比特数）；(e) 编程工作量 E = V / L（L 是实现层级），预测开发所需的心智努力。

**功能点**（Function Points, FP）由 Allan Albrecht 于 1979 年在 IBM 开发，从用户视角度量软件的功能规模。基于五类功能组件的计数：(a) 外部输入 EI；(b) 外部输出 EO；(c) 外部查询 EQ；(d) 内部逻辑文件 ILF；(e) 外部接口文件 EIF。每类按复杂度（低/中/高）赋予权重，加权求和得未调整功能点数 UFP，再乘以技术复杂度调整因子 TCF 得到最终 FP。功能点的价值在于：与实现语言无关，可用于早期需求阶段的规模估算。

## 关键结论

- McCabe 建议圈复杂度不超过 10——超过即应拆分函数或重构
- Halstead 度量在现代 DevOps 实践中较少直接使用，但其核心思想影响了代码质量工具（如 SonarQube）
- 功能点是 ISO 标准（ISO/IEC 20926）的软件规模度量方法，广泛用于政府/大型外包项目的合同计费
- LOC（代码行数）是最简单但最有争议的度量——受编程语言、编码风格影响大，激进者认为"最小化 LOC 才是王道"
- 好的度量驱动好的行为——但要小心"古德哈特定律"（当一个度量成为目标时，它就不再是一个好的度量）
- 代码异味（Code Smells）可视为定性的代码质量"度量"

## 易错点

1. McCabe 圈复杂度中对多分支语句（switch/case）的计算：每个 case 都算作一个判定节点——switch 的圈复杂度可能很高
2. 功能点中混淆 ILF 和 EIF：ILF 是系统**内部维护**的数据（可增删改），EIF 是**引用外部系统**的数据（通常只读）
3. LOC 中是否计算注释和空行：不同的度量标准不同——物理行（Physical LOC）vs 逻辑行（Logical LOC，以语句计）

## 例题

**例题1**：计算以下代码的 McCabe 圈复杂度。
```python
def classify(x):
    if x > 0:
        if x > 10:
            return 'large'
        else:
            return 'small'
    elif x < 0:
        return 'negative'
    else:
        return 'zero'
```

**解答**：判定节点：`if x > 0`（1个）、`if x > 10`（1个）、`elif x < 0`（1个）。共 3 个判定节点。V(G) = 3 + 1 = 4。四條基本路径：(1) x>10 → 'large'；(2) 0<x≤10 → 'small'；(3) x<0 → 'negative'；(4) x=0 → 'zero'。

**例题2**：功能点分析为何优于直接使用 LOC 估算项目规模？

**解答**：LOC 高度依赖编程语言（同样功能 C 语言比 Python 行数多）、开发人员编码风格和项目复杂度。功能点从用户需求视角出发，与实现技术无关——在需求阶段即可估算，且不同语言和平台的项目可直接比较规模。
