## 核心定义

软件文档（Software Documentation）是描述软件系统各方面信息的书面或电子化材料，贯穿整个软件生命周期。文档是软件工程"三个维度"（方法、工具、过程）的粘合剂——它连接需求→设计→实现→测试→部署→维护各阶段的信息流。

软件文档按受众和目的分为两大类：

**用户文档**（面向最终用户）：(a) 用户手册（User Manual）：操作步骤、功能说明、常见问题；(b) 快速入门指南（Quick Start Guide）；(c) 在线帮助系统（Online Help）；(d) 培训教程；(e) 安装指南。

**技术文档**（面向开发/维护人员）：(a) 需求规格说明（SRS）；(b) 架构设计文档（Architecture Decision Records, ADR）；(c) 详细设计文档（含类图、顺序图）；(d) API 文档（Swagger/OpenAPI 规范、Javadoc/Doxygen）；(e) 数据库设计文档（ER 图、表结构说明）；(f) 测试文档（测试计划、测试用例、测试报告）；(g) 构建和部署文档（DevOps Runbook）；(h) 运维手册（SOP, Standard Operating Procedure）；(i) 代码注释——"代码即文档"。

**文档质量特征**：准确性（与产品实际行为一致）、完整性、清晰性（无歧义）、可维护性（与代码同步更新）、可追溯性（文档条目关联到需求和代码）。

**敏捷中的文档处理**：敏捷宣言指出"可工作的软件高于详尽的文档"——但不等于没有文档。敏捷倾向于：(a) 活文档（Living Documentation），随代码自动生成（如 Cucumber/SpecFlow 的 BDD 测试报告、Swagger 的 API 文档）；(b) 最小必要文档（Just Enough Documentation），只写对团队有价值的内容；ADR（架构决策记录）是轻量级文档的代表。

## 关键结论

- 文档是软件的一部分——不完整或过时的文档导致维护成本飙升
- API 文档是开发者的"用户手册"——良好 API 文档降低集成成本
- 活文档优于静态文档——自动化生成减少了"文档漂移"（Document Drift）
- ADR 记录"为什么做了这个架构决策"——"为什么"比"是什么"更难得
- 代码注释解释"为什么"（意图），不解释"是什么"（显而易见的代码行）
- README.md 是开源项目的"名片"——应包含：项目简介、安装步骤、使用示例、贡献指南

## 易错点

1. 文档与代码分离导致过期——文档应放在与代码同一个仓库（Docs as Code），跟随代码一起版本化和审查
2. 注释过多或过少：`x = x + 1 // 把x加1` 是无用注释；关键算法不写注释是另一种极端——注释应为"意外的、非显而易见的"内容
3. 用户文档写成了技术文档——充斥 JWT、OAuth、微服务等术语，用户不知道也不关心

## 例题

**例题1**：为一个 RESTful API 设计文档应包含哪些部分？

**解答**：至少包括：(a) 概述（API 的功能、版本、Base URL）；(b) 认证方式（Bearer Token / API Key）；(c) 端点列表（Endpoint），每个含 HTTP 方法、路径、请求参数、请求体示例、成功/错误响应体示例、状态码说明；(d) 速率限制和分页说明；(e) 版本变更记录。建议使用 OpenAPI/Swagger 规范编写，可自动生成交互式文档。

**例题2**：比较传统 SRS 和敏捷用户故事作为需求文档的优劣。

**解答**：传统 SRS 优势：全面、详尽、可作为合同依据、支持可追溯矩阵；劣势：编写耗时、不易更新、倾向于过度规定实现细节。敏捷用户故事：轻量、聚焦用户价值、支持增量细化（通过验收标准补充）。劣势：碎片化（单个故事不展现系统全貌），需要额外手段（如 Story Map）整合全局视图。实践中常混合使用——高层用 SRS 或产品大纲，细节用用户故事 + 验收标准。
