## 核心定义

软件项目进度管理（Project Schedule Management）是规划、制定、管理和控制项目时间安排的过程，确保项目在规定时间内完成。进度管理的核心挑战是：在不确定性和资源约束下准确估算工作量、安排任务顺序并跟踪实际进展。

**甘特图**（Gantt Chart）由 Henry Gantt 于 1910 年代发明，是最直观的项目进度可视化工具。横轴为时间（天/周/月），纵轴为任务列表，每个任务用水平横条表示其起止时间。甘特图可以显示：(a) 任务间的依赖关系（箭头连接）；(b) 里程碑（菱形标记关键节点如"需求确认"、"Beta 发布"）；(c) 进度（横条中填充色表示完成百分比）；(d) 资源分配（任务旁标注负责人）。常用工具：Microsoft Project、Jira Timeline、Notion Gantt、Excel。

**关键路径法**（CPM, Critical Path Method）由 DuPont 和 Remington Rand 于 1950 年代开发，是确定项目最短完成时间和识别对整体工期有决定影响的任务序列的技术。关键路径是项目网络图中从起点到终点耗时最长的路径，关键路径上的任务称为关键任务——任何关键任务的延迟都将直接导致整个项目的延迟。

CPM 的关键参数：
- **最早开始时间 ES**（Earliest Start）：前驱任务都完成后此任务最早能开始的时间
- **最早完成时间 EF**（Earliest Finish）：ES + 任务工期
- **最晚开始时间 LS**（Latest Start）：不影响项目总工期的前提下最晚能开始的时间
- **最晚完成时间 LF**（Latest Finish）：LS + 任务工期
- **总浮时**（Total Float/Slack）：LF - EF = LS - ES，即任务可延迟而不影响总工期的时间量。关键任务的浮时为零。

CPM 计算采用两遍扫描法：(1) **正推**（Forward Pass）：从起点开始，ES=0, EF=ES+duration, 遇到多个前驱取 max(所有 EF)；(2) **逆推**（Backward Pass）：从终点开始，LF=项目总工期, LS=LF-duration, 遇到多个后继取 min(所有 LS)。浮时 = LS - ES = LF - EF，浮时为零的任务构成关键路径。

**PERT**（Program Evaluation and Review Technique，计划评审技术）由美国海军在 Polaris 导弹项目中开发，是对 CPM 的扩展——允许任务工期存在不确定性。每个任务使用三点估算：(a) 乐观时间 O（Optimistic）；(b) 最可能时间 M（Most Likely）；(c) 悲观时间 P（Pessimistic）。期望工期 TE = (O + 4M + P) / 6，方差 σ² = ((P - O) / 6)²。PERT 可以计算在某个时间点完成项目的概率（基于正态分布假设）。

**关键链法**（CCPM, Critical Chain Project Management）由 Eliyahu Goldratt 提出，在 CPM 基础上考虑资源约束——不仅要考虑任务依赖，还要考虑同一资源不能同时做两个任务。关键链法引入缓冲（Buffer）概念：项目缓冲（Project Buffer，保护总工期）、汇入缓冲（Feeding Buffer，保护关键链免受非关键链延迟影响）、资源缓冲（Resource Buffer，提前预警资源需求）。

## 关键结论

- 关键路径是项目中最长的依赖链——优化非关键任务不能缩短总工期，只有缩短关键路径才能提前交付
- 浮时（Total Float）为零的任务构成关键路径——一个项目可能有多条关键路径
- 甘特图适合沟通和跟踪，CPM/PERT 适合分析和优化——实践中通常结合使用
- PERT 的三点估算过于依赖 Beta 分布假设——实际软件开发中的不确定性远高于传统工程
- "学生综合征"（Student Syndrome）：人们倾向于在最后期限前才开始工作，缓冲时间常被浪费
- 滚动式规划（Rolling Wave Planning）：近期工作详细规划，远期工作粗粒度规划——随项目进展逐步细化

## 易错点

1. 将 PERT 的期望工期公式 (O+4M+P)/6 中的系数记错——系数是 4 不是 3 或 5
2. CPM 仅考虑任务依赖不考虑资源约束——实际项目中资源冲突可能产生比关键路径更长的"关键链"
3. 甘特图中忘记标记依赖关系——只有横条的甘特图无法分析关键路径，本质上是"任务列表带日期"
4. 将工期的总浮时当作可以自由使用的缓冲——如果多个任务共享同一段浮时，其中一个用完后续任务就没有浮时了

## 例题

**例题1**：某项目任务如下表，画出网络图并求关键路径和总工期。

| 任务 | 前驱 | 工期(天) |
|------|------|----------|
| A    | -    | 3        |
| B    | A    | 4        |
| C    | A    | 2        |
| D    | B,C  | 5        |
| E    | C    | 3        |
| F    | D,E  | 2        |

**解答**：正推：A[0→3], B[3→7], C[3→5], D[max(7,5)=7→12], E[5→8], F[max(12,8)=12→14]。总工期=14天。逆推：F[12→14], D[min(12)=12, LF=12→LS=7], E[LF=12→LS=9], B[LF=7→LS=3], C[LF=min(7,9)=7→LS=5], A[LF=min(3,5)=3→LS=0]。浮时计算：C的浮时=7-5=2天, E的浮时=12-8=4天。关键路径（浮时为0）：A(3)→B(4)→D(5)→F(2)，总工期14天。

**例题2**：用 PERT 估算某模块开发时间。乐观 3 天，最可能 5 天，悲观 10 天。求期望工期和标准差。该模块在 7 天内完成的概率约为多少？

**解答**：TE = (3 + 4×5 + 10)/6 = 33/6 = 5.5 天；σ = (10 - 3)/6 ≈ 1.17 天。7 天对应的 Z 值 = (7 - 5.5)/1.17 ≈ 1.28。查正态分布表，Z=1.28 对应概率约 0.8997。即约 90% 的概率在 7 天内完成。
