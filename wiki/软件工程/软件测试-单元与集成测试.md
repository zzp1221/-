## 核心定义

软件测试（Software Testing）是通过执行程序来发现错误和验证软件满足需求的过程。测试不是"证明程序没bug"，而是"发现尽可能多的bug"（Dijkstra）。测试按粒度层次分为多个级别。

**单元测试**（Unit Testing）是对软件中最小的可测试单元（通常是函数或方法）进行验证。单元测试应当：(a) 自动化、快速（毫秒级）；(b) 独立（测试之间不相互依赖）；(c) 可重复（同样输入永远同样输出）。测试用例遵循 Arrange-Act-Assert（AAA）模式：准备（Arrange）→ 执行（Act）→ 断言（Assert）。好的单元测试应当是 FIRST：Fast（快速）、Independent（独立）、Repeatable（可重复）、Self-Validating（自验证）、Timely（及时编写）。

**测试替身**（Test Doubles）：在单元测试中替代真实依赖的对象。包括：(a) **Dummy**：仅填充参数不实际使用；(b) **Stub**：返回预设固定值的简单实现；(c) **Mock**：记录交互并验证行为的替身（如"方法被调用了 N 次"）；(d) **Fake**：轻量级工作实现（如内存数据库替代真实数据库）。Mock 框架（Mockito, unittest.mock）简化 Mock 对象创建。

**集成测试**（Integration Testing）验证多个模块/组件组合在一起后能否正确协作。策略包括：(a) **大爆炸集成**（Big Bang）：一次组合所有模块——风险高，难以定位问题；(b) **自顶向下集成**：先测试顶层，逐步用真实模块替代桩模块（Stub）；(c) **自底向上集成**：先测试底层，逐步组合到顶层；(d) **三明治集成**：两者结合。现代微服务架构中，集成测试常指 API 契约测试和事件流集成测试。

**测试驱动开发**（TDD, Test-Driven Development）将测试从"事后验证"提前到"事前设计"：遵循 **Red-Green-Refactor** 循环——先写失败的测试（Red）→ 写最少代码使测试通过（Green）→ 重构保持代码整洁（Refactor）。TDD 的核心收益：迫使开发者先思考"接口和预期行为"，后思考实现。

## 关键结论

- 测试金字塔（Test Pyramid, Mike Cohn）：底层大量单元测试（快速、低成本）→ 中层适量集成/API测试 → 顶层少量端到端测试（慢、成本高）
- TDD 不仅是一种测试方法，更是一种设计方法论——引导产生可测试、松耦合的代码
- Mock 的过度使用可能导致"测试与实现细节耦合"——只 Mock 跨越边界的依赖（数据库、网络）
- 单元测试覆盖率（Code Coverage）不是越高越好——追求 100% 覆盖率常导致大量低价值测试
- 集成测试应关注模块间的接口契约（Contract），而非模块内部实现

## 易错点

1. 测试环境的"雪花"效应：测试依赖特定的执行顺序或外部状态（如系统时间、文件系统状态）——应通过依赖注入隔离
2. Mock 与 Stub 的混用：Mock 验证**行为**（"是否调用了方法"），Stub 提供**状态**（"返回固定的值"）——过度 Mock 使测试脆弱
3. Red-Green-Refactor 中跳过 Refactor 步骤：导致技术债务在测试代码中累积

## 例题

**例题1**：为以下 `Calculator` 类编写单元测试用例（含正常路径和边界条件）。
```java
public int divide(int a, int b) { return a / b; }
```

**解答**：测试用例：
- 正常：`divide(10, 2)` 应返回 5
- 零除数：`divide(10, 0)` 应抛出 `ArithmeticException`
- 负数：`divide(-10, 2)` 应返回 -5
- 边界：`divide(0, 5)` 应返回 0
- 边界：`divide(Integer.MAX_VALUE, 1)` 应返回 Integer.MAX_VALUE
每个用例独立运行，通过 JUnit `@Test` 注解标记，异常用例用 `assertThrows`。

**例题2**：自顶向下集成与自底向上集成各有什么优缺点？

**解答**：
自顶向下优点：早期可看到程序框架，桩模块逐步替换为真实模块，可及早发现顶层设计问题。缺点：底层模块测试延迟，桩模块编写量大。
自底向上优点：底层模块早期充分测试，驱动模块（Driver）编写简单。缺点：顶层模块最后集成，关键设计缺陷发现晚。
