## 核心定义

软件维护（Software Maintenance）是软件交付后为纠正错误、改进性能、适应变化的环境或增加新功能而修改软件的过程。维护占软件生命周期总成本的 50-70%（部分研究表明甚至更高）。IEEE 将维护分为四类：

(1) **纠错性维护**（Corrective Maintenance）：修复用户报告的缺陷和错误——"修理坏掉的"。包括重现 bug、定位根因、修复并回归测试。占比约 20%。

(2) **适应性维护**（Adaptive Maintenance）：修改软件以适应外部环境的变化——操作系统升级、数据库版本升级、新法规合规要求（如 GDPR）、第三方 API 变化等。占比约 25%。

(3) **完善性维护**（Perfective Maintenance）：根据用户反馈增强和改进已有功能——提升性能、改进用户界面、增加客户请求的小功能。占比约 50%，是占比最大的维护类型。

(4) **预防性维护**（Preventive Maintenance）：主动修改软件以提高其未来可维护性和可靠性，防止将来的问题——代码重构、文档更新、优化存储。占比约 5%。

**重构**（Refactoring）是 Martin Fowler 推广的在不改变软件外部行为的前提下改善其内部结构的系统化技术。重构不是"重写"——它是一小步一小步地进行、每步都运行测试验证行为不变。典型重构手法：提取方法（Extract Method）、重命名变量、内联临时变量、以多态替换条件逻辑、拆分大类、合并重复代码等。重构与性能优化不同：重构旨在改善设计，优化旨在改善速度——有时两者目标冲突。

**技术债务**（Technical Debt, Ward Cunningham 1992）：选择"快速但不干净"的方案而非"更好但耗时"方案所产生的长期隐性维护成本。技术债务需要偿还（重构），否则债务利息（维护越来越困难）会累积。

## 关键结论

- 维护 ≠ "修补bug"——完善性维护（增强功能）是维护类型中占比最大的
- 维护困难的根本原因是代码可读性差和文档缺失——"代码就是文档"在维护阶段失效
- 重构是预防性维护的核心武器——定期重构防止技术债务积累
- 重构应当在有测试保护的情况下进行——"没有测试的重构是危险的随意改动"
- Lehman 软件演化定律：(1) 持续变化：有用的软件必须持续改变；(2) 复杂度增加：若不主动降低复杂度，软件会越来越复杂
- 代码审查中发现的"代码异味"是重构的触发器信号

## 易错点

1. 将重构与"功能修改"混在一起做：应将重构和功能修改放在**不同的 commit** 中——功能不改时仅重构，重构完毕后测试全绿再做功能改动
2. 在项目截止日前"以后重构"的承诺很少兑现——技术债务如信用卡债：复利增长下后期还债更痛苦
3. 预防性维护过度：对所有代码无差别地进行预防性修改 → 引入新 bug，且投入产出比低下

## 例题

**例题1**：一段代码从单租户改为多租户架构属于哪类维护？修正计算错误的 bug 属于哪类？将 UI 从 Swing 迁移到 JavaFX 属于哪类？

**解答**：
- 多租户改造：适应性维护（适应新的架构需求）
- 修复计算错误：纠错性维护
- UI 框架迁移：适应性维护（适应技术环境变化）

**例题2**：重构以下代码：
```java
if (employee.type == "MANAGER") {
    salary = base * 1.5 + bonus;
} else if (employee.type == "ENGINEER") {
    salary = base * 1.2 + bonus;
}
```
**解答**：用多态替换条件逻辑。创建 `abstract class Employee { abstract double calculateSalary(double base, double bonus); }`，`Manager` 和 `Engineer` 各自实现 `calculateSalary`。客户端代码变为 `salary = employee.calculateSalary(base, bonus);`（无 if-else），更符合开闭原则——新增员工类型无需修改客户端代码。
