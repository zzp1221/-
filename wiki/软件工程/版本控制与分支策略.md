## 核心定义

版本控制（Version Control / Source Control）是管理软件代码变更的系统，记录每次修改的历史、谁做了什么、何时做的、为什么这样做。现代版本控制系统分为集中式（CVCS，如 SVN，中央仓库为权威来源）和分布式（DVCS，如 Git，每个本地仓库都是完整仓库）。Git 是当今最主流 DVCS，其核心概念包括：(a) **工作区**（Working Directory）、**暂存区**（Staging Area / Index）、**本地仓库**（Local Repository）、**远程仓库**（Remote Repository）；(b) 核心操作：`add`→暂存区、`commit`→本地仓库、`push`→远程仓库、`pull`/`fetch`→从远程获取；(c) 快照（Snapshot）而非差异（Diff）：Git 以文件系统快照存储每次提交，而非存储文件间的差异。

**分支策略**（Branching Strategy）是团队使用分支管理代码变更的约定，决定了如何隔离开发中的功能、如何合并回主线、何时以及如何发布。主要策略：

**(1) Git Flow**（Vincent Driessen 2010）：最详尽的分支模型。两个永久分支：`master`（生产就绪代码）和 `develop`（集成开发）。三类临时分支：`feature/*`（功能开发分支，从 develop 分出 → 合并回 develop）、`release/*`（发布准备分支，从 develop 分出 → 合并到 master 和 develop）、`hotfix/*`（紧急修复分支，从 master 分出 → 合并到 master 和 develop）。适合有明确发布周期的项目。

**(2) GitHub Flow**（GitHub 推荐，Scott Chacon 2011）：极简模型，仅一个主分支 `main`（始终可部署）。所有变更通过 Feature Branch → Pull Request → 代码审查 → 合并到 main。适合持续部署（CD）的 Web 应用。

**(3) Trunk-Based Development**（TBD，基于主干开发）：所有开发者频繁（至少每天一次）将小变更直接合并到主干 `trunk/main`。短生命周期的 feature flag 控制未完成功能的可见性（而非通过长期存在的 feature 分支隔离）。适合成熟的 CI/CD 和高绩效 DevOps 团队。

**(4) GitLab Flow**：环境分支模型——`master`→`pre-production`→`production`，代码逐级"上推"经过各环境。

**Pull Request / Merge Request**：提交合并请求时触发自动 CI 检查 + 代码审查（Code Review）。审查者评估代码质量、功能正确性、安全性，通过后才可合并。

## 关键结论

- Git Flow 适合有固定版本发布节奏的项目，但其复杂性也带来维护成本
- GitHub Flow 和 Trunk-Based Development 适合持续部署环境——简单、快速
- 分支的生命周期越短越好——长时间不合并的分支导致痛苦的集成冲突
- 约定式提交（Conventional Commits, `feat:`, `fix:`, `BREAKING CHANGE:`）使提交历史规范化，便于自动生成版本号和变更日志
- `.gitignore` 防止将编译产物、依赖、敏感文件（`.env`）等纳入版本控制
- Git Hooks（如 pre-commit hook）可在提交前执行Lint、格式化检查，确保代码质量

## 易错点

1. 将大文件/二进制文件纳入 Git 仓库——应使用 Git LFS（Large File Storage）或外部存储
2. 使用 `git push --force` 在共享分支上——会覆盖他人提交，应仅对私有分支使用，在共享分支使用 `--force-with-lease`
3. Merge Commit 与 Rebase 的选择：Rebase 保持线性历史但改变了提交 Hash，对已推送的分支 Rebase 会导致他人基于旧历史的工作失效

## 例题

**例题1**：团队使用 GitHub Flow。一个新功能开发的完整流程是什么？

**解答**：(1) 从 `main` 创建 feature 分支：`git checkout -b feat/new-payment`；(2) 在 feature 分支上开发和提交，定期 `push`；(3) 在 GitHub 上创建 Pull Request → CI 自动运行测试 → 请求同事 Code Review；(4) 根据 Review 反馈修改代码；(5) CI 和 Review 通过后，Squash & Merge 到 `main`；(6) 删除 feature 分支。`main` 上代码自动触发 CD 部署到生产环境。

**例题2**：解释 Git 中 `git merge` 和 `git rebase` 的区别及各自适用场景。

**解答**：`merge` 保留完整的提交历史和分支结构——产生一个 merge commit 将两条历史线接在一起。适合将 feature 分支合并到共享分支（保留上下文）。`rebase` 将 feature 分支的提交"重新播放"到目标分支顶端——产生线性历史但提交 Hash 改变。适合在推送前整理本地提交历史（squash commits 等）。共享分支禁用 rebase。
