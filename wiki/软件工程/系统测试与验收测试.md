## 核心定义

**系统测试**（System Testing）是在完整的、集成的系统上进行的端到端测试，验证系统作为一个整体是否满足指定的需求。与单元/集成测试不同，系统测试关注系统的整体行为和质量属性，通常由独立于开发团队的测试团队执行。系统测试类型包括：(a) **功能测试**（Functional Testing）：按 SRS 逐条验证功能需求；(b) **性能测试**（Performance Testing）：验证系统在预期负载下的响应时间、吞吐量和资源利用率；(c) **压力测试**（Stress Testing）：在极端负载（超出设计上限）下观察系统行为——是否优雅降级；(d) **安全测试**（Security Testing）：验证系统的安全性——认证、授权、数据加密、防注入等；(e) **可用性测试**（Usability Testing）：评估系统的易用性和用户体验；(f) **兼容性测试**（Compatibility Testing）：验证在不同操作系统、浏览器、设备上的表现。

**验收测试**（Acceptance Testing）是部署前最后一道测试，由客户或最终用户执行，目的是确认系统是否准备好投入生产使用。(a) **Alpha 测试**：在开发者环境中由用户在开发者指导下测试；(b) **Beta 测试**：将软件分发给外部真实用户进行试用并收集反馈——是"真实世界的测试"。UAT（User Acceptance Testing，用户验收测试）通常由业务方基于验收标准编写测试场景。

**回归测试**（Regression Testing）是重新运行已有的测试以确保最近的代码变更没有破坏已有功能——"新的改动不应该让旧的测试变红"。回归测试通常自动化程度高（CI/CD 流水线每次提交自动运行回归测试套件）。回归测试策略：重测全部 vs 选择性重测（根据代码变更范围只运行相关测试）vs 测试用例优先级排序。

**冒烟测试**（Smoke Testing）是初步测试，快速验证关键功能是否正常——"这个 build 是否可以交给测试团队进行更深入的测试？"

## 关键结论

- 系统测试是基于需求规格（SRS）的——每条功能需求至少对应一个系统测试用例
- 验收测试关注"业务需求是否满足"而非"技术是否无bug"
- 回归测试是 CI/CD 的核心环节——构建失败时阻断部署
- Beta 测试的优势在于真实用户环境和多样化操作模式——发现开发团队意想不到的问题
- 性能测试应尽早进行（不要等到系统测试阶段才发现架构性能瓶颈）
- 自动化回归测试的投资回报率最高——重复执行次数多、人工执行成本高

## 易错点

1. 系统测试时使用开发团队的测试数据——应使用接近生产环境的真实数据（脱敏后）以暴露数据相关的问题
2. 验收测试中客户代表不了解业务——验收测试应由真正使用系统的业务人员执行，而非 IT 代表
3. 回归测试数量膨胀——随着系统增长，回归测试套件越来越大，维护成本增加，需定期清理过时的测试

## 例题

**例题1**：为一个电商网站的"下单"流程设计系统测试用例。

**解答**：
- TC-FUNC-01：已登录用户选择商品 → 加入购物车 → 结算 → 确认订单 → 支付成功 → 订单状态变为"已支付"
- TC-FUNC-02：库存不足 → 用户下单 → 提示"库存不足"
- TC-PERF-01：500 名并发用户同时下单 → 响应时间 < 3 秒
- TC-SEC-01：未登录用户尝试下单 → 重定向到登录页面
- TC-COMPAT-01：在 Chrome / Firefox / Safari 上各执行一遍正常下单流程

**例题2**：生产环境中 report 了一个严重 bug。如何进行回归测试？

**解答**：(1) 开发团队修复 bug 并添加专门针对该 bug 的**回归测试用例**（复现原来的失败场景）；(2) 在 CI/CD 流水线中运行完整的回归测试套件；(3) 若资源有限，运行与此次修复相关的模块的"选择性回归测试"；(4) 构建通过后部署到预发环境再做一轮冒烟测试验证核心功能。确保"同样的bug不会再次出现"。
