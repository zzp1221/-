## 核心定义

系统设计（System Design）是将需求规格映射为软件结构和实现方案的工程活动——从"做什么"到"怎么做"的桥梁。设计分为两个层次：

**概要设计**（Preliminary Design / High-Level Design / Architectural Design）：关注系统的宏观架构。(a) 确定系统的整体架构风格（分层、微服务、事件驱动等）；(b) 将系统分解为子系统/模块，定义模块间的接口；(c) 设计数据存储方案（数据库选型、分表策略）；(d) 识别关键技术选型和第三方组件；(e) 输出为概要设计文档和架构图。

**详细设计**（Detailed Design / Low-Level Design）：深入到每个模块的内部实现。(a) 定义模块内的类、函数、数据结构的详细规划；(b) 细化算法逻辑；(c) 定义接口的精确签名；(d) 输出伪代码、类图、顺序图等详细设计文档。

设计过程中遵循的核心原则：
- **抽象**（Abstraction）：关注本质特征而忽略不重要的细节
- **逐步求精**（Stepwise Refinement）：从顶层抽象开始，逐步添加细节——自顶向下
- **模块化**（Modularity）：将系统分解为功能内聚、接口清晰的独立模块
- **信息隐藏**（Information Hiding）：模块的内部实现细节对外不可见，仅通过公共接口交互
- **功能独立**（Functional Independence）：每个模块功能单一、职责明确

## 关键结论

- 概要设计回答了"系统由哪些大部件组成、它们如何交互"；详细设计回答了"每个部件内部如何实现"
- 良好的设计应追求高内聚（High Cohesion）和低耦合（Low Coupling）
- 设计的可追溯性：每条设计决策应能追溯到一条或多条需求
- 接口设计（API/协议）是不同团队和子系统协作的"合同"——一旦确定则变更代价高昂
- 设计评审（Design Review）是发现设计缺陷的有效手段——同行评审和架构评审委员会的正式审查
- 现代软件设计中，UML 图、C4 模型（Context-Container-Component-Code）等建模工具有助于可视化设计

## 易错点

1. 跳过概要设计直接进行详细设计——这导致缺乏大方向，各模块设计各自为政，最终集成困难
2. 过度设计（Over-Engineering）：为未来不确定的需求设计了极其复杂的灵活机制，导致不必要的复杂性——遵循 YAGNI 原则（You Aren't Gonna Need It）
3. 接口设计仅考虑正常场景未考虑异常路径——鲁棒的接口设计必须处理超时、部分失败和边界情况

## 例题

**例题1**：为"在线书店"系统进行概要设计，划分功能模块并定义接口。

**解答**：
模块划分：
- 商品模块（Product）：商品信息维护、搜索、分类浏览
- 订单模块（Order）：购物车、下单、订单状态跟踪
- 用户模块（User）：注册、登录、个人信息管理
- 支付模块（Payment）：对接支付宝/微信支付接口
- 库存模块（Inventory）：库存管理、补货提醒

接口示例：订单模块调用商品模块获取价格 → `GET /api/products/{id}/price`；订单模块调用库存模块扣减库存 → `POST /api/inventory/reserve`。

**例题2**：解释详细设计中"伪代码"的作用，并举例。

**解答**：伪代码（Pseudocode）用接近自然语言的半形式化方式描述算法逻辑，介于自然语言和编程语言之间，便于讨论和审查而不陷入具体语法细节。
示例：
```
FUNCTION calculateDiscount(order):
    total = sum(order.items.price * order.items.quantity)
    IF total > 500:
        discount = total * 0.15
    ELSE IF total > 200:
        discount = total * 0.05
    ELSE:
        discount = 0
    RETURN total - discount
END FUNCTION
```
