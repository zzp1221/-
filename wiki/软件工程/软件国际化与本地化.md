## 核心定义

**国际化**（Internationalization, 缩写 i18n，因首字母 i 和末字母 n 间有 18 个字母）是在软件设计和开发阶段就做好支持多种语言和区域的能力——使得软件不需修改核心代码即可适应不同语言和地区。**本地化**（Localization, 缩写 l10n）是将软件适配到特定语言和地区的过程——不仅要翻译用户界面的文字，还要适应日期格式、货币符号、数字格式、排序规则、颜色文化含义、法律法规合规等。

两者的关键区别：国际化是**一次性的架构能力建设**（开发时做），本地化是**针对每个目标市场的适配**（可多次做、可外包给翻译公司）。没有国际化基础的本地化需要修改核心源代码，成本极高。

**全球化**（Globalization, g11n）是国际化+本地化的综合战略——它涵盖产品设计、开发、市场适配的全过程。一些公司还区分**世界可读性**（World-Readiness）：产品在技术层面已准备好被翻译和区域适配。

### 核心概念

**字符编码**：本地化最基本的挑战是编码。历史上各语言有独立的编码系统（ASCII 英语、GB2312/GBK 中文、Shift_JIS 日文、ISO-8859 系列），互不兼容——一页文本不能混用。**Unicode** 从根本上解决了这一问题——为世界上几乎所有书写系统的每个字符分配一个唯一码点（Code Point），当前 Unicode 15.1 收录了 15 万个字符。Unicode 的编码方式：UTF-8（变长，1-4 字节，向后兼容 ASCII，Web 标准）、UTF-16（2 或 4 字节，Java/Windows 内部使用）、UTF-32（固定 4 字节）。现代软件开发中 UTF-8 是事实标准——数据库、API、HTML、编程语言源码都应使用 UTF-8。

**区域设置**（Locale）：Locale 是一组参数组合——语言（zh）、文字（Hans/Hant）、国家/地区（CN/TW/SG）——用来确定日期、时间、数字、货币、排序的格式。如 `zh-Hans-CN` 表示中文简体-中国，`zh-Hant-TW` 表示中文繁体-台湾。BCP 47 标签规范定义了 locale 的命名。常见差别示例：
- 数字：`1,234.56`（英语）vs `1 234,56`（法语）vs `1.234,56`（德语）
- 日期：`05/02/2026`（美式 M/d/yyyy）vs `02/05/2026`（英式 dd/MM/yyyy）vs `2026-05-02`（ISO 8601 标准）
- 货币：`$1,234.56` (USD) vs `¥1,234` (JPY) vs `1.234,56 €` (EUR)
- 排序：不同语言有不同的字符排序规则（Collation）——德语中 ä 排在 a 之后还是 ae 之后？这取决于 locale 设置（DIN 标准 vs 电话簿排序）

**资源分离**（Resource Externalization）：国际化核心架构原则——将所有用户可见的字符串、图标、布局尺寸从源码中分离到外部资源文件。常见格式：(a) gettext `.po/.mo` 文件（Linux/开源世界标准）；(b) Java `.properties` 文件或 ResourceBundle；(c) .NET `.resx` 文件；(d) JSON/YAML 格式（React/Vue 前端框架常用 react-intl, vue-i18n）；(e) Flutter 的 ARB（Application Resource Bundle）文件。

**翻译管理**（Translation Management）：处理 30+ 语言的翻译需要工具支撑：(a) TMS（Translation Management System）如 Crowdin、Lokalise、Phrase，支持翻译版本同步、翻译记忆库（TM）、模糊匹配（"欢迎回来，{name}"在多数语言中可复用）；(b) ICU MessageFormat：支持复数形式（`{count, plural, one {# item} other {# items}}`）、性别、选择等跨语言格式化语法；(c) 伪本地化（Pseudolocalization）：自动生成模拟翻译文本（添加重音符号、长度扩展字符）来检测硬编码字符串和 UI 截断——如 "Settings" → "[Šéţţïñğš---]" 用于测试宽度余量。

**设计和布局的国际化考量**：(a) 文本长度差异：同一段内容从英语翻译到德语可能扩展 35%，翻译到日语可能缩短 30%——UI 必须支持文本弹性（不依赖固定宽度）；(b) 从右到左（RTL）语言：阿拉伯语、希伯来语、波斯语的整个 UI 需要镜像翻转——不仅是文字方向，图标（如箭头、"后退"按钮）、布局顺序都需要镜像；(c) 色彩象征差异：红色在西方是"警告"在中国是"喜庆"，绿色在伊斯兰国家是神圣的颜色——需要可定制的 UI 色彩方案。

**多语言数据库设计**：(a) 每个需要翻译的列建独立的翻译表（Product → ProductTranslation），关联 locale 字段；(b) 存储 JSON 或 HStore 字段（一个行中包含 "title_en", "title_zh" 列, 对于静态翻译可使用）（但不易做模糊搜索）；(c) 继承翻译框架（如 Ruby 的 Globalize gem, Django 的 django-modeltranslation）。

**法律与合规**：部分国家对软件语言有强制规定——法国（Toubon 法要求法语界面）、加拿大魁北克（法语必须与英语同地位）、中国（在国内销售的软件必须有中文版本或中文支持）。GDPR 对用户数据的多语言政策展示、同意声明有要求。

## 关键结论

- i18n 必须从第一天开始——事后添加国际化成本是前期设计的 3-5 倍
- 字符串外部化不等于 i18n——还涉及日期、货币、排序、RTL、编码等
- 永远不要用字符串拼接构建面向用户的消息——"你购买了 " + count + " 件商品" 在中文成立但在很多语言中复数形式不同，必须使用 ICU MessageFormat 模板
- 伪本地化是开发中发现的 i18n bug 的最有效手段——在 CI 中自动生成伪本地化版本并运行 UI 测试
- 翻译应与代码版本同步——不协调的版本节奏导致翻译与功能 UI 错位（旧翻译对应新按钮文本）

## 易错点

1. 硬编码字符串假定位——使用常量文件（`const LOGIN = '登录'`）而不使用 i18n 机制，改语言仍需修改代码常量定义
2. 将 locale 等同于语言——`zh-CN` 和 `zh-TW` 是同一语言（中文）但日期格式、货币符号、一些词汇不同
3. 翻译时忽略上下文——同样的英文词在不同上下文中可能需要不同的翻译（如 "Post" 可以是"帖子"或"邮寄"，翻译系统应允许提供上下文注释）
4. 忘记翻译错误消息和日志——面向用户的技术报错信息（如 "500 Internal Server Error"）也应做 i18n

## 例题

**例题1**：设计支持 3 种语言（中/英/日）的电商 App 的 i18n 架构（前端 React + 后端 Java/Spring Boot）。

**解答**：
前端：(a) 使用 react-intl 库，提取所有用户可见字符串到 JSON 文件（en.json, zh.json, ja.json）；(b) 由用户语言选择或浏览器语言偏好决定 locale；(c) 使用 `<FormattedMessage>` 组件替代硬编码字符串；(d) 货币用 `Intl.NumberFormat` API (带 locale 参数)，日期用 `<FormattedDate>`。
后端：(a) 使用 Spring 的 `MessageSource` + `ReloadableResourceBundleMessageSource`，对应 `.properties` 文件 (messages_en.properties, messages_zh_CN.properties, messages_ja.properties)；(b) API 返回的业务错误码使用 code + 参数，客户端查表翻译；(c) 多语言电子邮件模板使用 Thymeleaf 模板 + 语言特定模板文件。
数据库：(a) Product 表存价格等无需翻译的数据；(b) ProductTranslation 表 (product_id, locale, title, description) 存储多语言名称和描述；(c) API 根据请求头 `Accept-Language` 返回对应语言的产品信息。

**例题2**：以下代码有何 i18n 问题？请修复。

```java
String msg = "Hello, " + username + "! You have " + unreadCount + " new messages.";
System.out.println(msg);
```

**解答**：问题：(a) 硬编码英文字符串不可翻译；(b) 字符串拼接忽略复数形式（`unreadCount` 为 1 时说 "1 new messages" 语法错误——英文中应为 "1 new message"）。
修复：
```java
// 使用 MessageFormat - messages.properties:
// welcome.message=Hello, {0}! You have {1} new {1,choice,0#messages|1#message|1<messages}.
String msg = MessageFormat.format(
    bundle.getString("welcome.message"), 
    username, unreadCount
);
```
更好的方案是使用 ICU MessageFormat：`Hello, {username}! You have {count, plural, one {# new message} other {# new messages}}.`
