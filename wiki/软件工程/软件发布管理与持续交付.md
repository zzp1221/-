## 核心定义

软件发布管理（Release Management）是规划、调度、协调将软件版本从开发环境部署到生产环境的过程。发布管理确保新版本以可控、可预测、可回滚的方式交付给用户，是连接开发和运维的桥梁。在 DevOps 文化下，发布管理从仪式性的"发布日"转型为日常性的部署操作。

**持续交付**（Continuous Delivery, CD）指软件的每个变更经过自动化流水线（构建→测试→打包）后始终处于可发布状态——团队可以按需在任何时刻将任何一个通过验证的版本部署到生产环境（通过一键部署或自动化触发）。持续交付不等于持续部署（Continuous Deployment）——持续交付强调"随时可发布"（发布决策由业务决定），持续部署强调"每次通过流水线都自动发布到生产"。

发布管理的核心概念：

**语义化版本**（Semantic Versioning, SemVer 2.0）：版本号格式为 MAJOR.MINOR.PATCH。MAJOR（主版本）：不兼容的 API 变更；MINOR（次版本）：向后兼容的功能新增；PATCH（修订版本）：向后兼容的 bug 修复。预发布和构建元数据可附加（如 `2.1.0-beta.1`）。

**部署策略**：
1. **蓝绿部署**（Blue-Green Deployment）：同时维护两套完全相同的生产环境——Blue（当前活跃）和 Green（新版本）。部署到 Green 后做冒烟测试，然后将所有流量从 Blue 切换到 Green。优点：零停机时间，瞬间切换/回滚（若新版本有问题直接切回 Blue）。缺点：需要双倍基础设施资源。
2. **金丝雀发布**（Canary Release）：先让少部分用户（5-10%）使用新版本，监控各项指标（错误率、性能、用户反馈），确认稳定后逐步扩大到全量。命名源于煤矿中的金丝雀（用来检测瓦斯泄漏的早期预警）。
3. **滚动部署**（Rolling Deployment）：逐个实例替换——先停掉一个旧实例，启动一个新实例，完成后再处理下一个，直到全部更新。无额外资源需求但部署时间较长。
4. **A/B 测试部署**：同时运行两个版本，根据用户特征路由到不同版本，用于业务决策（如哪个 UI 转化率更高）而非纯技术验证。

**功能开关**（Feature Toggles/Feature Flags）：通过配置开关在运行时动态启用或禁用功能，将功能发布与代码部署解耦。类别：(a) 发布开关（Release Toggles）：未完成功能的隐藏；(b) 实验开关（Experiment Toggles）：A/B 测试；(c) 运维开关（Ops Toggles）：紧急关闭高负载功能；(d) 权限开关（Permission Toggles）：按用户等级显示功能。

**发布回滚**（Rollback）策略：最简单的回滚是将流量切回上一版本的部署环境（蓝绿部署中瞬间回滚）。更复杂的情况涉及数据库 Schema 变更——应使用"扩展-迁移-收缩"模式（Expand-Contract）：先添加新列（向后兼容），双写新旧列，最后才删除旧列——确保任何版本都可独立运行。

**发布流水线**（Release Pipeline）：CI → 单元测试 → 构建制品 → 部署到开发环境 → 集成测试 → 部署到预生产环境 → 验收测试 → 生产部署 → 冒烟测试 → 监控。

## 关键结论

- 发布是"将变更安全地交付给用户"，部署是"将软件安装到环境"——两者不等同：功能开关可以 "部署但不发布"
- 金丝雀发布的核心价值是"爆炸半径控制"——将问题影响限制在小范围的用户中
- 蓝绿部署最适合有状态应用升级——因为可以维持旧环境的完整性
- 发布频率与可靠性正相关（DORA 研究）——高频发布的团队其变更失败率反而更低（更小的变更、更快的反馈、更多自动化）
- 所有部署策略的根本原则：使回滚比修复更快——恢复服务比修复 bug 优先级更高

## 易错点

1. 混淆持续交付和持续部署——持续交付允许手动批准发布，持续部署是全自动的
2. 蓝绿部署忽略数据库兼容性——切换流量容易但数据库回滚困难，需要精心设计的数据库迁移策略
3. 功能开关不清理——功能稳定后未及时移除开关代码积累技术债务，配置复杂性随开关数量爆炸

## 例题

**例题1**：团队使用蓝绿部署。新版本部署到 Green 后冒烟测试通过，切换流量后 5 分钟发现支付失败率飙升。应该怎么处理？分析此场景下蓝绿部署的优势。

**解答**：立即将所有流量切回 Blue（旧版本），Blue 环境仍在运行且状态完好。回滚时间等于流量切换时间（通常数秒），远快于任何 bug 修复。若没有蓝绿部署，传统"就地升级"需要手动恢复旧版本代码和数据库，可能产生数小时停机。蓝绿部署的优势在此场景体现为：(a) 毫秒级回滚；(b) 旧版本环境未经修改，回滚风险近乎零；(c) 故障时间最小化。

**例题2**：设计一个功能开关支持新推荐算法的灰度发布。需要哪些组件？

**解答**：(1) 配置中心（如 Consul/Etcd/LaunchDarkly）：存储开关状态；(2) 运行时开关评估：应用启动时从配置中心拉取开关规则，或在请求级别动态评估；(3) 分流逻辑：按用户 ID 哈希取模将 10% 用户路由到新算法，其余走旧算法；(4) 监控和对比：新老算法分别在指标平台（如 Grafana）上展示推荐点击率、转化率；(5) 紧急关闭：运维开关允许在检测到异常时一键切回 0% 新算法。必要时支持"粘性"——同一用户在整个实验期间始终看到同一版本。
