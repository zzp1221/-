## 核心定义

软件安全（Software Security）是使软件在面对恶意攻击时继续正确运行的工程实践。安全不是可事后添加的功能——它必须"内建"（Built-in）在软件架构和开发过程中。安全开发生命周期（Secure SDLC / Microsoft SDL）将安全活动嵌入传统 SDLC 各阶段：(a) 需求阶段：安全需求分析、隐私影响评估；(b) 设计阶段：威胁建模（Threat Modeling，如 STRIDE 方法）、安全架构评审；(c) 实现阶段：安全编码规范（输入验证、输出编码、参数化查询）、静态代码安全分析（SAST, Fortify/Checkmarx/SonarQube）；(d) 测试阶段：动态安全测试（DAST）、渗透测试（Penetration Testing）、模糊测试（Fuzzing）；(e) 部署阶段：安全配置审查、密钥管理；(f) 运维阶段：安全监控与事件响应（SIEM）、漏洞管理和补丁策略。

**威胁建模**（Threat Modeling）是系统化地识别、枚举和优先排序潜在安全威胁的过程。STRIDE 是最常用的威胁分类模型：Spoofing（身份欺骗）、Tampering（数据篡改）、Repudiation（否认）、Information Disclosure（信息泄露）、Denial of Service（拒绝服务）、Elevation of Privilege（权限提升）。结合数据流图（DFD），逐元素分析每种威胁类型。

**OWASP Top 10**：Web 应用最关键的十大安全风险：(1) 失效的访问控制；(2) 加密失败；(3) 注入（SQL注入、命令注入等）；(4) 不安全的设计；(5) 安全配置错误；(6) 易受攻击和过时的组件；(7) 身份认证和授权失败；(8) 软件和数据完整性失败；(9) 安全日志和监控失败；(10) 服务端请求伪造（SSRF）。

## 关键结论

- 安全三要素（CIA Triad）：机密性（Confidentiality）、完整性（Integrity）、可用性（Availability）
- 安全设计原则：最小权限、纵深防御（Defense in Depth）、默认安全（Secure by Default）、不要信任用户输入、保持简单（KISS）、开放设计（安全性不依赖保密设计本身）
- 安全不是二进制（安全 vs 不安全）——安全是一个风险管理过程
- 通过模糊测试（随机/半随机输入轰炸程序）可发现许多手工测试难以触发的安全漏洞
- 左移安全（Shift Left）：将安全活动尽早移到开发早期
- 漏洞管理生命周期：发现 → 分类（CVSS 评分）→ 修复 → 验证 → 披露

## 易错点

1. 仅依赖边界防火墙而忽视应用层安全——"城堡外墙很厚但城门大开"
2. 密码学中使用自制算法——"不要自己发明密码"（Schneier 定律），应使用经过验证的标准库
3. 在生产环境中开启调试模式或详细错误信息——这等于给攻击者泄露系统内部信息

## 例题

**例题1**：为"用户登录"功能进行威胁建模（STRIDE）。

**解答**：
- S(Spoofing)：攻击者使用泄露的密码冒充合法用户 → 缓解：多因素认证、验证码
- T(Tampering)：中途修改登录请求 → 缓解：HTTPS 全程加密
- R(Repudiation)：用户否认曾登录 → 缓解：完整的审计日志
- I(Information Disclosure)：错误消息泄露"用户名不存在" → 缓解：统一错误提示"用户名或密码错误"
- D(Denial of Service)：暴力密码尝试 → 缓解：账户锁定策略、速率限制
- E(Elevation of Privilege)：普通用户设法获取管理权限 → 缓解：严格的角色权限控制

**例题2**：如何防止 SQL 注入攻击？

**解答**：
1. 始终使用参数化查询（Prepared Statement）：`SELECT * FROM users WHERE id = ?`（而非字符串拼接）
2. ORM 框架通常内置防注入保护
3. 如需使用动态表名/字段名（无法参数化），使用白名单验证输入与预期的表名/字段名集合对比
4. 对数据库账户使用最小权限原则（应用的 DB 账户不应有 DROP TABLE 权限）
5. WAF（Web Application Firewall）作为额外防御层
