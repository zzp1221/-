## 核心定义

数据一致性实战是将一致性理论应用于实际业务场景的技术实践。在真实的分布式系统中，完全的强一致性往往代价太高，需要根据业务特点选择合适的一致性策略。

**缓存一致性**是最高频的一致性问题。缓存（如Redis）和数据库之间的数据可能不一致。

**Cache Aside Pattern**（旁路缓存模式）：
- **读**：先读缓存，未命中则读数据库，然后写入缓存
- **写**：先更新数据库，再删除缓存（而非更新缓存）
- **为什么删除而不是更新缓存**：避免并发写入时的缓存脏数据；缓存值可能涉及复杂计算

**缓存与数据库的一致性问题**：
- **先更新数据库，再删除缓存**：可能出现短暂不一致（数据库已更新，缓存未删除）
- **先删除缓存，再更新数据库**：可能出现更严重的不一致（删除缓存后，读请求从数据库读到旧值并写入缓存）
- **延迟双删**：先删缓存 → 更新数据库 → 延迟一段时间 → 再删缓存

**数据库主从一致性**：
- **异步复制**：主从之间有复制延迟，从库可能读到旧数据
- **强制读主**：写操作后的一段时间内，读操作走主库
- **半同步复制**：至少一个从库确认后才返回

**跨服务一致性**：
- 使用消息队列实现最终一致性
- 使用分布式事务（TCC、Saga）实现强一致性
- 使用对账机制修复不一致

## 关键结论

- **Cache Aside Pattern**是最常用的缓存策略，先更新数据库再删除缓存
- 缓存一致性没有完美方案——最终一致性是大多数场景的合理选择
- **延迟双删**可以减少不一致窗口，但不能完全消除
- **对账**是最终一致性的兜底机制——定期比对数据，修复不一致
- 分布式系统中的一致性问题往往需要**多层保障**：缓存策略 + 消息队列 + 对账

## 易错点

1. **先删缓存再更新数据库**：这会导致严重的并发问题——删除缓存后，读请求从数据库读到旧值并写入缓存
2. **缓存更新而非删除**：更新缓存可能导致并发写入时的脏数据。删除缓存更安全
3. **忽视缓存穿透和雪崩**：缓存穿透（查询不存在的数据）、缓存雪崩（大量缓存同时过期）也会影响一致性

## 例题

**题目**：一个电商系统的商品服务，数据库中商品价格为100元，缓存中也是100元。现在有两个并发请求：
- 请求A：更新价格为120元
- 请求B：读取价格

使用Cache Aside Pattern，可能出现什么不一致？如何解决？

**解答**：

**可能的不一致场景**：

**场景1：先更新数据库，再删除缓存**
```
T1: 请求A更新数据库价格为120元
T2: 请求B读取缓存，得到100元（旧值）
T3: 请求A删除缓存
```
结果：请求B读到了旧值100元，但这是短暂的——下次读取会从数据库得到新值

**场景2：先删除缓存，再更新数据库**
```
T1: 请求A删除缓存
T2: 请求B读取缓存未命中，从数据库读取100元
T3: 请求B将100元写入缓存
T4: 请求A更新数据库为120元
```
结果：缓存中是100元，数据库中是120元——**严重不一致**，且会持续

**解决方案**：

**方案一：延迟双删**
```python
def update_price(new_price):
    redis.delete("product_price")  # 1. 先删缓存
    db.update("UPDATE products SET price = ?", new_price)  # 2. 更新数据库
    time.sleep(0.5)  # 3. 延迟（大于一次读请求的耗时）
    redis.delete("product_price")  # 4. 再删缓存
```

**方案二：消息队列异步删除**
```python
def update_price(new_price):
    db.update("UPDATE products SET price = ?", new_price)
    mq.send("cache_invalidation", {"key": "product_price"})  # 发送失效消息
```

**方案三：使用分布式锁**
```python
def update_price(new_price):
    lock.acquire("product_price")  # 加锁
    db.update("UPDATE products SET price = ?", new_price)
    redis.delete("product_price")
    lock.release("product_price")  # 释放锁

def get_price():
    lock.acquire_read("product_price")  # 读锁
    price = redis.get("product_price")
    if not price:
        price = db.query("SELECT price FROM products")
        redis.set("product_price", price)
    lock.release_read("product_price")
    return price
```

**推荐**：大多数场景使用**方案一**（延迟双删），简单有效。高一致性要求使用**方案三**（分布式锁）。
