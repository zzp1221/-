## 核心定义

云原生（Cloud Native）是一种构建和运行应用程序的方法，充分利用云计算的优势。CNCF（Cloud Native Computing Foundation）的定义是：云原生技术使组织能够在公有云、私有云和混合云等现代动态环境中，构建和运行可弹性扩展的应用。

**核心技术栈**：
- **容器化（Containerization）**：使用Docker将应用及其依赖打包为轻量级容器，保证环境一致性
- **容器编排（Container Orchestration）**：使用Kubernetes自动化容器的部署、扩缩容、负载均衡和故障恢复
- **微服务（Microservices）**：将单体应用拆分为独立部署的小服务，每个服务独立开发、部署和扩展
- **DevOps/CI/CD**：持续集成和持续交付，自动化构建、测试和部署流程
- **可观测性（Observability）**：Metrics + Traces + Logs的统一监控体系

**Kubernetes核心概念**：
- **Pod**：最小部署单元，包含一个或多个容器
- **Service**：为Pod提供稳定的网络端点和负载均衡
- **Deployment**：声明式管理Pod的副本数和更新策略
- **ConfigMap/Secret**：配置和密钥管理
- **Ingress**：HTTP路由和负载均衡

**Twelve-Factor App**：云原生应用的12条最佳实践，包括：代码库一份、显式声明依赖、配置存储在环境变量、端口绑定、无状态进程等。

## 关键结论

- 云原生的本质是**面向失败设计**——假设一切都会失败，通过自动化和冗余保证系统可用
- **不可变基础设施**：容器镜像一旦构建就不修改，更新通过替换而非修改
- **声明式API**：Kubernetes使用声明式配置（YAML），描述期望状态，系统自动达成
- **服务网格**是云原生网络通信的演进方向——将网络逻辑从应用中剥离
- **GitOps**：以Git仓库为唯一真实来源，通过Pull Request管理基础设施变更

## 易错点

1. **将云原生等同于Kubernetes**：Kubernetes是云原生的核心组件，但云原生还包括容器化、微服务、DevOps等
2. **忽视可观测性**：在微服务架构中，没有可观测性就无法调试和运维
3. **过度微服务化**：小团队不应过度拆分微服务，会增加运维复杂度

## 例题

**题目**：某公司计划将单体应用迁移到云原生架构。当前系统：一个Java单体应用，MySQL数据库，部署在物理服务器上。请设计迁移方案。

**解答**：

**迁移方案**：

**Phase 1：容器化**
- 将Java应用打包为Docker镜像
- 使用Docker Compose在开发环境测试
- 建立CI/CD流水线（自动构建镜像、运行测试）

**Phase 2：引入Kubernetes**
- 部署Kubernetes集群（如使用云厂商的托管K8s）
- 将应用部署为Deployment
- 使用Service暴露应用
- 配置HPA（水平自动扩缩容）

**Phase 3：数据库迁移**
- MySQL迁移到云数据库（如RDS）或使用Operator管理
- 引入缓存层（Redis）
- 配置数据库读写分离

**Phase 4：微服务拆分**
- 识别独立的业务模块（如用户、订单、商品）
- 逐步拆分为独立服务
- 引入API网关统一入口
- 引入服务发现（如Consul）

**Phase 5：可观测性**
- 部署Prometheus + Grafana（Metrics）
- 部署Jaeger（Traces）
- 部署EFK（Logs）
- 配置告警规则

**架构图**：
```
用户 → Ingress → API Gateway → 微服务集群（K8s）
                                   ↓
                              云数据库（MySQL）
                              缓存（Redis）
                              消息队列（Kafka）
```
