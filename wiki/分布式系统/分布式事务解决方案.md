## 核心定义

分布式事务是跨多个服务或数据库的事务，需要保证**原子性**——所有参与者要么全部成功，要么全部回滚。在微服务架构中，每个服务有自己的数据库，传统的ACID事务无法跨越服务边界。

**主流解决方案**：

**两阶段提交（2PC）**：协调者先Prepare再Commit。强一致但阻塞性能差。适合数据库层面的分布式事务。

**TCC（Try-Confirm-Cancel）**：
- **Try**：预留资源（如冻结库存）
- **Confirm**：确认操作（如扣减冻结的库存）
- **Cancel**：释放预留（如解冻库存）
- 优点：业务层面的补偿，灵活性高
- 缺点：代码侵入性强，需要三个接口

**Saga模式**：将事务拆分为一系列本地事务+补偿事务。详见[Saga模式与长事务]。

**本地消息表（Local Message Table）**：
1. 业务操作和消息写入同一个本地事务
2. 后台任务轮询消息表，发送到MQ
3. 消费者处理消息，实现最终一致性
- 优点：实现简单，可靠性高
- 缺点：最终一致性，有延迟

**最大努力通知（Best Effort Notification）**：
- 通过MQ通知对方，失败后重试一定次数
- 适用于对一致性要求不高的场景（如支付结果通知）

**事务消息（Transactional Message）**：
- RocketMQ支持事务消息：先发送半消息，本地事务成功后提交消息，失败后回滚
- 实现了本地事务和消息发送的原子性

## 关键结论

- 分布式事务的方案选择取决于**一致性要求**和**性能要求**的权衡
- **TCC**适合对一致性要求高、业务逻辑简单的场景（如金融转账）
- **本地消息表**是最实用的最终一致性方案，实现简单且可靠
- **Saga**适合长事务和跨服务编排，但需要设计补偿事务
- 实际系统中，大多数场景可以接受**最终一致性**，只有少数场景需要强一致性

## 易错点

1. **过度使用分布式事务**：很多场景可以通过业务设计避免分布式事务（如将相关操作放在同一服务）
2. **TCC的空回滚问题**：Try请求未到达（如网络超时），但Cancel请求先到达。需要判断是否执行过Try
3. **Saga的中间状态可见性**：Saga执行过程中，其他事务可以看到中间状态。需要通过语义锁处理

## 例题

**题目**：一个电商系统的下单流程涉及三个服务：订单服务、库存服务、账户服务。请分别使用TCC和本地消息表实现分布式事务，并比较优劣。

**解答**：

**方案一：TCC实现**

```
下单流程：
1. Try阶段：
   - 订单服务：创建订单（状态=待确认）
   - 库存服务：冻结库存（available -= 1, frozen += 1）
   - 账户服务：冻结余额（balance -= 100, frozen += 100）

2. Confirm阶段（所有Try成功）：
   - 订单服务：订单状态=已确认
   - 库存服务：扣减冻结库存（frozen -= 1）
   - 账户服务：扣减冻结余额（frozen -= 100）

3. Cancel阶段（任一Try失败）：
   - 订单服务：删除订单
   - 库存服务：解冻库存（available += 1, frozen -= 1）
   - 账户服务：解冻余额（balance += 100, frozen -= 100）
```

**方案二：本地消息表实现**

```
下单流程：
1. 订单服务：
   - 创建订单（状态=待处理）
   - 写入消息表：{topic: "order_created", payload: {orderId, productId, amount}}
   - 以上两步在同一个本地事务中

2. 后台任务：
   - 轮询消息表，发送消息到MQ
   - 消息发送成功后标记为已发送

3. 库存服务消费消息：
   - 扣减库存
   - 记录已处理的消息ID（幂等）

4. 账户服务消费消息：
   - 扣减余额
   - 记录已处理的消息ID（幂等）
```

**比较**：
| 维度 | TCC | 本地消息表 |
|------|-----|----------|
| 一致性 | 强一致 | 最终一致 |
| 性能 | 较低（3阶段） | 较高 |
| 代码侵入 | 高（3个接口） | 低 |
| 复杂度 | 高 | 中 |
| 适用场景 | 金融、库存 | 电商、物流 |
