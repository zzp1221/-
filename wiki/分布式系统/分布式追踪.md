## 核心定义

分布式追踪（Distributed Tracing）是在微服务架构中追踪请求完整调用链路的技术。一个用户请求可能经过多个服务，分布式追踪将这些分散的日志和指标串联起来，形成完整的调用视图。

**核心概念**（基于Google Dapper论文）：
- **Trace**：一个请求的完整调用链，由多个Span组成
- **Span**：一个操作的执行单元，包含：操作名、开始/结束时间、标签（Tags）、日志（Logs）、上下文（Context）
- **Trace ID**：全局唯一标识，贯穿整个调用链
- **Span ID**：单个Span的唯一标识
- **Parent Span ID**：指向父Span，形成树形结构

**传播机制**：Trace上下文通过**HTTP Header**（如`traceparent`、`tracestate`）或**gRPC Metadata**在服务间传播。每个服务在收到请求时提取上下文，在发送请求时注入上下文。

**采样策略**：
- **全量采样**：记录所有请求（开销大，适合低流量调试）
- **概率采样**：按比例采样（如1%），降低开销
- **自适应采样**：根据流量和错误率动态调整采样率
- **尾部采样（Tail-based Sampling）**：先收集完整Trace，再决定是否保留（如保留所有错误Trace）

**主流实现**：
- **Jaeger**：Uber开源，支持OpenTracing，CNCF项目
- **Zipkin**：Twitter开源，类似Dapper的设计
- **OpenTelemetry**：统一了OpenTracing和OpenCensus，提供Traces+Metrics+Logs的统一方案

## 关键结论

- 分布式追踪的核心是**上下文传播**——通过Header在服务间传递Trace ID和Span ID
- **采样**是生产环境的必要策略——全量追踪会带来巨大的存储和性能开销
- **OpenTelemetry**已成为可观测性的事实标准，统一了Traces、Metrics和Logs
- 追踪数据的价值不仅在于**调试**，还用于**性能分析**、**依赖关系可视化**和**SLA监控**
- 追踪系统的存储通常使用**列式存储**（如Cassandra）——因为查询模式以Trace ID为主

## 易错点

1. **上下文传播丢失**：异步调用（如消息队列、线程池）中容易丢失Trace上下文。需要在异步边界手动传播
2. **采样率过高导致存储爆炸**：高流量系统全量采样会产生TB级追踪数据。建议使用自适应采样
3. **Span粒度不当**：Span太细导致数据量大、分析困难，Span太粗导致信息不足

## 例题

**题目**：一个电商下单请求经过以下服务：API Gateway → Order Service → Inventory Service → Payment Service。请设计追踪方案，包括Span的划分和上下文传播。

**解答**：

**Trace结构**：

```
Trace ID: abc123
│
├── Span 1: API Gateway (root span)
│   ├── Operation: POST /orders
│   ├── Duration: 200ms
│   └── Tags: http.method=POST, http.url=/orders
│
│   ├── Span 2: Order Service
│   │   ├── Operation: createOrder
│   │   ├── Duration: 180ms
│   │   └── Tags: order.id=ORD-001
│   │
│   │   ├── Span 3: Inventory Service
│   │   │   ├── Operation: deductStock
│   │   │   ├── Duration: 50ms
│   │   │   └── Tags: product.id=SKU-123, quantity=2
│   │   │
│   │   └── Span 4: Payment Service
│   │       ├── Operation: processPayment
│   │       ├── Duration: 100ms
│   │       └── Tags: amount=99.99, currency=CNY
```

**上下文传播**：

1. **API Gateway → Order Service**（HTTP调用）：
   - 注入Header：`traceparent: 00-abc123-span1-01`
   - Order Service提取Header，创建Span 2（parent=Span 1）

2. **Order Service → Inventory Service**（HTTP/gRPC调用）：
   - 注入Header：`traceparent: 00-abc123-span2-01`
   - Inventory Service提取Header，创建Span 3（parent=Span 2）

3. **异步消息传播**（如果Payment通过MQ调用）：
   - 将Trace上下文写入消息Header
   - Payment Service从消息中提取上下文

**采样策略**：
- 错误请求：100%采样
- 正常请求：1%采样
- 慢请求（>1秒）：100%采样
