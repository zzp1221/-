## 核心定义

错误处理策略（Error Handling Strategy）是语言和框架定义的程序对意外条件的检测、传播和恢复的约定体系。错误处理的常见策略有五：异常（Exception）、返回值/错误码（Error Codes）、Result/Either 类型、断言和终止（Fail Fast）、条件恢复（Common Lisp Condition System / Restart）。

### 1. 异常（Exception）

将正常控制流和错误处理完全分离——正常逻辑在 try 块中写，错误处理在 catch 块（可在调用栈上层），避免正常控制流被错误检查污染。

**优势**：正常代码路径清晰；错误不能被忽略（至少传播）；调用栈信息自动捕获便于调试。

**劣势**：隐式控制流（throw 后你不知道跳到哪里）；性能成本（抛出时收集调用栈昂贵）；类型系统无法标注函数可能抛出什么（Java 受检异常除外——但这也是最受争议的特性）。

适用语言：Java, C++, C#, Python, JavaScript。

### 2. 返回值/错误码（Error Code / Go Style）

函数返回 `(value, error)` 元组，成功时 error 为 nil，失败时 value 应该被忽略。由调用方显式检查错误并处理或传播。

**优势**：错误路径完全显示于常规代码中；无隐式控制流（控制流保持线性）；无性能开销创建异常对象和栈追踪。

**劣势**：样板代码（Go 的 `if err != nil { return err }` 在每个调用点重复）；容易忽略或默默处理错误（`val, _ := foo()`）。

适用语言：Go、C。

### 3. Result / Either 类型（函数式风格）

函数的返回类型在类型层面区分成功和失败路径——`Result<T, E>`（Rust）或 `Either<L, R>`（Haskell/Scala/Kotlin）。成功时返回 `Ok(value)`，失败时返回 `Err(error)`。使用模式匹配或链式方法（`map`, `and_then`, `or_else`）处理错误。

**优势**：类型安全——调用方必须处理错误分支（编译时强制，Rust 语言强制）；可组合——错误处理可链式串联而非嵌套 if；可附加上下文——方法链中每个 `map_err` 可添加更多逻辑上下文信息。

**劣势**：学习曲线——需要对 Monad/Map/FlatMap 的理解；与异常互操作的复杂性（Rust 和 Java 协作需边界设计）。

适用语言：Rust（核心哲学）, Haskell, Scala, Swift (Result), Kotlin (Result), TypeScript (fp-ts Either)。

### 4. 断言（Assertion）和终止（Abort / Panic）

断言用于检测程序逻辑 bug——如 out-of-bounds index, divide by zero——这类错误应"不存在"在正确程序中。断言表示程序员的不变式预期已被破坏——立即终止（或触发调试器）是最合理的响应。

- C/C++：`assert(ptr != NULL)` 仅在 debug 构建中检查（release 中被禁掉——安全隐患）
- Java：`assert` 默认被 JVM 忽略——仅在 `-ea` 启用时才生效（不推荐用于生产环境检查）
- Python：`assert condition` 在运行时启用，但可通过 `-O` 优化标志禁用
- Rust：`panic!` 用于不可恢复错误——展开栈并清理资源（可通过设置 abort 改为直接终止）
- Go：`panic` / `recover`——通常在无法继续（cannot proceed）时 panic

**"断言 vs 错误"的区分**：
- 断言：针对程序 bug（不可能条件），终止是合适的行为
- 错误处理：针对预期可发生的运行时条件（文件不存在、网络超时），需要恢复路径

### 5. 防御性编程（Defensive Programming）vs 契约式编程（Design by Contract）

**防御性编程**：对外部输入做最大程度校验——包括参数、返回值、中间结果都要校验。适合对外 API（公开库代码）。

**契约式编程**（Bertrand Meyer 的 Eiffel）：每个函数有前置条件、后置条件和不变量。调用方保证前置条件，实现方保证后置条件——双方都遵守契约时无需过多防御。现代语言如 Rust 的 `require!` 标准库就是契约风格——"若违反 contract（如 index out of bounds）则 panic"。

## 关键结论

- 错误处理的选择不是审美——它深刻影响 API 设计、并发行为、性能特性和可维护性
- Result 类型正越来越流行——Swift、Kotlin、Scala 都采纳了 Result，TypeScript 在 fp-ts/Effect 生态中
- 最坏情况的错误处理：无声吞噬错误——`catch (Exception e) {}` 或 `val, _ := foo()`。让错误传播到有足够上下文的层去决定如何处理
- Rust 的问号操作符 `?` 是错误传播的优雅解决方案：跳出了显式样板又保持了类型安全

## 易错点

1. catch 后什么都不做（不管日志）——错误被完美隐藏，难以调试
2. 忘记受检异常的文档意义——Java 的 throws 声明在语义上是与返回值的契约一样重要的部分
3. 异步代码中的异常可能被丢失——Promise 的错误通过 reject 传播，需 `.catch` 或 `try/catch` 包围 await

## 例题

**例题1**：将以下 Go 错误处理的层层传递简化为更可组合的形式（假设 Go 支持 Rust-like `?` 风格的错误传播）。

```go
func processFile(path string) (string, error) {
    f, err := os.Open(path)
    if err != nil { return "", fmt.Errorf("open: %w", err) }
    defer f.Close()
    data, err := io.ReadAll(f)
    if err != nil { return "", fmt.Errorf("read: %w", err) }
    result, err := transform(data)
    if err != nil { return "", fmt.Errorf("transform: %w", err) }
    return result, nil
}
```

**解答**：Go 不支持 `?` 语法——这是设计选择。若在 Rust 中等价代码：
```rust
fn process_file(path: &str) -> Result<String, Error> {
    let data = std::fs::read_to_string(path)
        .context("open and read")?;
    let result = transform(&data)
        .context("transform")?;
    Ok(result)
}
```
`?` 在遇到 Err 时自动传播，`.context()` 为错误链追加当前操作的上下文信息——代码只关注成功路径。

**例题2**：在以下情况分别应使用哪种错误处理：

(a) 一个仅供内部工具使用的脚本（个人项目）；
(b) 一个航空控制系统的软件（人命关天）；
(c) 一个公开的 REST API 库。

**解答**：
(a) Go-style 或 Python 异常 —— 工具脚本的生命期短, 轻量处理是快速的。Python `try/except` 在关键点捕获即可。
(b) 多层级的错误恢复（Common Lisp 条件的哲学）—— 不崩溃。系统应区分 transient fault（可重试）和 permanent fault（需降级/旁路）。Erlang/Elixir 的监督树（容错）或 Rust 的显式错误路径保证所有可能的失败场景都已覆盖。
(c) Result 类型 + 清晰的错误类型层次 —— 公开 API 需要为每个可能的错误返回有文档的标准错误码和消息。Rust 和 Swift 的 Result、Kotlin/Scala 的 sealed class 错误类型最有可维护性——库的使用方明确知道可能出现的错误并能模式匹配它们。
