## 核心定义

编译型语言和解释型语言描述的是源代码转化为可执行形式的两种基本策略。在现实中，大多数现代语言采用了混合方案——这个区分已经不再二元对立，而是"何时做多少编译"的光谱。

### 编译型语言（Compiled Languages）

**传统编译**（AOT, Ahead-of-Time Compilation）：开发者在部署前使用编译器将整个源代码一次性翻译为机器码（或目标代码），生成独立的可执行二进制文件。编译过程在生产环境无需再发生——运行时直接执行机器码。

编译流程（以 GCC/LLVM 为例）：源代码 → 预处理 → 词法分析 → 语法分析 → 语义分析 → 中间表示（IR）生成 → 优化 → 目标代码生成 → 链接。

**优势**：
- 启动速度：直接执行机器码无 JIT 预热延迟
- 执行性能：编译器有充分时间作全局优化（跨函数内联、静态分析、全程序优化、链接时优化 LTO）
- 可发现错误前置：类型错误、语法错误等在编译阶段暴露
- 低资源消耗：不需要运行时编译器（代码少、启动快、内存占用低）——适合嵌入式

**劣势**：
- 修改-编译-运行周期长（开发迭代慢）
- 编译后的二进制文件绑定特定 CPU 和 OS 平台

**代表语言**：C、C++、Rust、Go、Haskell（GHC）

### 解释型语言（Interpreted Languages）

**纯解释**：解释器逐条读取源代码、解析并执行——无编译、无机器码。历史上 BASIC、早期的 LISP 和 APL 是真解释型。实际上极少现代语言是"纯解释"——大多数至少做编译到某种内部表示。

**优势**：直接运行——无需显式编译步骤，开发反馈快；跨平台——同一源码在不同平台上由解释器运行。

**劣势**：执行开销——解释开销通常是编译后执行的 1-100 倍；运行时错误（如拼写错误的变量名）只在代码到达时才暴露。

**代表语言**：Shell 脚本、早期 BASIC、MATLAB（部分）

### 混合方案（现代主流）

**字节码 + 虚拟机**（Bytecode VM）：编译器将源码编译为字节码（Bytecode, 中间表示），运行时由虚拟机解释或 JIT 编译执行。Java 的 `javac` → `.class` 字节码 → JVM；Python 的 `.pyc` 字节码 → CPython VM；C# 的 `csc` → CIL → CLR；JavaScript 的现代引擎（V8, JSC, SpiderMonkey）也会将 JS 编译为字节码。

**JIT 编译**（Just-In-Time Compilation）：在运行时将字节码（或热点路径）动态编译为机器码。JVM 的 HotSpot 将频繁运行的方法编译为优化的机器码——甚至可以在观察到运行时行为后执行针对性的优化（如多态内联缓存）。

**AOT + JIT 混合**：.NET的 ReadyToRun（AOT + 运行时 JIT 补齐）、Android ART（安装时编译 dex → 机器码）、GraalVM Native Image（AOT 编译 Java → 独立二进制）。

### 语言实现的选择权衡

| 实现策略 | 启动速度 | 峰值性能 | 开发周期 | 跨平台 | 内存占用 |
|---------|---------|---------|---------|--------|---------|
| AOT (C/Rust) | 极快 | 最高 | 慢 | 需编译 | 小 |
| JIT (Java/C#) | 中 | 高（预热后） | 中 | ✓ | 中 |
| 解释 (CPython) | 快 | 低 | 快 | ✓ | 中 |
| AOT+GC (Go) | 快 | 中高 | 中 | 需编译 | 中 |

### 高级编译器技术

- **LLVM 架构**：模块化编译框架——多种前端（Clang/C++, Rust, Swift）共享相同中间表示（LLVM IR）和后端优化及代码生成，支持多目标平台
- **PGO**（Profile-Guided Optimization）：先编译一个有仪表代码的版本运行并收集数据，再用数据指导第二次编译做针对性优化（分支预测、内联决策）
- **链接时优化**（LTO / Link-Time Optimization）：在链接阶段对整个程序做跨翻译单元的优化——传统编译仅优化单个 .o 文件

## 关键结论

- 编译/解释不是二分而是光谱——"多少工作在编译时做，多少在运行时做"
- 编译付出了"编译时间"换取"执行性能"——对用户期望的优化取决于应用（一次性分析 vs 持续运行的服务）
- "解释型语言慢"的刻板印象在现代变得复杂——JIT 可以将热点路径优化到接近 AOT 编译的速度
- Go 是"编译但感觉像解释"的独特案例——编译速度快（无沉重的模板展开）、二进制独立部署、但有 GC 在运行时

## 易错点

1. 将 Python 称为"编译型语言"因为 `.pyc` 文件——`.pyc` 是字节码缓存（非机器码），CPython 仍是解释执行的
2. 将 Java 称为"解释型"——标准的 HotSpot JVM 对热点代码执行 JIT 编译为机器码，非逐行解释
3. 混淆 JIT 和编译器优化——JIT 的优势不仅是"编译后缓存"而是"基于运行时观测的推测优化"

## 例题

**例题1**：比较 Go 和 Java 的执行模型差异——两者都编译但路径不同。

**解答**：
Go：源码 → go compiler (gc) → 单一静态链接的二进制机器码（AOT）。二进制包含 Go 运行时（GC、goroutine 调度器）。部署仅需复制单个文件。启动瞬时，内存足迹小。
Java：源码 → javac → `.class` 字节码 → JVM。JVM 首次以解释模式运行，识别热点 → C1（客户端编译器，快但较少优化）→ 热点升级 → C2（服务器编译器，深入优化）。启动需要 JVM 预热（通常 > 30 秒可达峰值性能），内存足迹较高。
选择：Go 适合 CLI 工具、微服务启动快速；Java 适合长期运行的服务器（JIT 充分预热后性能卓越）。

**例题2**：简述 WebAssembly（Wasm）与编译/解释谱系的关系。

**解答**：WebAssembly 是一个低级的、可移植的字节码格式——不是"编程语言"本身。C/Rust/Go 等源码编译为 `.wasm` 字节码（编译 + AOT），浏览器运行时将 Wasm 编译为机器码（JIT 或 AOT，取决于浏览器）。Wasm 设计为紧凑（文件小）、快速解码、快速编译——是介于"传统字节码"和"机器码"之间的独特层次。它的存在使得编译型语言输出的代码可在浏览器中以接近原生的性能运行。
