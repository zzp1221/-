## 核心定义

异常处理（Exception Handling）是编程语言的管理运行时错误（异常事件）的机制。异常是程序执行中出现的意外条件——除以零、文件不存在、网络连接失败、空指针引用等。异常处理的目标是：(a) 将错误处理逻辑与正常业务逻辑分离；(b) 在适当的抽象层级处理错误；(c) 防止程序在不一致状态继续执行；(d) 提供有意义的错误信息用于诊断。

**主要异常处理模型**：

**1. try-catch-finally 模型（Java/C++/C#/JavaScript/Python）**：
- `try` 块包含可能抛出异常的代码
- `catch` 块捕获匹配的异常类型并处理
- `finally` 块无论是否异常都会执行（资源释放）
- `throw` 向调用栈上层抛出让调用者处理

**受检异常 vs 未受检异常**（Java 特有设计）：受检异常（Checked Exception, extends Exception）——调用者必须在方法签名中声明 `throws` 或 try-catch 处理（编译时强制）；未受检异常（Unchecked Exception, extends RuntimeException）——调用者不需要显式声明或捕获（如 NullPointerException）。受检异常是 Java 争议最大的设计之一——它强制调用者处理可恢复的错误，但也导致滥用（catch 后什么都不做）和接口污染。

**2. Result/Either 模型（Rust/Go/Haskell/Scala）**：
不用异常机制进行错误处理。函数返回 `Result<T, E>`（Rust）或 `Either`（Haskell/Scala）或 `(value, error)` 元组（Go）。成功时返回 `Ok(value)`，失败时返回 `Err(error)`。调用方必须在编译时处理错误分支（Rust 的 `match`、`?` 运算符传播错误）。优势：错误处理是显示的且受到类型检查，错误不会意外逃逸再导致未捕获运行时崩溃。

**错误传播**（Error Propagation）：将错误从当前位置沿调用栈向上传递给能处理它的调用者。
- try-catch 模型中：不被 catch 的异常自动向上传播
- Rust 的 `?` 运算符：`let val = func()?;` 等价于 `let val = match func() { Ok(v) => v, Err(e) => return Err(e.into()) };`
- Go 的惯用模式：`if val, err := func(); err != nil { return err }`

**资源清理**（Resource Cleanup）：
- try-catch-finally 的 `finally` 块、Python 的 `with` 语句（上下文管理器）、C# 的 `using`、Java 的 try-with-resources
- RAII（C++）：通过对象析构函数自动释放资源——栈展开（stack unwinding）触发持有资源的对象的析构函数
- defer（Go）：延迟执行清理函数——`defer f.Close()` 在函数返回前执行

**异常安全的四个保证级别**（C++ 术语但适用于其他语言）：
1. 无抛出保证（No-throw guarantee）：操作决不抛出异常（析构函数、swap）
2. 强保证（Strong exception guarantee）：操作失败后程序状态回滚到操作前（原子提交）
3. 基本保证（Basic exception guarantee）：操作失败后程序保持有效状态（不变式不破裂，但状态可能已改变）
4. 无保证（No guarantee）：最严重——所有保证被弃权

## 关键结论

- 异常应该用于"意外条件"而非"预期的控制流"——抛出异常的成本远高于返回错误码（栈收集、对象分配）
- 异常处理不应静默——catch 并什么都不做（空 catch 块）是危险的——隐藏了程序已进入不一致状态的信号
- finally 块不一定执行——JVM崩溃、`System.exit()`、守护线程被强制终止可 bypass finally
- Result 类型相比异常的优势：错误路径是显示的、类型系统强制覆盖、性能成本不含栈追踪
- 最安全的错误处理发生在编译时——静态类型系统甚至阻止了错误的可能性

## 易错点

1. 捕获过于宽泛的异常——`catch (Exception e) {}` 捕获了 RuntimeException（如 programming bug 导致的 NPE）应该尽早暴露而非隐藏
2. 在 finally 块中使用 return——会覆盖 try 块中的 return 值和抛出的异常（Java 警告）
3. 抛出原始异常类型——Java 应抛 `IllegalArgumentException` 而非 `Exception`；Python 应抛 `ValueError` 而非 `Exception`
4. Go 中忘记检查 error 就使用结果——`val, _ := func()` 丢弃 error 等于无视潜在错误

## 例题

**例题1**：以下 Java 代码有何问题？如何改进？

```java
try {
    connection = getConnection();
    stmt = connection.prepareStatement(sql);
    rs = stmt.executeQuery();
    while (rs.next()) { /* 处理 */ }
} catch (Exception e) {
    e.printStackTrace();
} finally {
    rs.close();
    stmt.close();
    connection.close();
}
```

**解答**：问题：(a) 过于宽泛的捕获 `Exception`；(b) `finally` 中的 `close()` 各自可能抛 NullPointerException（若 `connection` 创建失败则 `connection` 是 null）；(c) `rs/stmt/connection.close()` 各可能抛 SQLException 导致后面的资源未关闭。改进：使用 try-with-resources（Java 7+）：
```java
try (Connection conn = getConnection();
     PreparedStatement stmt = conn.prepareStatement(sql);
     ResultSet rs = stmt.executeQuery()) {
    while (rs.next()) { /* 处理 */ }
} catch (SQLException e) {
    throw new DataAccessException("Query failed", e);
}
```

**例题2**：Rust 的 `?` 运算符如何实现错误传播？等价代码是什么？

**解答**：
```rust
fn read_config() -> Result<Config, ConfigError> {
    let file = File::open("config.toml")?;  // 若 Err 则立即返回
    let content = read_to_string(file)?;
    let config = toml::from_str(&content)?;
    Ok(config)
}
```
等价的 match 写法：
```rust
fn read_config() -> Result<Config, ConfigError> {
    let file = match File::open("config.toml") {
        Ok(f) => f,
        Err(e) => return Err(ConfigError::from(e)),
    };
    // ... 类似
}
```
`?` 运算符还会通过 `From` trait 自动完成错误类型转换（`e.into()`），如果调用函数的错误类型实现了 `From<当前错误类型>` 则转换是透明的。
