## 核心定义

内存管理（Memory Management）是控制程序对计算机内存的分配和释放的机制。不同的语言提供不同级别的内存管理抽象，从完全手动（C/C++）到完全自动（Java/Python/Go/Haskell），以及提供编译时确定性管理的新型方式（Rust 的所有权系统）。

**内存区域模型**（Memory Layout）：

**栈（Stack）**：LIFO 结构，存储函数的调用帧（Activation Record / Stack Frame）——局部变量、返回地址、帧指针。特点是：(a) 超快分配和释放（仅移动栈指针）；(b) 数据大小必须在编译时已知（或语言支持变长数组）；(c) 后进先出的生命期——函数调用返回后帧即释放。栈上变量的引用不能用于比变量自身生命期长的上下文（悬挂引用）。多线程中每个线程有独立栈（通常 1-8 MB）——避免过深的递归或过大局部变量导致栈溢出（Stack Overflow）。

**堆（Heap）**：动态分配的全局可访问内存——不受作用域限制。由内存分配器管理（malloc/free, allocator）。分配和释放成本高于栈。分配的生命期由程序员控制（手动）或由运行时管理（GC）。多个线程共享同一堆空间——需同步机制保护并发访问。

**静态/全局区域**：存储全局变量、static 变量、字符串字面量——程序启动时一次性分配，结束时释放。

### 内存管理策略

**1. 手动管理**（C/C++）：程序员显式调用 `malloc/new` 分配、`free/delete` 释放。优势是最大控制和零运行时开销。劣势是需要极高纪律——内存泄漏（忘记释放）、悬空指针（释放后使用）、双重释放都是常见且难以调试的 bug。现代 C++ 通过 RAII（Resource Acquisition Is Initialization）使用智能指针（unique_ptr, shared_ptr, weak_ptr）使手动管理更安全——资源生命期绑定到对象生命期。

**2. 垃圾回收**（Garbage Collection, GC）：运行时自动识别和回收不再可达的对象。

GC 基本算法：
- **标记-清除**（Mark-Sweep）：从根集（全局变量、栈、寄存器）出发标记所有可达对象，然后清除未标记对象。简单但有碎片化问题。
- **标记-复制**（Copying / Semi-space）：将存活对象从一半堆复制到另一半（紧凑放置），清扫旧半区。无碎片但需要双倍内存。
- **标记-压缩**（Mark-Compact）：标记可达对象后将存活对象移到一端——兼顾无碎片和节约内存。
- **引用计数**（Reference Counting, RC）：每个对象维护被引用次数，计数为 0 时立即回收。优点：确定性回收，无 GC 停顿。缺点：无法回收循环引用（需要额外 GC 或弱引用来破环）——Python 的组合方案：RC 为主 + 循环检测 GC 定期扫描。Swift 使用 ARC（Automatic Reference Counting，编译时插入的 RC 操作）实现确定性回收但需要 `weak` 引用破环。
- **分代回收**（Generational GC）：基于"弱代假说"（Most objects die young）——新生代（Young Gen / Nursery）频繁收集（Minor GC，使用 Copying）；幸存对象晋升到老年代（Old Gen），较少收集（Major GC）。Java HotSpot、.NET CLR、V8 均用分代回收。
- **追踪式 vs 引用计数**：RC 回收是增量的（每次引用变动时执行少量工作），延迟分散；追踪 GC 是批量的，可能在任意时刻导致停顿（STW）。现代 GC 通过并发/并行/增量技术（Java ZGC/Shenandoah、Go GC）将停顿限制在 < 1ms。

**3. 所有权系统**（Rust 的独特方案）：不用 GC 也无需手动管理——编译时通过所有权规则静态确定资源何时释放。核心规则：(a) 每个值有唯一的拥有者（Owner）；(b) 任何时刻，要么有一个可变引用，要么有多个不可变引用，不能同时；(c) 引用必须始终有效（生命期分析，Lifetime）。所有权系统使 Rust 在同一代码中防止悬空指针、数据竞争、双重释放——且所有这些检查在编译时发生，零运行时开销。

### 内存泄漏的常见场景

- 忘记释放内存（C/C++）
- 循环引用阻止 GC/RC 回收（Java/Python 中两个对象相互引用且均无外部引用——分代 GC 可以收集，但 RC 不行）
- 缓存无限增长（无容量限制或 TTL 过期策略）
- 闭包中的意外保留——闭包持有了大对象的引用而生命期超过预期
- 事件监听器未取消注册（Event Listener Leak）——注册后的回调持有对象的强引用
- 线程局部存储没有被清空

## 关键结论

- 没有任何内存管理策略是"最佳"的——是在吞吐量、延迟、内存开销、开发易用性之间的权衡
- 引用计数是确定性的（变慢但可预测），追踪 GC 是批量的（大部分时间快但偶尔暂停）
- Rust 的所有权系统证明：编译时内存安全不一定需要 GC——可以在编译时保证内存安全且零运行时成本
- "内存泄漏"不一定表现为物理内存增长——也可能是逻辑泄漏（对象不可达但还未被 GC 回收之前占用大量内存）
- 内存管理策略影响语言的设计和编程风格——Rust 的所有权规则阻止了传统的图结构（需要用 Rc/RefCell 等显式构造）

## 易错点

1. 将栈地址返回给调用者：`int* f() { int x = 5; return &x; }` —— 局部变量在函数返回后无效，返回的地址是悬空指针
2. C++ 中 `shared_ptr` 的循环引用导致内存泄漏——需要用 `weak_ptr` 打断循环
3. Go 中 `defer` 释放资源时忘记检查 nil——`defer resp.Body.Close()` 应该在 nil 检查之后
4. JavaScript 的闭包意外保留大对象——`const large = new Array(1e7); setInterval(() => { console.log(large.length); }, 1000);` 使 `large` 永远不会被 GC

## 例题

**例题1**：以下 Go 代码存在什么内存隐患？

```go
func process(data []string) {
    ch := make(chan int)
    for _, s := range data {
        go func() {
            ch <- len(s)
        }()
    }
    for range data { <-ch }
}
```

**解答**：Go 的 for-range 循环中 `s` 变量在每次迭代中共享——goroutine 闭包捕获了 `s` 的地址而非值。当 goroutine 执行时循环可能已结束，`s` 为最后一个元素（或值未定义）。修复：在循环体内复制 `s`——`go func(s string) { ch <- len(s) }(s)`。

**例题2**：对比 Java G1 和 Go GC 的不同设计目标和技术策略。

**解答**：
设计目标差异：(a) Java G1 追求高吞吐量和可预测的低停顿（<200ms），适用于大型堆（数百 GB）的服务器应用；(b) Go GC 追求非常低的延迟（<1ms）和可接受的吞吐量牺牲（当前约 25% CPU 给 GC），适用于微服务和低延迟服务。
技术差异：(a) Go 使用并发标记 + 三色标记 + 写屏障（并发 GC），同时进行标记和清扫（非 STW）；(b) Java G1 将堆分为 Region 独立进行回收（避免整堆 STW），年轻代使用 Copying，老年代使用 Mark-Compact；(c) 关键差异：Go GC 仅需维护写屏障（标记阶段），Java G1 需要读写屏障（维护 Remember Set）；导致 Go 的寄存器分配压力低于 Java，但吞吐量低于 Java。
选择：对延迟敏感（如 API Gateway）选 Go；对吞吐量敏感和大堆选调优后的 Java（ZGC 已达亚毫秒）。
