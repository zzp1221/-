## 核心定义

函数式编程（FP, Functional Programming）是将计算建模为**数学函数的求值**并**避免改变状态和可变数据**的编程范式。在 FP 中，函数是"一等公民"（First-Class Citizen）——函数可以赋值给变量、作为参数传递、作为返回值。FP 源于 Lambda 演算（Alonzo Church, 1930s），强调"做什么"而非"如何做"。

**核心原则**：

1. **纯函数**（Pure Function）：给定相同的输入总是返回相同的输出，且不产生可观察的副作用（side effect）——不修改全局变量、不进行 I/O、不改变传入的参数。纯函数具有**引用透明性**（Referential Transparency）：函数调用可被其结果值替换而不改变程序的含义——使得推理、测试、并行化和组合更为安全。

2. **不可变性**（Immutability）：数据一旦创建就不能被改变。对数据的"修改"操作实际上是创建一个新的数据拷贝——这避免了共享可变状态引发的大部分并发问题。不可变数据结构通常使用"结构共享"（Structural Sharing）来避免内存浪费——如 Clojure/Persistent.js 的持久化数据结构（修改后仅重建路径节点，大部分树结构被共享）。

3. **高阶函数**（Higher-Order Function, HOF）：接受一个或多个函数作为参数，和/或返回一个函数。经典的高阶函数：
   - `map`：`[a] → (a → b) → [b]`，对每个元素应用函数，返回新集合
   - `filter`：`[a] → (a → bool) → [a]`，保留满足谓词的元素
   - `reduce/fold`：`[a] → b → (b → a → b) → b`，将集合归约为单个值
   - `compose`：`(b → c) → (a → b) → (a → c)`，将多个函数组合成一个
   - `flatMap/bind`：`M a → (a → M b) → M b`，Monad 的绑定操作

4. **柯里化**（Currying）：将一个接收多个参数的函数转化为一系列接收单一参数的函数。`f(a, b, c)` → `f(a)(b)(c)`。柯里化使部分应用（Partial Application）变得自然——`add(1)` 返回一个"加 1"的新函数。

5. **模式匹配和代数数据类型**（Pattern Matching + ADT）：描述和操作代数数据类型的方式——根据数据的结构匹配合适的处理分支。Haskell/Rust/Ocaml 的模式匹配在编译时检查"你是否覆盖了所有情况"。

6. **惰性求值**（Lazy Evaluation）：表达式直到其结果真正被需要时才被计算（Haskell 默认惰性，其他语言通过 `lazy` 关键字或 Generator 支持）。惰性允许操作无限数据结构——`take 5 (all_primes)` 只生成前 5 个质数。

7. **Monad**（单子）：一个 Monad 是对类型构造子的抽象，定义了链式操作的方式。每个 Monad 有两项操作：`return/unit`（将纯值包裹入 monadic 上下文）、`bind >>=`（在 monadic 上下文中应用函数并产出新的 monadic 值）。常见 Monad：Maybe/Option（可能失败的处理）、Either/Result（错误处理）、List（非确定性计算）、IO（纯函数式语言的输入输出）、Reader/State/Writer。Monad 在 FP 中是"可编程的分号"——控制如何链式组合计算。

**函数式编程在现代语言中的体现**：
- Python：`map`, `filter`, `lambda`, `functools.reduce`, list comprehensions, `dataclass`（不可变可选）
- Java 8+：`Stream<T>`, `Optional<T>`, lambda, method reference (`::`)
- JavaScript/TypeScript：`Array.prototype.map/filter/reduce`, arrow function, 不可变库（Immutable.js, Immer）
- Rust：Iterator trait, `Option`, `Result`, 模式匹配, 不可变默认绑定, `map`/`and_then`/`or_else`
- Go：函数为第一等公民（但无泛型支持不方便 map/filter, 1.18 后改善），通过 struct 和不可变 copy 模拟不可变性

## 关键结论

- FP 的核心优势不是"写更少行数"——而是代码行为更可预测、更易测试和并行化
- 纯函数使得"等式推理"（Equational Reasoning）成为可能——你可以仅通过看代码的逻辑推导它的行为，不受隐晦的外部状态影响
- 不可变性消除了时间维度的依赖——无需追踪"在何时写/在何时读"，多线程下数据竞争消失
- "函数式"不是"0 副作用"——而是副作用在系统的边界被隔离和控制（Effect System / IO Monad）
- Monad 是函数式编程的"设计模式"——通常引入学习曲线最高但也是收益最大的概念

## 易错点

1. 在 Python 等非惰性语言中使用 map 但忘记它是惰性的——`map(f, [1,2,3])` 返回的是 map 对象（迭代器），不产生值直到消费（`list(map(f, [1,2,3]))`）
2. Lambda 过多丧失可读性——多行 lambda 通常提示应该定义一个命名函数
3. 柯里化和偏应用混淆——柯里化将多元函数分解为一系列一元函数；偏应用是固定若干个参数返回剩余参数的函数
4. 不可变数据做频繁的追加/删除而不用持久化数据结构——导致 O(n) 复制，用错的工具失去高性能

## 例题

**例题1**：用 Java Stream 和 Haskell 两种方式实现：从一组订单中获取每个用户的总消费额，只保留消费超过 100 的用户，按消费降序排列。

**解答**：
Java Stream：
```java
Map<String, Double> result = orders.stream()
    .collect(groupingBy(Order::getUser, summingDouble(Order::getAmount)))
    .entrySet().stream()
    .filter(e -> e.getValue() > 100)
    .sorted(Map.Entry.<String,Double>comparingByValue().reversed())
    .collect(toMap(Map.Entry::getKey, Map.Entry::getValue, (a,b)->a, LinkedHashMap::new));
```
Haskell：
```haskell
topUsers :: [Order] -> [(String, Double)]
topUsers orders = 
    sortOn (Down . snd) 
    . filter ((>100) . snd) 
    . map (\g -> (head g, sum (map amount g))) 
    . groupBy (\a b -> user a == user b) 
    . sortOn user 
    $ orders
```

**例题2**：解释 Monad 的三个定律（左单位元律、右单位元律、结合律）并用 `Optional` Monad 验证。

**解答**：
左单位元：`pure a >>= f` = `f a`。即 `Optional.of(a).flatMap(f)` == `f.apply(a)`。
右单位元：`m >>= pure` = `m`。即 `opt.flatMap(Optional::of)` == `opt`。
结合律：`(m >>= f) >>= g` = `m >>= (\x -> f x >>= g)`。即 `opt.flatMap(f).flatMap(g)` == `opt.flatMap(x -> f.apply(x).flatMap(g))`。
验证：若 `opt` 为空，左脚 `Optional.empty().flatMap(f).flatMap(g)` = empty；右脚也一样 empty。若 `opt` 为 `Optional.of(v)`，左脚 = `f(v).flatMap(g)`；右脚 = `f(v).flatMap(g)`。等值成立。这三个定律保证了 Monad 链式操作的"括号位置不改变语义"。
