## 核心定义

程序的运行时内存空间划分为若干区域——栈（Stack）、堆（Heap）、全局区、常量区和代码区。理解每个区域的特征是编写正确和高效的代码的基础。

### 栈（Stack）

栈是一块由操作系统/运行时管理的 LIFO 内存区域。每次函数调用创建一个新的栈帧（Stack Frame / Activation Record），包含：
- 函数的局部变量
- 返回地址（函数返回后继续执行的指令地址）
- 调用方的帧指针（Frame Pointer / Base Pointer）
- 被保存的寄存器
- 参数（取决于调用约定通过寄存器或栈传递）

当函数返回时其栈帧被立即弹出——栈指针（Stack Pointer）向增长方向移动帧大小，执行 O(1) 释放（数据残留但标记为可重用）。

**栈的特点**：
- 分配/释放极快：仅移动栈指针（sp）几字节，极少 CPU 周期
- 大小有限：Windows 默认线程栈 ~1 MB，Linux ~8 MB。超出导致栈溢出（Stack Overflow）——进程崩溃
- 局部性极佳：连续内存 + 频繁访问使 CPU 缓存命中率高
- 绑定作用域：栈上变量的生命期严格等同于其所在作用域——不能灵活控制生命期
- 数据大小需编译时已知（除 `alloca`/变长数组外）

### 堆（Heap）

堆是动态分配的、由程序员或运行时显式管理的大块内存区域。堆变量通过 `malloc/new` 创建，手动 `free/delete` 或 GC 自动回收。

**堆的特点**：
- 生命期灵活：变量在其生命期超出函数作用域时仍需存活——堆分配是唯一方法
- 空间大：受虚拟内存和物理 RAM 限制，远大于栈
- 分配/释放成本高：分配器需搜索空闲块、更新内部簿记、可能的系统调用（`sbrk/mmap`）
- 碎片化（Fragmentation）：频繁分配和释放不同大小的块可能导致堆中存在大量无用小空闲块（无足够的连续空间满足分配请求）——外部碎片
- 线程安全成本：多线程堆分配需同步（现代分配器如 jemalloc、tcmalloc 分配每个线程独立的分配缓存减轻锁竞争）

**栈 vs 堆的直观对比**：
| 特性 | 栈 | 堆 |
|------|----|----|
| 分配速度 | ~1 CPU 指令 | ~数百 CPU 周期 |
| 大小 | 固定（1-8 MB） | 动态（GB） |
| 释放 | 自动（函数返回） | 手动 / GC |
| 碎片 | 无 | 可能严重 |
| 局部性 | 非常好 | 不可预测 |
| 线程安全 | 天然（各线程独立栈） | 需同步 |

### 其他内存区域

- **数据段**（Data Segment）：已初始化的全局变量和静态变量
- **BSS 段**（Block Started by Symbol）：未初始化和初始化为零的全局/静态变量
- **常量区**（Read-Only Data / Rodata）：字符串字面量、编译时常量——通常只读，修改导致段错误
- **代码段**（Text Segment）：可执行指令，通常只读

**内存布局图示**（进程虚拟地址空间，x86-64 Linux 简化图）：
```
0xFFFFFFFF: 内核空间
(栈向下生长: rsp ←)
  ... 栈 ...
  (间隙)
  ... (共享库 mmap 区) ...
  ... 堆 (向上生长: brk →) ...
  ... BSS ...
  ... 数据段 ...
  ... 常量区 ...
  ... 代码段 ...
0x00000000: 未映射（NULL 保护）
```

## 关键结论

- 尽量使用栈——快、自动、无泄漏风险。仅在需要超出作用域的生命期或大内存时用堆
- 栈溢出比堆耗尽更难恢复——栈溢出通常是立即崩溃（信号被杀），递归过深或极大小局部变量
- 堆碎片导致性能退化——频繁小分配是主要凶手（可预分配或对象池优化）
- 内存布局知识对于安全漏洞理解很重要——缓冲区溢出可篡改栈中返回地址（执行攻击代码）

## 易错点

1. 返回栈地址的指针——几乎每个 C/C++ 初学者犯过：函数内创建局部变量，返回其地址，导致悬空指针
2. 递归过深——无基况或基况太远导致栈溢出（Python 有递归深度限制 1000）
3. 滥用 `alloca`/变长数组分配不可控——潜在大分配因栈过载导致不可恢复崩溃

## 例题

**例题1**：以下变量分别在哪个内存区域？

```c
int global = 42;           // (a)
static int counter = 0;    // (b)
void func(int param) {     // (c)
    int local = 5;         // (d)
    int* heap_ptr = malloc(sizeof(int)); // (e)
    char* str = "hello";   // (f)
}
```

**解答**：(a) 在数据段；(b) 在数据段或 BSS（本例为数据段因有初始值）；(c) 在栈中（参数放在栈帧中）；(d) 在栈中（local 变量）；(e) `heap_ptr` 在栈中，其指向的内存来自堆；(f) 指针 `str` 在栈中，字符串 `"hello"` 在常量区（只读）。

**例题2**：Java 中所有对象都在堆上分配是否正确？“栈上分配"优化在 JVM 中可能吗？

**解答**：经典 Java 中所有对象在堆上分配是正确的。但现代 JVM 通过**逃逸分析**（Escape Analysis）可在检测到对象不会逃逸当前线程和当前调用栈范围时将对象"栈上分配"或"标量替换"（将对象的字段分解为局部变量直接在栈上分配）——避免堆分配和后续 GC 压力。逃逸分析是 JVM C2 编译器的一项优化，对用户透明。
