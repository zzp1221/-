## 核心定义

数据结构（Data Structure）是计算机中组织和存储数据的方式，使得数据可以被高效访问和修改。数据结构的选择直接影响算法的时间复杂度、空间复杂度和代码的可维护性。

**基础线性结构**：

1. **数组**（Array）：连续内存中存储同一类型元素的定长序列。特点：O(1) 随机访问、CACHE 友好（空间局部性）。缺点：插入/删除中间元素需移动后续元素 O(n)；容量固定需预分配。动态数组（ArrayList, Vector, Vec）通过倍增策略（满时容量×2）分摊插入为 O(1) 均摊。

2. **链表**（Linked List）：节点包含数据和指向下一个/前一个的指针。单向链表（Singly Linked）、双向链表（Doubly Linked）、循环链表。特点：O(1) 头部插入/删除、不需要连续内存。缺点：O(n) 随机访问、额外指针内存开销、缓存不友好。

3. **栈**（Stack）：LIFO 结构。操作：push（入栈）、pop（出栈）、peek/top（查看栈顶）。O(1) 所有操作。应用：函数调用栈、回溯、撤销操作、括号匹配、表达式求值。

4. **队列**（Queue）：FIFO 结构。操作：enqueue（入队）、dequeue（出队）。O(1) 操作。变体：双端队列 Deque（两端都可出入）、优先队列 Priority Queue（按优先级出队，非 FIFO）。

**高级结构**：

5. **哈希表**（Hash Table / HashMap）：键-值对存储，通过哈希函数将键映射到桶索引。期望 O(1) 增删查。冲突解决策略：(a) 拉链法（Separate Chaining）：每个桶是一个链表/平衡树；(b) 开放寻址法（Open Addressing, Java IdentityHashMap、Python dict）：冲突时换另一个桶（线性探查/二次探查/双重哈希）。负载因子（Load Factor, 元素数/桶数）增长时触发 rehash（扩容 + 重建）。哈希函数的要求：确定性、均匀分布、快速计算。

6. **树**（Tree）：层次化的节点-边结构。
   - **二叉搜索树**（BST）：左子树 ≤ 根 ≤ 右子树。期望 O(log n) 操作但退化最坏 O(n)（如按顺序插入）。
   - **平衡二叉搜索树**（AVL / 红黑树）：保证高度 O(log n)——AVL 平衡因子 ≤ 1；红黑树宽松但旋转更少（C++ std::map, Java TreeMap）。
   - **B树 / B+树**：针对磁盘 I/O 优化的多路搜索树——数据库索引和文件系统的核心。每个节点包含多个键（数百个），深度低（通常 3-4 层），磁盘 I/O 次数最小化。
   - **前缀树**（Trie）：用于字符串前缀匹配——自动补全、拼写检查、IP 路由。

7. **堆**（Heap）：一种特殊的完全二叉树，满足堆性质——父节点 ≥（最大堆）或 ≤（最小堆）子节点。O(log n) 插入/删除，O(1) 获取极值。应用：优先队列、堆排序、Top-K 问题、Dijkstra 最短路径。变种：二叉堆（数组实现）、斐波那契堆（O(1) 均摊插入）。

8. **图**（Graph）：顶点和边的集合。表示方法：(a) 邻接矩阵（稠密图 O(V²) 空间，O(1) 边检查）；(b) 邻接链表（稀疏图 O(V+E) 空间）。常见的图算法：BFS（最短无权重路径）、DFS（拓扑排序、连通分量检测）、Dijkstra（单源最短路）、Bellman-Ford（负权边最短路）、Kruskal/Prim（最小生成树）。

**数据结构的选择准则**：
- 需要频繁随机访问 → 数组/Vector
- 需要频繁头部/中间插入删除 → 链表
- 需要按键快速查找 → 哈希表（无序）、平衡树（有序）
- 需要最小/最大元素优先处理 → 堆/优先队列
- 需要前缀搜索/自动补全 → Trie
- 表示网络/连接关系 → 图

## 关键结论

- "数据结构 + 算法 = 程序"（Niklaus Wirth）——数据结构决定了存储方式，算法决定了处理方式
- 哈希表是 O(1) 期望但不是 O(1) 保证——最坏情况 O(n) 出现在大量哈希冲突（DoS 攻击利用：若预知哈希函数则发送冲突键淹没服务）
- 树的变种是"为特定存储介质定制的"——B 树为磁盘优化、LSM 树为写入优化、内存中红黑树为平衡查写优化
- 使用标准库的数据结构而非自己实现——除非有证明表明标准库版本成为瓶颈（性能或内存）

## 易错点

1. 遗忘 Java 的 `HashMap` 非线程安全——多线程并发修改导致死循环（1.7）或数据丢失
2. 可变键在 HashMap 中的哈希码变化导致"丢失"元素——放入 HashMap 的键不要修改其 hashCode 依据的字段
3. 二分搜索仅适用于有序随机访问序列——链表 O(n) 二分搜索实际不如线性搜索
4. 动态规划 vs 贪心——使用堆（贪心/优先级）解决的问题不一定符合最优子结构

## 例题

**例题1**：实现 LRU（Least Recently Used）缓存，支持 O(1) get 和 put。数据容量满时淘汰最久未使用的数据。

**解答**：HashMap + 双向链表。
```python
class LRUCache:
    class Node:
        def __init__(self, key=0, val=0):
            self.key, self.val = key, val
            self.prev = self.next = None

    def __init__(self, capacity):
        self.cap, self.cache = capacity, {}
        self.head, self.tail = self.Node(), self.Node()
        self.head.next, self.tail.prev = self.tail, self.head

    def get(self, key):
        if key not in self.cache: return -1
        node = self.cache[key]
        self._move_to_head(node)
        return node.val

    def put(self, key, value):
        if key in self.cache:
            self.cache[key].val = value
            self._move_to_head(self.cache[key])
        else:
            node = self.Node(key, value)
            self.cache[key] = node
            self._add_head(node)
            if len(self.cache) > self.cap:
                removed = self._pop_tail()
                del self.cache[removed.key]
    # _add_head, _remove_node, _move_to_head, _pop_tail 等辅助方法省略
```
HashMap 提供 O(1) 查找，双向链表维护访问顺序（头结点最近访问，尾结点最久未访问）。更新节点只需调整指针，O(1)。

**例题2**：Top-K 问题——从 10 亿个数字中找到前 K 个最大数（K=100）。

**解答**：使用最小堆维护当前最大的 K 个。遍历数据——若元素 > 堆顶（当前第 K 大），替换堆顶并重新堆化。时间 O(N log K)，空间 O(K)。当 N >> K 时远比全排序 O(N log N) 高效（仅维护 K 个元素）。
