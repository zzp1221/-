## 核心定义

元编程（Metaprogramming）是编写程序来操作程序（包括自身或另一程序）的技术。即将程序本身作为数据对象生成、修改、检查。元编程发生在两个层次：(a) **编译时元编程**（Compile-time Metaprogramming）：在编译阶段执行代码生成代码，生成物参与后续编译——无运行时开销；(b) **运行时元编程**（Runtime Metaprogramming）：在程序运行时检查和修改代码结构（通过反射、动态执行、`eval`、猴子补丁）。

**编译时元编程**：

1. **C++ 模板元编程**（Template Metaprogramming, TMP）：利用 C++ 模板系统在图灵完备的编译时类型计算——模板实例化本身就是一种函数式纯编程语言。可做到编译时阶乘计算、类型列表操作、编译时触发器。C++11 `constexpr` 大大简化和替代了 TMP（允许在编译时运行普通函数）。C++20 `consteval`（立即函数）强制函数仅在编译时执行。

2. **宏**（Macro）：
   - **C/C++ 宏**（预处理器 `#define`）：文本替换级别的宏——在编译的预处理阶段展开。简单但容易出错（运算优先级、多重求值）且无作用域。
   - **Rust 声明宏**（Declarative Macro, `macro_rules!`）：基于模式匹配的编译时代码生成——`vec![1,2,3]` 展开成 `Vec::new(); v.push(1); v.push(2); ...`。不需要跨文件预处理器——宏在 AST 级别操作，保证语法合法。
   - **Rust 过程宏**（Procedural Macro）：函数获取 TokenStream 返回 TokenStream——可以自定义 derive（`#[derive(Debug)]` 自动生成 `Debug` trait 实现）、attribute-like macro（`#[tokio::main]`）、function-like macro。

3. **Scala/Kotlin 的编译时元编程**：Scala 3 的 Macro、Kotlin 的 KSP (Kotlin Symbol Processing) + compiler plugin。

4. **Go Generate**：并非真正的编译时元编程——`go generate` 是一种构建时代码生成指令（按照源代码中的特殊注释运行代码生成工具）。

**运行时元编程**：

1. **反射**（Reflection）：程序在运行时检查、内省和修改自身结构和行为。Java 的 `java.lang.reflect` 包：`Class.forName`, `getMethods()`, `getFields()`, `setAccessible(true)` 破解私有字段；C# 的 `System.Reflection`；Go 的 `reflect` 包；Python 的 `type()`, `dir()`, `getattr()`, `setattr()`。反射的价值：框架和库（ORM、序列化、DI 容器、测试框架）可以通过反射发现和操作未知类型。

2. **动态执行**（Eval）：运行时将字符串解析为代码并执行——JavaScript `eval()`、Python `eval()`/`exec()`、Groovy `Eval`。`eval` 是强大但危险的——代码注入（Code Injection）是主要安全风险。

3. **猴子补丁**（Monkey Patching）：运行时修改或扩展类/模块——Python 中可以直接 `MyClass.new_method = lambda self: ...`，Ruby 开放类和模块可运行时修改，JavaScript prototype 修改。

4. **装饰器 / 注解**（Decorator / Annotation）：Python 装饰器是一种运行时元编程——包装函数、类以改变其行为，无修改原有代码：
   ```python
   @log_calls
   def compute(x, y):
       return x + y
   # compute = log_calls(compute)
   ```
   Java 注解（`@Override`, `@Autowired`）是编译时（或运行时处理）的元数据标记，配合反射或注解处理器工作。

5. **代理对象和动态代理**（Dynamic Proxy）：Java `java.lang.reflect.Proxy` 在运行时动态生成实现接口的代理类，截获所有方法调用。C# `System.Reflection.Emit` 在运行时生成 IL。

## 关键结论

- 元编程是"双刃剑"——可以消除样板代码、使 API 优雅、复用横切关注点；也可以生成不可调试的代码、增加编译器复杂性、降低可读性
- 编译时元编程优于运行时元编程（当有选择时）——编译时错误捕获、零运行时性能损耗、不依赖反射API
- 反射降低了类型安全——`getAttr(obj, random_string)` 的错误在运行时而非编译时暴露
- 宏不是函数——Rust 声明宏展开在类型检查之前（与泛型相反），错误信息的可读性不如函数版本
- "代码生成"的元编程是最简单且最可靠的形式——生成常规的代码而非使用魔法运行时机制

## 易错点

1. Python 默认参数在定义时仅求值一次——`def f(lst=[])` 是常见的元编程坑（使用 None + 内部检查替代）
2. C 宏的多重求值副作用：`#define SQUARE(x) (x*x)` 在 `SQUARE(a++)` 中展开为 `(a++)*(a++)`——a 递增两次，未定义行为
3. `eval` 执行不可信的输入可直接导致远程代码执行攻击（RCE）
4. Java 反射 `setAccessible(true)` 破坏封装——在模块化系统（Java 9+）中受限于模块导出/开放

## 例题

**例题1**：用 Python 元类（metaclass）实现：自动将所有属性的 setter 记录日志。

**解答**：
```python
class LoggedMeta(type):
    def __new__(cls, name, bases, dct):
        for attr_name, attr_value in dct.items():
            if attr_name.startswith('set_') and callable(attr_value):
                dct[attr_name] = cls._log_wrapper(attr_value)
        return super().__new__(cls, name, bases, dct)
    
    @staticmethod
    def _log_wrapper(func):
        def wrapper(self, *args, **kwargs):
            print(f"[LOG] Calling {func.__name__} with args={args}")
            return func(self, *args, **kwargs)
        return wrapper

class User(metaclass=LoggedMeta):
    def set_name(self, name): self.name = name
    def set_age(self, age): self.age = age
# User().set_name("Alice") → 自动打印日志
```

**例题2**：Java 注解处理器（APT）和反射处理注解的区别和各自适用场景。

**解答**：
APT（编译时）：
- 在 Java 编译期间运行（javac 调用注解处理器）
- 可以生成新的源文件、类文件（如 Lombok 生成 getter/setter, MapStruct 生成 mapper 实现）
- 性能：零运行时开销（生成代码等同于手写代码）
- 局限：只能访问编译时可用的符号（类名、方法签名、注解值），不能访问运行时数据
- 适用：样板代码消除、编译时代码校验

反射（运行时）：
- 在 JVM 运行时读取注解和元数据
- 访问运行时状态（实例值、当前上下文）
- 性能：反射访问较慢（Method.invoke 开销）
- 局限：类型错误在运行时暴露，缺乏编译时安全保障
- 适用：DI容器 (Spring @Autowired)、测试框架 (JUnit @Test)、序列化 (Jackson)
