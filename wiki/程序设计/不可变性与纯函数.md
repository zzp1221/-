## 核心定义

不可变性（Immutability）和纯函数（Pure Function）是函数式编程的两个基石，共同产生了"可预测的、可测试的、可并行化的代码"。不可变性确保数据不意外改变，纯函数确保函数行为没有隐藏的影响。

### 不可变性

不可变对象在被创建后不能再被修改。任何看似修改的操作都实际创建了一个新的对象（旧对象保持不变）。不可变性是数学值（2 + 3 = 5 不修改 2 或 3 创建了 5）在编程中的映射。

**不可变性的优势**：
1. **线程安全**——不可变对象不需要同步（无写操作，天然线程安全）
2. **可预测性**——已知值不会在调用函数之间"神奇地"变化
3. **缓存友好**——不可变对象可安全地缓存和共享（哈希值稳定）
4. **时间旅行调试**——每次"修改"都保留了历史状态（Easy undo / Redux DevTools）
5. **并行化**——共享数据只读不写消除数据竞争

**不可变性的代价**：
- 内存开销——每次"修改"都需要复制（持久化数据结构缓解）
- 性能开销——大量复制而非原地修改，对实时系统可能有压力
- 学习曲线——命令行思考从"改变变量"切换到"转换数据"

**持久化数据结构**（Persistent Data Structure）：实现不可变性而不全量复制的高效数据结构——通过**结构共享**（Structural Sharing）使新版本与旧版本共享绝大部分内部节点，仅在修改路径上创建新节点。典型实现：
- Clojure 的 vector / hash-map（基于 Tries / Bitmap-indexed nodes）
- Immutable.js（JS）的 List / Map
- Immer（JS）通过 Proxy 创建草稿（draft），突变 draft 后 immutable 产生新版本

**各语言中不支持变量的方法**：
- Haskell：默认一切不可变，`let` 不是赋值——`let x = 5 in x + 1` 中是绑定而非赋值（x 永为 5）
- Rust：变量默认不可变（`let x = 5;`），需 `let mut x = 5;` 显式声明可变
- Java：`final` 关键字保证引用不可变（不能重赋值），但引用指向的对象若可变仍可被修改
- Python：元组不可变，`frozenset` 不可变集合，dataclass `frozen=True`。对象属性默认可变——可用 `NamedTuple` / `@dataclass(frozen=True)` 创建不可变记录
- JavaScript：`const` 变量不可重绑定，但对对象属性无约束。`Object.freeze()` 浅不可变（深不可变需库支持）

### 纯函数

纯函数满足两个条件：
1. **确定性的**（Deterministic）：给定相同输入，始终返回相同输出
2. **无副作用**（No Side Effects）：函数不修改外部状态——不改变参数、不写入文件、不发送网络、不修改全局变量、不打印到控制台

```javascript
// 纯函数
const add = (a, b) => a + b;

// 非纯：依赖外部状态
let taxRate = 0.1;
const calculateTax = (amount) => amount * taxRate;  // 结果依赖 taxRate（外部状态）

// 非纯：副作用
const addToCart = (cart, item) => {
    cart.push(item);  // 修改了参数（副作用）
    return cart;
};
```

**引用透明**（Referential Transparency）：一个表达式 e 是引用透明的，如果可以在程序中用 e 的值替换 e 的出现而不改变程序含义。纯函数调用天然是引用透明的——这意味着编译器可以自由地重排序、缓存、延迟计算纯函数调用。

**副作用的分类**（Haskell 和函数式语言的观点）：
- I/O 操作（打印、文件读写、网络）
- 异常/错误抛出（改变控制流）
- 修改参数（mutating arguments）
- 修改全局/静态变量

**隔离副作用的实践**：完全消除副作用是不可能的（纯函数式程序用什么与外部世界交互），重点是"隔离而非消除"——将副作用推到应用的边界（最外层、入口点），保持核心业务逻辑纯。

## 关键结论

- 纯函数和不可变性共同创造"局部推理"（Local Reasoning）——理解一段代码时不需要追踪外部世界的状态变化
- 不可变性不是全有或全无——关键业务数据不可变（账户余额、订单状态），性能热点结构可变（高位率的数值计算）
- 纯函数是测试的理想形态——不需要 mock 外部环境，输入→输出直接断言
- 引用透明使得编译器优化和惰性求值成为可能（Haskell 的极致依赖这一点）

## 易错点

1. `const` 在 JavaScript 中引用不可变但内容可变——`const arr = []; arr.push(1)` 合法
2. Python 默认参数 mutable 共享——`def f(lst=[])` 在多次调用间 mutates 同一列表
3. React 中的浅比较——`prevProps !== nextProps` 可能误判（对象内容变了但引用相同或相反）
4. 所有函数都是纯函数 → 无法实现任何有意义的程序（无 I/O）——正确的目标：纯核心 + 不纯外壳

## 例题

**例题1**：将以下非纯函数改为纯函数：

```python
balance = 100
def withdraw(amount):
    global balance
    balance -= amount
    if balance < 0: balance += amount; return False
    return True
```

**解答**：将状态作为参数传入并返回新状态：
```python
def withdraw(balance, amount):
    if balance >= amount:
        return balance - amount, True
    return balance, False
# 使用：balance, success = withdraw(balance, 50)
```

**例题2**：解释为什么 React 强调不可变性对于性能优化（PureComponent/memo）是必要的。

**解答**：React 的 `PureComponent` 和 `React.memo` 通过浅比较（Shallow Comparison）判断是否需要重新渲染——如果 props 和 state 的引用与前一次相同则跳过渲染。若直接修改（mutate）已存在的对象，浅比较判断 `prevObj === nextObj` 为 true（同一引用）——React 错误地认为没有改变并跳过必要渲染。不可变更新（每次创建新对象）保证引用变化，浅比较有效触发正确重渲染。
