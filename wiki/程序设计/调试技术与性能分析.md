## 核心定义

调试（Debugging）是通过观察程序运行时的内部状态来定位和修复缺陷（bug）的系统性过程。性能分析（Profiling）是通过测量程序的资源消耗（CPU、内存、I/O 等）来定位性能瓶颈的过程。两者都是软件工程日常实践中不可或缺的技能。

### 调试技术

**调试的核心方法**：
1. **科学方法**（Scientific Debugging）：观察症状→构建设想→设计实验测试设想→分析结果→重复直到根因定位
2. **二分搜索**（Binary Search Debugging）：通过注释/禁用/隔离代码段快速缩小可疑范围
3. **Rubber Duck Debugging**：向另一个人（或甚至是一个橡皮鸭）解释代码逻辑——在解释中显现自身推理的缺陷
4. **Delta Debugging**（Zeller）：自动化地用二分法在多个输入变更中定位导致 bug 的最小变更集

**调试工具**：
- **交互式调试器**（Interactive Debugger）：GDB（C/C++/Rust）、LLDB（Swift/C/C++）、PDB（Python 内置）、Chrome DevTools（JavaScript）、Java Debugger（在 IDE 中——IntelliJ/Eclipse 的调试器）+ 远程调试
- **断点**（Breakpoint）：条件断点（仅在满足条件时暂停）、日志断点（打印而不暂停执行）、watchpoint（监视变量变化）
- **反向调试**（Reverse Debugging）：执行历史可倒回——GDB 的 `record`/`reverse-continue`，rr（Mozilla 的时间旅行调试器）。将"发生了什么"重复任意多次——无再现失败问题
- **strace / dtrace / SystemTap**：追踪程序对 OS 的系统调用——文件 I/O 失败、权限问题、网络连接问题快速定位

**日志和观测性**（Observability）：
- **结构化日志**（Structured Logging）：日志为机器可解析的键值对 JSON 格式——而非自由文本。方便筛选、聚合、告警
- **日志级别**（Log Level）：TRACE < DEBUG < INFO < WARN < ERROR < FATAL。生产环境常规为 INFO 级别
- **分布式追踪**（Distributed Tracing）：Jaeger/Zipkin 将跨多个微服务的请求全链路追踪——依据 OpenTelemetry 标准
- **OpenTelemetry**（OTEL）：合并了 OpenTracing 和 OpenCensus，一站式可观测性框架——Traces + Metrics + Logs

**系统性调试的常见套路**：
1. 确认 bug 能稳定复现 → 隔离最小复现案例
2. 用二分搜索区分"问题在数据还是代码"（换一份正常数据测试）
3. 版本回归二分搜索：`git bisect` 自动二分搜索定位引入 bug 的 commit
4. "变了什么"——最近改动是引入 bug 的最可能来源

### 性能分析（Profiling）

**性能分析的黄金法则**：先测量，再优化。不基于测量数据的优化是猜测。

**CPU Profiling**：
- **采样（Sampling）**：按固定频率暂停程序并检查调用栈（低开销，适合生产环境）。工具：Linux `perf`、Java `jstack`/`async-profiler`、`pyspy`（Python 采样 profiler）、Go `pprof`。
- **插桩（Instrumentation）**：在函数入口/出口记录时间戳（精确但开销高）。工具：`cProfile`（Python）、`go test -cpuprofile`。
- **火焰图**（Flame Graph, Brendan Gregg）：可视化 CPU 采样结果——x 轴宽度与样本数成正比（函数占用 CPU 比例），y 轴为调用栈深度。火焰图一眼识别热点函数。

**内存分析**（Memory Profiling）：
- 堆分析（Heap Profile）：找出内存分配最多的函数和对象类型
- 内存泄漏检测：随时间收集堆快照（Heap Snapshots）对比——找出意外增长的对象
- 工具：Valgrind (C/C++)、LeakSanitizer、`heaptrack`、Java `jmap`/`jhat`、Go pprof heap、Python `memray`/`tracemalloc`

**锁争用分析**（Lock Contention Profiling）：分析哪些锁被争用导致线程阻塞——`jstack` 检测死锁和阻塞、Go pprof mutex profile。

**I/O Profiling**：识别磁盘 IO 瓶颈——`iotop`、`iostat`、`strace -T`（跟踪每个系统调用的耗时）。网络 IO —— WireShark/tcpdump 包分析。

**性能分析和优化的最佳实践**：
1. 确立性能基线（Baseline）——优化前后对比必需
2. 使用代表性负载和真实环境（开发机优化无意义的）
3. 优化最热路径（Amdahl 定律——总的加速受限于非优化部分的比例）
4. 先优化算法降低复杂度级别，后微操作（汇编/内联）

## 关键结论

- 调试是"结构化的猜测-检验"过程——越系统化越快找到问题，越随性越浪费时间
- `git bisect` 是最被低估的调试利器——自动二分搜索数百个 commit 中找出引入 bug 的那一个
- 火焰图使 CPU 性能分析从"查阅数字表格"变为"一眼看懂"——优先优化最宽的条块
- 生产环境性能问题很难复现——采样式 profiler（低开销）加观测性平台（Grafana + Prometheus）才是出路

## 易错点

1. Print 调试 "够了"——但条件断点、表达式监测、后进调试（reverse debug）是更高效的专业工具
2. profiler 测量对象过窄——一个 CPU profile 只展示 CPU 消耗，I/O 等待不可见——需要用对应的工具分析对应瓶颈类型
3. 过早优化——Knuth 的名言"过早优化是万恶之源"不是反对优化，是反对**没有测量的猜测式优化**

## 例题

**例题1**：如何使用 Linux `perf` 工具生成火焰图分析程序的 CPU 热点？

**解答**：
```bash
# 1. 记录采样数据（99 Hz, 使用调用栈）
perf record -F 99 -g -- ./my_program
# 2. 生成 perf.data
perf script > out.perf
# 3. 折叠调用栈（FlameGraph 工具）
stackcollapse-perf.pl out.perf > out.folded
# 4. 生成火焰图（SVG）
flamegraph.pl out.folded > flamegraph.svg
```
火焰图中每个方块代表一个函数，宽度表示采样中的CPU占比。顶部方块是叶节点（最深的函数调用），通常 CPU 消耗集中在顶部。

**例题2**：内存泄漏的排查方法——Java 应用 heap 使用量持续增长最终 OOM。

**解答**：
步骤 1：设置 `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/dump.hprof` 在 OOM 时自动生成堆快照。
步骤 2：导入 hprof 到 Eclipse MAT 或 IntelliJ Profiler → 分析 "Dominator Tree"（支配树）——找出累计占比最大的对象及其 GC 引用链。
步骤 3：查看 "Leak Suspects" 自动报告——MAT 识别"有相似对象大量聚集"的模式。
步骤 4：连接源代码——找到持有泄漏对象的根引用（static field 或长生命期的集合），修复未清理的引用或未关闭的资源。
步骤 5：预防——用 `WeakHashMap`（Java）/`WeakRef` 缓存、设置最大缓存大小限制。
