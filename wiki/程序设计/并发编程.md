## 核心定义

并发（Concurrency）是程序同时管理多个任务的执行——这些任务在时间上重叠，但不一定在同一时刻物理并行（单核通过时间分片实现）。并行（Parallelism）是多个任务真正同时执行在不同的物理核心/处理器上。并发关注的是"同时处理"；并行关注的是"同时执行"。

并发编程面临的核心挑战来自**共享可变状态**（Shared Mutable State）——多个线程同时读写共享变量而不加以控制将导致竞争条件。

**线程**（Thread）是操作系统调度的最小执行单位。同一进程内的线程共享地址空间（可以看到相同的全局变量和堆），但每个线程拥有自己的栈和寄存器上下文。线程的生命周期：创建（New）→ 就绪（Runnable）→ 运行中（Running）→ 阻塞（Blocked/Waiting）→ 终止（Terminated）。

**线程安全**（Thread Safety）：在多线程环境下，一个代码段（类、方法、数据结构）不需要额外的同步即可被多个线程同时访问而不产生错误行为。实现线程安全的策略：

1. **互斥**（Mutual Exclusion / Mutex）：使用锁保证在同一时刻只有一个线程可以访问临界区。`synchronized` (Java)、`lock`/`mutex` (C++/Rust/Go)、`ReentrantLock` (Java/C#)。
2. **不可变性**（Immutability）：共享对象设计为不可变——一旦创建不能被修改，天然线程安全。Java 的 `String`、Scala/Clojure 默认不可变数据结构。
3. **线程隔离**（Confinement）：不将数据共享给多个线程。每个线程拥有自己的可变对象，不逃逸到其他线程。ThreadLocal（线程局部变量）。
4. **原子操作**（Atomic Operations）：某些简单操作可通过硬件支持的原子指令（CAS, Compare-And-Swap）实现无锁线程安全。`AtomicInteger` (Java)、`std::atomic` (C++)、`atomic` 包 (Go)。
5. **消息传递**（Message Passing）：线程/进程之间不直接共享数据，而是通过通道（Channel）传递消息——"不要通过共享内存来通信，要通过通信来共享内存"（Go 的 CSP 哲学）。Erlang/Elixir 的 Actor 模型、Rust 的 `mpsc` channel。

**常见并发问题**：

1. **数据竞争**（Data Race）：两个或更多线程并发访问同一内存位置，且其中至少有一个是写操作，且没有同步控制。数据竞争是未定义行为（C/C++）或可导致细微的正确性问题（Java）。

2. **死锁**（Deadlock）：两个或多个线程互相等待对方持有的资源，形成循环等待——所有线程无限期阻塞。死锁的四个必要条件（Coffman Conditions）：(a) 互斥（至少一个资源不可共享）；(b) 持有并等待（持有锁时等待其他锁）；(c) 不可剥夺（锁不能被强行释放）；(d) 循环等待（形成环状等待关系）。死锁预防：破坏任一条件——如按全局固定顺序获取锁（防止循环等待）、使用超时（tryLock with timeout）、死锁检测算法。

3. **活锁**（Livelock）：线程不断改变状态以响应其他线程但没有实际进展——像两个人面对面让路，同时左移右移永远撞在一起。

4. **饥饿**（Starvation）：某线程长时间得不到执行机会（CPU 时间片、锁获取），虽然系统整体在进展。

5. **ABA 问题**：CAS 操作中值先变为 A→B→A，线程视为值未改变——但在无锁数据结构中可能有隐患（通常用带版本的 CAS 解决）。

**高级同步工具**：
- **信号量**（Semaphore）：计数锁——允许多个线程同时访问临界区，超出上限后阻塞
- **读写锁**（R/W Lock）：允许多个并发读者，但写者是互斥的——优化读多写少场景
- **条件变量**（Condition Variable）：线程等待某个条件为真——`wait` / `notify` / `signal`
- **倒计时闩**（CountDownLatch / Barrier）：多个线程在某个点集合，等待所有线程到达后再一起推进

**线程池**（Thread Pool）：预创建一组线程，任务提交给线程池执行——线程复用避免频繁创建/销毁线程的系统开销。Java ExecutorService, C# Task.Parallel, Rust rayon。

## 关键结论

- 并发 ≠ 并行 —— 单核可并发（时间片切换）但不可并行
- 最小化共享状态是并发编程的第一原则——无共享架构（Shared-Nothing）消除了整个类别的并发 bug
- 锁是降低并发度的——过大的锁粒度（锁整个方法而非关键部分）使并发退化为串行
- 不可变对象天然线程安全——代价是每次更新都要复制，需要在内存和安全性间权衡
- 并发 bug 的重现率低、难以调试——引入变化时序的随机因素增加了不可预测性

## 易错点

1. `i++` 不是原子操作——读-改-写三步组成，多线程并发执行导致非法结果
2. `volatile` 保证可见性但不保证原子性——`volatile int` 的 `++` 仍然不安全
3. `synchronized` 的锁对象是 `String` 字面量——JVM 字符串驻留导致完全不相关的代码共享同一把锁
4. 双检锁（Double-Checked Locking）在 Java 5 之前与未修饰 `volatile` 一起使用存在缺陷

## 例题

**例题1**：分析以下代码的死锁原因并修复。

```java
void transfer(Account from, Account to, int amount) {
    synchronized(from) {
        synchronized(to) {
            if (from.balance >= amount) {
                from.balance -= amount; to.balance += amount;
            }
        }
    }
}
```

**解答**：死锁场景：线程 A 执行 `transfer(a, b, 100)`，持有 a 锁等待 b 锁；线程 B 执行 `transfer(b, a, 100)`，持有 b 锁等待 a 锁——循环等待死锁。修复：按一致的全局顺序获取锁——对不同 Account 按 ID 排序或使用 `System.identityHashCode()` 排序：
```java
void transfer(Account from, Account to, int amount) {
    Account first = from.id < to.id ? from : to;
    Account second = from.id < to.id ? to : from;
    synchronized(first) {
        synchronized(second) {
            if (from.balance >= amount) { from.balance -= amount; to.balance += amount; }
        }
    }
}
```

**例题2**：对比 Java 的 `synchronized` 和 `ReentrantLock` 的功能差异。

**解答**：
功能差异：(a) `synchronized` 是语言内置关键字，自动释放锁（退出同步块或异常时）；`ReentrantLock` 是 API 级别（java.util.concurrent）——必须显式在 `finally` 块 `unlock()`；(b) `ReentrantLock` 支持公平锁（先到先得）、可中断的锁获取（`lockInterruptibly()`）、条件绑定（多个 Condition）；(c) `synchronized` 强制使用内部条件变量（单一 `wait/notify`），`ReentrantLock` 可通过 `newCondition()` 创建多个条件；(d) `ReentrantLock` 可查询锁状态（`isLocked`, `getQueueLength`）便于调试和监控。选择：一般场景优先用 `synchronized`（简单、安全），需高级功能时选 `ReentrantLock`。
