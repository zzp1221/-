## 核心定义

类型系统（Type System）是编程语言中将"类型"（Type）属性赋予计算中的各构造（变量、表达式、函数、模块）并规定这些类型如何交互的规则集合。类型系统的主要目标：(a) 类型安全（Type Safety）：防止非预期的类型混合操作（如将整数除以字符串）；(b) 抽象（Abstraction）：将底层内存表示为有意义的逻辑概念；(c) 文档（Documentation）：函数签名（int → string → list）本身就是部分契约文档；(d) 优化（Optimization）：编译器利用类型信息生成更高效的代码。

**类型系统分类**：

|      | 静态类型 (Compile-time)          | 动态类型 (Runtime)             |
|------|--------------------------------|------------------------------|
| 强类型 | Haskell, Rust, Java, TypeScript | Python, Ruby, Elixir          |
| 弱类型 | C (指针类型转换)                 | JavaScript, PHP, Perl          |

注意："强类型"/"弱类型"是非形式术语——学术上更倾向于"类型安全程度"这一维度。

**静态类型系统**（Static Typing）：变量或表达式的类型在编译时确定且不可变。优点：(a) 编译时捕获大多数类型错误；(b) 编译器可用于类型级优化（Rust 的单态化、Haskell/w GHC rewrite rules）；(c) IDE 支持（自动补全、重构）更强大。代价：(a) 更多的代码——类型注解增加代码量（现代语言通过类型推断缓解）；(b) 无法表达某些运行时的动态模式——"鸭子类型"需借助泛型或接口替代。

**类型推断**（Type Inference）：编译器自动推断变量和表达式的类型而无需显式注解。Hindley-Milner 类型系统（ML/Haskell）实现了最完全的类型推断。现代语言程度不同的类型推断：
- Rust：`let x = 42` → i32 可推断，函数参数则需显式类型
- TypeScript：根据初始值推断变量类型，复杂情况下需注解
- C++11+：`auto` 关键字做局部变量自动推断
- Java 10+：`var` 关键字做局部变量自动推断

**动态类型系统**（Dynamic Typing）：类型在运行时检查，变量可在其生命周期中引用不同类型的值。优点：(a) 灵活——写代码更快且代码量更少；(b) 表达力——元编程、猴子补丁、原型继承等动态模式无拗口类型体操。代价：(a) 类型错误仅在运行时暴露；(b) IDE 支持弱——无法可靠地自动补全或重构；(c) 性能——运行时类型检查和动态分派有开销。

**多态（Polymorphism）是类型系统的重要分支**：
1. **参数多态**（Parametric Polymorphism, "泛型"）：`List<T>` 可用于任何类型 T，Java 的 `ArrayList<E>`、C# 的 `List<T>`、Rust 的 `Vec<T>`、Go 1.18+ 的 `[]T`（通过类型参数实现）。
2. **特设多态**（Ad-hoc Polymorphism, "重载/接口/Typeclass"）：相同的操作名称针对不同类型有不同实现。C++/Java 的函数重载、Haskell 的 Typeclass、Rust 的 Trait、Go 的 Interface。
3. **子类型多态**（Subtype Polymorphism, "继承与接口"）：子类对象可被当作父类对象使用。Java/C++ 的 `class Dog extends Animal`。
4. **行多态**（Row Polymorphism）：记录类型上的多态，如 TypeScript 的 `{name: string} & {age: number}`。

**高级类型系统特性**：
- **代数数据类型**（ADT, Algebraic Data Types）：Sum type（枚举/Union）+ Product type（Record/Tuple）。Rust 的 `enum`（带数据）、Haskell 的 `data`、TypeScript 的 Discriminated Union。
- **泛型约束/限定**（Bounded Generics）：限制类型参数必须满足某种条件——`T extends Comparable<T>` (Java)、`where T: Ord` (Rust)。
- **变型**（Variance）：针对泛型的子类型规则——协变（Covariant, `List<Cat>` 是 `List<Animal>` 的子类型如果 Cat ≤ Animal）、逆变（Contravariant, 函数参数位置翻转）、不变（Invariant, 两种方向都不允许）。
- **类型擦除**（Type Erasure）：Java 的泛型编译后擦除（生成的字节码中不保留泛型信息）vs C# 运行时的具体化泛型（Reified Generics）。类型擦除影响反射和运行时类型判断。
- **空安全**（Null Safety）：类型系统对 null/nil 的处理。Option/Optional 类型（Rust `Option<T>`、Java `Optional<T>`、Haskell `Maybe`、TypeScript `strictNullChecks`）在类型级别区分"可能为空的值"和"绝不空的值"。

## 关键结论

- 类型系统本质上是一个定理证明系统——良好设计的类型系统可以证明程序中不存在某些类型的错误
- 动态类型 = 一种大型单一类型（One Big Union Type），运行时标签追踪实际类型
- 类型推断不是万能的——在函数接口处显式类型注解可以提高可读性和错误定位
- Null 引用被称为"十亿美元的错误"（Tony Hoare 1970s）——现代语言推荐使用 Optional 类型在类型层面处理缺失值
- 类型的表达能力越强，程序员越能利用类型系统"使错误状态不可能被表示"（Make illegal states unrepresentable）

## 易错点

1. 混淆静态/动态类型与强/弱类型——它们是正交的维度
2. 将"动态类型"和"没有类型"等同——Python/Ruby 的变量有类型（int, str, list class），只是不事先标注在变量上
3. 类型强制转换（Coercion）是引发安全漏洞的常见来源——如 C 的 `int` 到 `char` 截断、JavaScript 的 `==`（松散相等）自动转换
4. 泛型的变型（Variance）在 Java 中容易出错——`List<String>` 不是 `List<Object>` 的子类型（Java 数组是协变但泛型是不变）

## 例题

**例题1**：以下代码各自有哪些类型安全问题？

```java
// Java
Object[] arr = new String[1];  // (1)
arr[0] = 1;                     // (2)
```

```rust
// Rust
let x = 42;                    // (3)
x = "hello";                   // (4)
```

```python
# Python
def add(a, b):
    return a + b
print(add(1, "2"))             // (5)
```

**解答**：
(1) Java 数组是协变的——`String[]` 是 `Object[]` 的子类型，所以此赋值编译通过。(2) 运行时抛 `ArrayStoreException` 因为数组底层实际类型为 `String[]`，不能接受 `Integer`。这是 Java 数组协变的历史设计缺陷。(3) Rust 的类型推断将 x 绑定为 `i32`。(4) 编译错误——`x` 是 `i32` 类型（静态）, 对整数变量赋值字符串不被允许。体现了静态类型的安全性。(5) Python 是动态强类型——运行时 `int.__add__` 收到 `str` 操作数，抛出 `TypeError: unsupported operand type(s) for +: 'int' and 'str'`。体现了 Python 的强类型（不隐式将 int 转为 str）。

**例题2**：用泛型实现一个类型安全的栈（Stack），支持 push, pop, peek, isEmpty。用 Java 和 Rust 两种语言说明差异。

**解答**：
```java
// Java 泛型栈
public class Stack<T> {
    private List<T> items = new ArrayList<>();
    public void push(T item) { items.add(item); }
    public T pop() { 
        if (items.isEmpty()) throw new EmptyStackException();
        return items.remove(items.size() - 1); 
    }
    public T peek() { /* ... */ }
    public boolean isEmpty() { return items.isEmpty(); }
}
```
```rust
// Rust 泛型栈
pub struct Stack<T> {
    items: Vec<T>,
}
impl<T> Stack<T> {
    pub fn push(&mut self, item: T) { self.items.push(item); }
    pub fn pop(&mut self) -> Option<T> { self.items.pop() }
    pub fn peek(&self) -> Option<&T> { self.items.last() }
}
```
差异：Rust 用 `Option<T>` 表达了 pop 可能失败的情况，编译时强制调用方处理 None 分支，而 Java 抛 `EmptyStackException`（未检查异常）在运行时才会显现。Rust 的类型系统"使错误状态不可表示"——若 pop 可能为空但调用方忽略了这会让代码不编译通过。
