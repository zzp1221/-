## 核心定义

面向对象编程（OOP, Object-Oriented Programming）是一种以"对象"（Object）为中心组织代码的编程范式。对象是数据（属性/字段）和行为（方法）的封装体。OOP 通过模拟真实世界中的实体和交互来实现软件设计，核心理念是：将复杂性分解为相互协作的独立对象，每个对象维护自己的内部状态并暴露行为接口。

**四大基本特征**：封装、继承、多态、抽象。

**1. 封装**（Encapsulation）：将数据和操作数据的方法捆绑在一起，并控制外部对内部状态的访问。封装通过访问修饰符（private/protected/public/package-private）实现，隐藏实现细节，仅暴露公开接口。封装的目的不是"保密"，而是隔离变化——内部实现的改变不影响客户端代码。封装的关键原则是"Tell, Don't Ask"（要求对象执行操作而非查询其内部状态再做决策）。

**2. 继承**（Inheritance）：子类从父类派生，自动获得父类的属性和方法，并可添加新的或覆盖已有的。继承实现了"is-a"关系（Dog is an Animal）。继承的优势：(a) 代码复用——共性的逻辑放在父类一次实现；(b) 多态——通过父类引用操作子类对象。继承的问题：(a) 白盒复用——子类曝光父类的内部实现从而破坏封装；(b) 深继承链难以理解和维护；(c) 子类和父类的耦合过于紧密。现代设计倾向于用**组合/接口**替代深度继承——Composite over Inheritance rule。

**3. 多态**（Polymorphism）：相同的接口可作用于不同类型的对象而产生不同的行为。两个主要形式：(a) **子类型多态**（运行时多态/动态分派）：父类引用指向子类对象，方法调用由实际对象的类型决定（通过虚表/方法表）。Java/C++ 的虚函数（virtual method）、接口实现。(b) **重载多态**（编译时多态）：同一方法名不同参数列表，在编译时根据参数类型决定调用哪个版本。

**4. 抽象**（Abstraction）：通过抽象类（abstract class）或接口（interface）定义对象的外部行为契约而不规定实现。抽象使客户端依赖于"what"而非"how"——依赖注入和控制反转（IoC）的基础。接口是纯粹的抽象——不含任何状态，只声明方法签名。抽象类可以包含部分实现和状态，但不能实例化。

**OOP 设计原则**（SOLID）：
- S（Single Responsibility）：一个类只有一个变化的原因
- O（Open/Closed）：对扩展开放、对修改关闭——新功能通过增加新代码而非修改已有代码
- L（Liskov Substitution）：子类型必须可以完全替代父类型（子类不应削弱父类的行为契约）
- I（Interface Segregation）：不强迫类依赖它不需要的接口方法
- D（Dependency Inversion）：高层模块不应依赖低层模块——二者都依赖抽象

**类和对象的关系**：类是蓝图（模板），对象是实例（具体的）。类定义对象有哪些属性（fields）和能做什么（methods）。`new/instantiate` 一个类即根据蓝图在内存中创建一个对象。

**对象间通信**：通过消息传递（方法调用）实现。对象 A 调用对象 B 的公开方法——A 不需要知道 B 的内部状态。

## 关键结论

- 继承是"最被误解和滥用的 OOP 特性"——"优先使用组合而非继承"（Gang of Four, 1994）
- 封装不是简单的 "private + getter/setter"——不加过滤地给每个字段提供 get/set 等于没有封装（公开了内部状态）
- 多态的核心价值不是"节省几个 if-else"而是"新增行为时不需要修改已有代码"（Open-Closed 原则）
- 好的 OOP 代码倾向于"贫血模型"向"充血模型"转变——领域逻辑在领域对象中而非在 Service 方法中

## 易错点

1. 继承的类型和组合/接口混淆——`Square extends Rectangle` 违反 Liskov 原则（Rectangle 的 `setWidth` 不适用于 Square），因为继承的 "is-a" 不成立（正方形的宽度改变必须同时改变高度）
2. Overload 和 Override 混淆——Overload 是编译期同一方法名不同参数列表；Override 是运行时子类重新定义父类方法（相同签名和返回类型或协变返回类型）
3. `getter` 返回可变内部对象引用破坏封装——应返回不可变副本或不可变视图

## 例题

**例题1**：以下代码违反了哪个 SOLID 原则？请重构。

```java
class Rectangle {
    double width, height;
    double area() { return width * height; }
}
class Square extends Rectangle {
    void setSide(double side) { width = height = side; }
    @Override void setWidth(double w) { super.setWidth(w); super.setHeight(w); }
}
```

**解答**：违反 Liskov 替换原则。Rectangle 的设定：`setWidth` 只改宽度，`setHeight` 只改高度——但 Square 的 `setWidth` 也改了高度。调用 `rect.setWidth(w)` 的代码期望 `area()` 变为 `w * originalHeight`，但对于 Square 变为 `w * w`——子类的错误行为。重构：不用继承，改为独立的 Shape 接口：`interface Shape { double area(); }`，Square 和 Rectangle 各自独立实现——组合优于继承。

**例题2**：描述 Java 中 `abstract class` 和 `interface` 的区别（包括 Java 8+ 接口的新特性）。

**解答**：
相同：两者都是抽象类型，不能直接实例化，用于定义契约。
不同：(a) abstract class 可以有实例变量和构造函数，interface 只能有 static final 常量（Java 8+ interfaces 没有实例变量）；(b) abstract class 可以有非抽象方法（有方法体），Java 8+ interface 可以用 `default` 关键字提供默认实现；(c) 一个类只能继承一个 abstract class（单继承），但可实现多个 interface；(d) abstract class 的访问修饰符灵活，interface 方法默认 public；(e) abstract class 可以有 protected 字段（供子类访问），interface 所有字段都是 public static final。
选择指南：共享代码用 abstract class（复用状态+行为）；定义跨层级的协议用 interface。
