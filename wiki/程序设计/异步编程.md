## 核心定义

异步编程（Asynchronous Programming）是一种不阻塞主执行线程的编程模型——发起 I/O 操作（网络请求、文件读取、数据库查询）后立即返回，当操作完成时通过回调、Promise 或协程等方式通知和处理结果。异步编程旨在提高 I/O 密集型应用的吞吐量和响应性，而非加速 CPU 计算本身。

同步 vs 异步：同步模型下，调用函数后暂停等待结果，完成后再继续。异步模型下，调用后立即返回（获得一个未来结果的句柄 Future/Promise），线程被释放去处理其他工作。

**核心概念**：

**事件循环**（Event Loop）：异步运行时引擎的核心——单线程或少量线程通过事件循环不断检测和分派就绪的 I/O 事件。Node.js 的事件循环（libuv）、Python 的 asyncio 事件循环、JavaScript 浏览器的事件循环（Web APIs + Task Queue）。事件循环解决了"如何在一个线程中同时处理 10000 个客户连接"的问题——本质上是通过非阻塞 I/O + 轮询就绪事件。

**回调**（Callback）：最原始的异步模式——将回调函数（或 lambda/闭包）作为参数传递，当异步操作完成时被调用。"圣诞树问题"（Callback Hell / Pyramid of Doom）——多步异步操作形成嵌套的箭头形代码，难以理解和维护。

**Promise / Future**：代表一个异步操作的最终完成（或失败）及其结果的代理对象。Promise 有三个状态：Pending（进行中）、Resolved/Fulfilled（已完成）、Rejected（已失败）。Promise 通过 `.then(resolveCallback, rejectCallback)` 链式处理，避免嵌套回调。Promise 的组合：`Promise.all`（等待全部完成）、`Promise.race`/`any`（任一完成后继续）、链式组合。

**async/await**：在 Promise/Future 之上的语法糖——使异步代码阅读起来像同步代码，同时保留非阻塞语义。`async` 函数自动返回 Promise/Future，`await` 暂停函数执行等待 Promise 完成（实际是在 await 点让出线程给事件循环）。Python `async def` + `await` （3.5+）, JavaScript `async function` + `await` (ES2017), Rust `async fn` + `.await`, C# `async Task` + `await` (C# 5.0)。

async/await 不是多线程——它是协作式多任务：一个执行线程在多个异步任务间切换，切换只发生在显式的 await 点（而非抢占式调度）。

**协程**（Coroutine）：可暂停和恢复的函数——是 async/await 的底层构造。与线程不同，协程的上下文切换由程序员控制（或编译器在 await 点自动控制），无需内核介入——切换成本远低于线程（纳秒级 vs 微秒级）。Go 的 goroutine 是抢占式的协程（由 Go 运行时调度而非用户代码决定挂起点）；Python 的 asyncio、Kotlin 的 suspend function、Lua 的 coroutine 是完全协作式的。

**非阻塞 I/O 原理**：操作系统提供非阻塞 I/O 系统调用（如 Linux 的 epoll、BSD 的 kqueue、Windows 的 IOCP）。应用注册文件描述符和感兴趣的事件（可读/可写），OS 负责监听——当事件就绪时通知应用处理。这使得一个线程可以同时管理数以万计的连接（C10K 问题→C10M 问题）。

## 关键结论

- async/await 是协作式而非抢占式——await 点之外的代码不会被中断，推排序需要更谨慎（死锁条件从 await 点外消除）
- 异步 ≠ 并行——异步可以在单个线程内处理多个 IO 操作；并行是利用多核心同时执行 CPU 密集任务
- 异步函数应避免同步阻塞调用——在 async 函数中调用同步 `sleep(10)` 会阻塞事件循环，应用 `await asyncio.sleep(10)` 让出控制权
- 异步 I/O 不是万能的——CPU 密集型计算（大量计算、图像处理）仍需要多线程/多进程

## 易错点

1. 将多线程和异步编程视为互斥——两者可组合使用（如 Rust 的 Tokio 使用线程池执行器，Python 的 `asyncio.to_thread` 将阻塞操作移到线程池）
2. async 函数中忘记 await——返回的是未完成的 Promise/协程对象而非结果
3. 并发循环中的变量捕获（与多线程同一问题）——`for url in urls: tasks.append(fetch(url))` 但 fetch 是异步的还要 await
4. 混合同步和异步——Python中不能在普通 `def` 函数中用 `await`，不能在 `async def` 中直接调用阻塞函数

## 例题

**例题1**：用 Python asyncio 实现并发抓取 10 个 URL，超时 5 秒，记录成功和失败统计。

**解答**：
```python
import asyncio
import aiohttp

async def fetch(session, url):
    try:
        async with session.get(url, timeout=aiohttp.ClientTimeout(total=5)) as resp:
            return url, resp.status, await resp.text()
    except asyncio.TimeoutError:
        return url, None, "timeout"
    except Exception as e:
        return url, None, str(e)

async def main(urls):
    async with aiohttp.ClientSession() as session:
        tasks = [fetch(session, url) for url in urls]
        results = await asyncio.gather(*tasks, return_exceptions=True)
    success = [(url, status) for url, status, _ in results if status == 200]
    failed = [(url, err) for url, _, err in results if err is not None]
    return success, failed
```

**例题2**：解释 Node.js 的 Event Loop 六个阶段（timers, pending callbacks, idle/prepare, poll, check, close callbacks）的作用。

**解答**：
(1) timers：执行 `setTimeout`/`setInterval` 过期回调。
(2) pending callbacks：执行延迟到下一轮的 I/O 回调。
(3) idle/prepare：Node 内部使用。
(4) poll：阻塞等待 I/O 事件（epoll_wait）。进入 poll 阶段时如果有设定 timers 则在时间到时立即执行；否则若有 setImmediate 回调则先执行它们；否则 poll 阻塞直到新事件或 timer 过期。既无 timer 也无 immediate 且无活跃 handle → 结束轮询。
(5) check：执行 `setImmediate` 回调。
(6) close callbacks：执行关闭事件的回调（如 `socket.on('close')`）。

**例题3**：Rust 中 async/await 的零成本抽象是什么意思？

**解答**：Rust 的 async/await 编译为状态机——编译器在编译时将 async 函数转换为实现 `Future` trait 的自动生成类型（enum with variants for each yield point），状态机的每个状态对应 await 之间的间隔。状态机的大小不超过原函数中最大跨 await 的变量集合大小——无堆分配、无虚函数调用、编译器内联优化使运行时开销趋向零。这与 Java 的 Project Loom（运行时 JVM 支持）形成对比——Rust 选择了"编译时变换"的答案换取零运行时开销。
