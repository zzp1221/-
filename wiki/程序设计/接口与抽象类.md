## 核心定义

接口（Interface）和抽象类（Abstract Class）是面向对象语言中实现抽象的主要工具——它们定义了"做什么"（契约）而不规定"怎么做"（实现），将接口与实现分离。

### 抽象类（Abstract Class）

不能直接实例化的类——存在是为了被继承。抽象类可包含：
- 抽象方法（无方法体，子类必须实现）
- 具体方法（有方法体，子类可继承或重写）
- 构造器（在子类构造时被调用）
- 实例变量（状态）

**语言差异**：
- Java/C#：`abstract class`，可包含任何成员（抽象/具体方法、字段、构造器）。单继承一个抽象类。
- C++：纯虚函数（`virtual void f() = 0`）使类成为抽象——用"等于 0"语法声明。
- Python：`abc.ABC` + `@abstractmethod` 装饰器使方法为抽象。

### 接口（Interface）

纯粹的抽象契约——声明一系列方法签名但不提供实现（不同语言程度上允许默认实现）。约定任何实现此接口的类都承诺提供这些方法。

**接口的关键特征**：
- 无实例变量（只有静态常量 `public static final`）
- 无构造器（接口不可实例化）
- 多实现：一个类可实现多个接口（接口多重继承的安全形式）

**各语言的接口机制**：
- Java：`interface` 在 Java 8+ 可有 `default` 方法（默认实现）和 `static` 方法。接口代表"能做"（Can-Do）关系。
- C#：`interface`——类似 Java 但无 `final` 常量。C# 8+ 支持 default 方法。
- Go：`interface` 是核心抽象机制——结构类型（Structural Typing）：不需要显式声明 `implements`，只要有匹配的方法签名即满足接口。Go 的接口设计是"小而灵"——单方法接口很常见（`io.Reader`: `Read([]byte) (int, error)`）。
- Rust：`trait` 是接口 + 抽象类混合体——接口定义可包含默认实现方法。Trait 不是类型本身，需要用 `dyn Trait`（类型擦除）或泛型（单态化）。
- TypeScript：`interface` 定义对象形状——结构类型，可扩展（`extends`）、合并。TS 接口仅编译时存在。

### 接口 vs 抽象类

| 维度 | 接口 | 抽象类 |
|------|------|--------|
| 目的 | 定义可被采纳的契约 | 为子类提取公共实现 |
| 关系 | Can-Do (如 Serializable) | Is-A (如 Animal) |
| 状态 | 无实例变量（常量除外） | 可有实例变量 |
| 实现 | 多实现 | 单继承 |
| 默认行为 | 各语言不同（Java 8+ default） | 有（具体方法可在抽象类中） |
| 演进 | 添加新方法容易 breaking changes (default 方法缓解) | 可安全添加非抽象方法 |

**何时使用接口**：
- 需要不相关类共享相同的行为（如 Comparable, Iterable）
- 需要多态但继承不自然（多个可组合行为）
- 定义 API 的边界（模块/库的入口出口）

**何时使用抽象类**：
- 多个密切相关类共享大量代码（模板方法模式）
- 需要定义可被子类使用的状态字段
- 确定只有一个基础类型（框架的 BaseController）

### 高级接口概念

**Traits / Mixins**：除了定义接口外还提供可复用的代码——介于接口和多重继承之间。Ruby 的 `module`（mixin into class）、Scala 的 `trait`（可包含方法和状态）、Kotlin 的 `interface`（可包含默认实现和属性声明）。

**协议**（Protocol）：Python 3.8+ 的 `typing.Protocol`——定义结构子类型（类似 Go 的接口或 TypeScript 的结构类型）。类不需显式继承 Protocol 就自动满足（"静态鸭子类型"）。

**密封接口**（Sealed Interface）：Java 17+ 和 Kotlin——限制哪些类可实现此接口，保证模式匹配的穷尽性。`sealed interface Shape permits Circle, Rectangle, Triangle {}`。

## 关键结论

- 接口代表"你能做什么"（Capability），抽象类代表"你是什么的一种"（Identity）
- "面向接口编程而非面向实现编程"——客户端依赖于接口（不易变）而非具体类（易变）
- Go 的隐式接口满足是最优雅的 Interfacing 设计——完全去耦实现和接口的定义
- 接口的最重要价值不是代码复用（那是继承的事）——而是解耦，允许多个实现者交换使用

## 易错点

1. 抽象类和接口混用——一个 AbstractService 既是 interface 又是 abstract class 中冗余的抽象方法
2. 接口爆炸——"每个类都得有接口"是教条。当只有一个实现且可预见未来仍只有一个实现时，接口是过度设计
3. Java 接口 default 方法滥用——使接口成了"多重继承的后门"，可能引发默认方法冲突（相同方法签名多个接口提供不同的 default——需显式 override）
4. C++ 没有接口 keyword 但有等价物——纯抽象类（全部虚函数 pure virtual + virtual 析构函数）

## 例题

**例题1**：设计一个"支付系统"——支持微信、支付宝、银行卡支付。说明接口和抽象类的选择及原因。

**解答**：
接口 `PaymentGateway`：定义 `pay(amount, account) → PaymentResult` 和 `refund(transactionId) → RefundResult`。
三个具体实现 `WechatPayGateway`, `AlipayGateway`, `BankCardGateway` 实现 `PaymentGateway`。
抽象类 `AbstractGateway`：包含公共逻辑——如签名算法、HTTP 调用封装、重试策略——避免在三个实现中重复。
选择理由：(a) 用 `PaymentGateway` 接口使调用方与具体支付商解耦——添加新支付渠道不需修改调用方；(b) 公共逻辑放在 `AbstractGateway` 减少代码重复——三个实现继承它并提供支付商特有的 `buildPaymentRequest` 和 `parseCallback` 的实现。

**例题2**：Go 中如何实现"多类型的打印"功能？

**解答**：使用 `fmt.Stringer` 接口（Go 标准库定义的 `String() string` 单方法接口）。
```go
type User struct { Name string; Age int }
func (u User) String() string {
    return fmt.Sprintf("User(%s, %d)", u.Name, u.Age)
}
// 任何实现了 String() string 的类型自动满足 fmt.Stringer 接口
fmt.Println(User{Name: "Alice", Age: 30})
```
这与 Java 中 `toString()` 重写的效果相同但接口满足是隐式的——User 不需要显式声明 `implements Stringer`。
