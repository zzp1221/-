## 核心定义

泛型编程（Generic Programming）是一种编程风格，其中算法和数据结构通过**类型参数**被编写为不依赖于具体类型的通用形式，随后在编译或运行时被实例化用于具体类型。泛型的核心价值是"将类型安全从具体类型提升到类型构造器级别"——编写一次处理任意类型的数据结构，而无需进行类型转换或放弃类型检查。

**不同语言的泛型实现**：

**Java 泛型**（类型擦除 Type Erasure）：泛型信息仅存在于编译时——编译器进行类型检查后所有泛型信息被擦除，生成的字节码中没有泛型类型参数。如 `ArrayList<String>` 和 `ArrayList<Integer>` 在运行时是同一个 `ArrayList` 类（raw type）。结果是：(a) 不能创建泛型数组（`new T[10]` 非法）；(b) 不能检查 `instanceof` 泛型（已擦除）；(c) 不能创建基本类型的泛型（int 需包装为 Integer）；(d) 编译器插入自动强制类型转换代码。
```java
List<String> list = new ArrayList<>();
list.add("hello");
String s = list.get(0); // 编译器插入 (String) 强制转换，擦除后 get 返回 Object
```

**C++ 模板**（单态化 Monomorphization / Template Instantiation）：模板是纯粹的编译时机制——编译器在使用模板的每个具体类型处生成完全独立的"副本"机器码。如 `vector<int>` 和 `vector<string>` 是两个完全独立的类，编译后产生两份代码。结果是：(a) 零运行时开销——无类型转换、无虚分派、编译器可为每种类型独立优化（如 `vector<bool>` 可用比特级压缩）；(b) 代码膨胀——大量使用的模板导致二进制文件增大（模板膨胀）；(c) 编译时间更长——编译器需展开、实例化、编译每一份实例；(d) 模板参数可以是任意值（非仅类型——如 `std::array<int, 1024>` 中的整数编译时维度）。

**C++20 Concepts**：约束参数类型的要求（类似 Rust Trait / Go Interface）——使模板错误信息从千行错误压缩为可读错误。`template<typename T> requires std::integral<T> or std::floating_point<T> T generic_sqrt(T x)` 的可读错误 vs 以前的"in instantiation of template error in line 42 ..."

**C# 泛型**（运行时具体化 Reified Generics）：在 CLR 层面保留泛型信息——`List<string>` 和 `List<int>` 在运行时是不同的类型。支持泛型类型判断（`is`、`as`）、类型参数在反射中可检查。C# 使用即时编译（JIT）——代码共享（`List<T>` 对引用类型 T 共享一份 JIT 后的代码，对值类型拆分为特定版本）。

**Rust 泛型**（单态化）：与 C++ 类似——编译器在每种具体类型的使用处生成独立代码。特化（Specialization）：允许为某特定类型提供优化实现。Rust 的泛型由 Trait bounds 约束——`fn foo<T: Display>(t: T) {}` 只能接受实现了 Display 的类型——编译时检查、零运行时开销、良好的错误信息。

**Go 泛型**（Go 1.18+, 基于约束的类型参数）：通过接口类型作为约束来实现类型参数化。使用方括号声明类型名字：`func Sum[T int | float64](vals []T) T { var total T; for _, v := range vals { total += v }; return total }`。Go 泛型编译时单态化，所有权衡就是编译时间和二进制大小的增加。

## 关键结论

- 泛型的核心价值是类型安全的复用——避免 Object/void* 级的手动类型转换（类型转换 ClassCastException 运行时暴露，而泛型编译时暴露）
- 类型擦除 vs 单态化是"共享代码减少内存 vs 为各类型优化性能"的经典权衡
- 泛型的最佳实践——"函数尽可能通用但不要超过需要的程度"（Rust 的 Trait 约束仅要求该有的操作）
- 泛型和接口/多态是互补的——泛型提供编译时多态（静态分派），接口/虚函数提供运行时多态（动态分派）

## 易错点

1. Java 的类型擦除导致"无法创建泛型数组"——`T[] t = new T[10]` 编译错误——替代方案：`(T[]) new Object[10]` 加 SuppressWarnings
2. C++ 模板导致巨大的错误信息——"In instantiation of ..." 后面跟着 100 行的展开信息——C++20 Concepts 改善了这一点
3. Go 泛型中类型约束定义在接口类型——限制多了使函数失去通用性，少了使类型推断失败
4. Java 泛型的 Wildcard（`? extends / ? super`）常用于协变/逆变但迷惑——PECS 规则：Producer Extends, Consumer Super

## 例题

**例题1**：用 Java 和 Rust 分别实现泛型的 `max` 函数（返回两值中较大的）。

**解答**：
Java: `public static <T extends Comparable<T>> T max(T a, T b) { return a.compareTo(b) > 0 ? a : b; }` — 类型擦除 + 虚方法调用 `compareTo` 有运行时开销。
Rust: `pub fn max<T: Ord>(a: T, b: T) -> T { if a > b { a } else { b } }` — 编译时单态化，但对每个 T 生成独立代码。对于 i32 调用 `max(3, 5)` 编译器内联了 > 运算符，零虚函数开销。

**例题2**：解释 Java 泛型中 `? extends T`（上边界通配符）和 `? super T`（下边界通配符）的使用场景。

**解答**：PECS 规则（Producer Extends, Consumer Super）：

上边界 (`? extends T`)：读取（生产）型——可以从集合中读取 T 或其子类元素。`List<? extends Number>` ——可以读 `Number n = list.get(0)` 但不能写 `list.add(5)` 因为实际类型可能是 `List<Integer>` 或 `List<Double>`，编译器无法确定 add 安全。
下边界 (`? super T`)：写入（消费）型——可以安全地将 T 或子类实例写入集合。`List<? super Integer>` ——可以 `list.add(5)` 但不能确定读出的类型 `Object o = list.get(0)` 因为实际类型可能是 `List<Number>` 或 `List<Object>`。

示例：`static <T> void copy(List<? super T> dest, List<? extends T> src)` — src 是"生产" T 型元素（读），dest 是"消费" T 型元素（写）——类型安全。
