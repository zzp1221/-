## 核心定义

序列化（Serialization / Marshalling）是将内存中的对象或数据结构转换为可存储、传输的格式的过程。反序列化（Deserialization / Unmarshalling）是将序列化后的数据还原为内存对象的过程。序列化是分布式系统、持久化、进程间通信的基础。

**序列化格式的分类**：

### 1. 文本格式（Human-Readable）

**JSON**（JavaScript Object Notation）：最通用的轻量数据交换格式。基本类型：string, number, boolean, null, object, array。优点：人类可读、语言无关、Web API 事实标准。缺点：无二进制支持（base64 编码增大 33%）、浮点数精度缺失（`3.141592653589793` 可能被截断）、无类型模式定义（schema-free）、冗余（key 在每个对象中重复出现）、解析较慢。

JSON Schema 约束定义了 JSON 的结构和类型——类似 XML Schema 的角色——用于 API 文档 OpenAPI/Swagger 中。

**XML**（Extensible Markup Language）：最早的数据交换标准。优点：有 Schema（XSD）验证、命名空间、内置注释——适合文档和数据并重的场景（如配置文件）。缺点：冗长（标签名成对出现）、解析开销高——现代 Web API 几乎被 JSON 取代，但在 SOAP、配置（Maven pom.xml, Android manifest）中仍有使用。

**YAML**（YAML Ain't Markup Language）：人类友好的配置格式——缩进决定层次、支持注释。作为配置语言广泛使用（Docker Compose, Kubernetes, Ansible, GitHub Actions）。作为数据交换较少使用（解析复杂、性能较低）。

### 2. 二进制格式（Machine-Efficient）

**Protocol Buffers**（Protobuf, Google）：定义 `.proto` schema 文件描述数据结构，编译器生成代码将结构化数据高效二进制序列化。字段由数字标签标识而非名称——紧凑、向后兼容（新字段只需新标签，旧代码忽略未知标签）。整数使用变长编码（Varints）——小整数仅用 1-2 字节。应用：gRPC的数据格式、Google 内部所有服务通信、微服务边界。

**MessagePack**：二进制 JSON 的直译——紧凑，但保持 JSON 的动态无 Schema 风格。比 JSON 小 30-50%，解析更快。适合需要 JSON 灵活性但更在意带宽和速度的场景（如游戏/移动端）。

**Avro**（Apache）：基于 Schema 的二进制序列化。Schema 随数据走或通过注册表管理（Schema Registry）。适合大数据/Hadoop 生态（Kafka 数据流序列化）——Schema 演变时对数据解码。

**Apache Thrift**（Facebook）：与 Protobuf 相似——定义 IDL + 编译器生成代码。支持多种数据结构和 RPC 框架（不仅是序列化）。与 Protobuf 的主要区别是没有 varint 编码（更大网络占用但可能更快编解码）。

**BSON**（Binary JSON）：MongoDB 的存储和传输格式——类似 JSON 但带额外类型（日期、二进制数据）。优化了遍历和索引（文档大小前缀允许快速跳过）。

### 序列化的关注点

**向前向后兼容**（Forward/Backward Compatibility）：
- 向后兼容：新代码可处理旧数据格式（新字段添加默认值）
- 向前兼容：旧代码不崩溃于新字段（Unknown fields handling）
Protobuf 和 Avro 在这两方面设计良好——旧程序忽略无法识别的字段（在 proto2 中默认为 unknown 保留，proto3 中丢弃但可配置保留）。

**安全性**：反序列化的输入不可信任——高知名度攻击：Java 反序列化漏洞（2015）、Python pickle RCE。规则：不反序列化不可信来源的序列化数据（或先用签名/签名验证后再反序列化）。

**性能对比**（近似级别）：Protobuf ≈ Avro ≈ MessagePack > JSON > XML > YAML。Protobuf 基准测试通常 2-10x 快于 JSON 解析且占用 3-10x 更少空间。

## 关键结论

- JSON 是通用互操作性的最低标准——几乎任何语言和平台都原生支持
- 服务间通信（微服务架构内部）应使用带 Schema 的格式（Protobuf/Avro）——带宽、解析速度和类型安全都远优于 JSON
- Schema 是反序列化格式的核心价值——自文档、编译时类型检查、版本兼容性验证
- 序列化是安全边界——不可信的序列化数据必须被验证或拒绝

## 易错点

1. JSON 数字精度——`9007199254740993` > JS Number MAX_SAFE_INTEGER、反序列化后精度丢失
2. 序列化的语言特定格式（Java `ObjectOutputStream`, Python `pickle`）仅有该语言可用——违背互操作性
3. Protobuf 字段编号修改——编号是二进制格式的 key，修改等于创建新字段（违反兼容）
4. JSON 中循环引用不可序列化——需打破循环或特殊编码（如 `@ref` id 机制）

## 例题

**例题1**：为何 gRPC 选用 Protobuf 而非 JSON 作为其数据格式？

**解答**：(a) 紧凑性——Protobuf 二进制编码不传输字段名（仅字段编号），对于 int 使用变长编码，体积远小于 JSON 减少网络占用；(b) 性能——Protobuf 编解码比 JSON 解析快 2-10x，尤其对高吞吐微服务至关重要；(c) Schema——`.proto` 文件提供类型安全、自动代码生成、明确的向后兼容策略；(d) gRPC 的 HTTP/2 + Protobuf 组合是"高性能微服务通信"的最佳实践——分帧、多路复用、双工通信配合高效载荷。

**例题2**：设计 API 序列化的版本演进策略——已有 `User {name, email}` 下一次版本要添加 `phone` 和移除 `name`。

**解答**：
Protobuf 方案：
```protobuf
message User {
    string name = 1 [deprecated = true];  // 保留字段编号，不再使用
    string email = 2;
    string phone = 3;  // 新增字段（使用新编号，旧客户端忽略）
}
```
- 旧客户端收到含 phone 字段的数据→忽略 phone（向前兼容）
- 新客户端收到含 name 字段的旧数据→忽略已弃用的 name（向后兼容）
- **关键**：**绝不在生产环境中删除字段的编号或改变现有字段的类型和编号**。仅 deprecate（标记废弃），永远不重用编号。这是 Protobuf 兼容性的核心法则。
