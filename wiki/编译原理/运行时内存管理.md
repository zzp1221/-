## 核心定义

编译器生成的代码在运行时需要管理两种内存区域：栈（Stack）和堆（Heap）。栈用于管理函数调用：每次函数调用创建一个栈帧（Stack Frame/Activation Record），函数返回时销毁。栈帧包含：返回地址、调用者的帧指针（动态链）、参数、局部变量、临时变量、保存的寄存器。栈帧的布局是编译时确定的（大小固定），使用栈指针SP和帧指针FP管理。堆用于动态内存分配：大小在编译时不确定的数据（如可变长度数组、对象实例）在堆上分配。堆管理方式：(1)显式管理（C/C++的malloc/free，程序员负责释放）；(2)自动管理（Java/Go的GC，运行时自动回收）；(3)区域推断（Rust的所有权系统，编译时确定释放点）。栈上分配比堆上分配快得多：栈分配只需移动SP指针（1条指令），堆分配需要查找空闲块（如buddy system、slab allocator）。内联（Inline）优化消除了函数调用的栈帧开销。

## 关键结论

- 栈帧大小在编译时确定（局部变量数量和大小已知），堆分配大小在运行时确定
- 栈溢出（Stack Overflow）通常由无限递归或过大的局部变量导致
- 堆碎片是显式内存管理的主要问题：频繁分配释放后堆中散布着无法使用的小空闲块
- 编译器的逃逸分析（Escape Analysis）可以将堆分配优化为栈分配：如果对象不逃逸出函数
- 某些语言（如C的alloca）在栈上动态分配，函数返回时自动释放，但有安全风险

## 易错点

1. 栈帧的"动态链"指向前一个栈帧（调用者），"静态链"指向静态外层函数的栈帧（嵌套函数）
2. 堆分配器的性能差异很大：jemalloc/tcmalloc比glibc的ptmalloc快数倍
3. 栈上分配的数据在函数返回后不可用：返回局部变量的指针是常见bug

## 例题

**例1：** 分析以下C代码的栈帧布局：
```c
int add(int a, int b) { int c = a + b; return c; }
int main() { int x = 1, y = 2; int z = add(x, y); return z; }
```

**解答：** main的栈帧：[返回地址(到OS)][旧FP][局部变量x=1][局部变量y=2][临时变量z(待赋值)]。调用add时：压入参数y(2)→压入参数x(1)→压入返回地址(到main中call指令的下一条)→压入旧FP→设置新FP=SP→移动SP分配局部变量c。add的栈帧：[返回地址][旧FP][参数a][参数b][局部变量c]。add返回：c=a+b=3，将结果放入寄存器eax，恢复SP和FP，跳转到返回地址。main中将eax赋值给z。注意：参数传递的顺序和方式（寄存器vs栈）取决于调用约定（cdecl/stdcall/System V AMD64 ABI等）。
