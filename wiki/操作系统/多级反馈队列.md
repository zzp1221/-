## 核心定义

多级反馈队列调度算法（MLFQ, Multilevel Feedback Queue）是现代操作系统中最通用、最复杂的调度算法之一。它将就绪队列按优先级分为多个级别，每个级别有自己的时间片大小（通常高优先级队列时间片短，低优先级队列时间片长）。基本规则：(1) 新进程进入最高优先级队列；(2) 调度时只当更高优先级队列为空时才处理低优先级队列（严格优先级）；(3) 若进程在当前队列用完时间片未完成，则降级到下一级队列（优先级降低，时间片增大）；(4) 若进程在时间片内因 I/O 阻塞，则通常保持在同级队列（或提升一级——奖励 I/O 密集型进程）；(5) 定期将所有进程提升到最高级（防止饥饿）。MLFQ 不需要预知执行时间，自然地区分了：短 I/O 密集型进程在高优先级快速完成（响应快），长 CPU 密集型进程沉到低优先级获得长 CPU burst（减少切换）。Windows 和 macOS 的调度器均基于 MLFQ 思想。

## 关键结论

- MLFQ 综合了 RR、优先级、SJF、FCFS 的优点
- 短作业在高级队列快速响应；长作业在低级队列获得长时间片以减少切换
- I/O 密集型进程留在高级队列；CPU 密集型进程逐步下沉
- 定期提升（Boost）防止饥饿：所有进程定期提升到最高优先级
- 不需要预知 CPU burst 长度，自适应特性强

## 易错点

1. "I/O 密集型进程留在高级队列"不是自动的——因为它在时间片用完前就因 I/O 阻塞了，所以不会降级，自然就留在高级队列
2. 定期提升（Boost）与给 I/O 进程提升是两个不同机制：前者防止 CPU 密集型进程饿死（定期全部提升），后者奖励交互性
3. MLFQ 中"反馈"的含义：根据进程的行为（CPU 使用量）来动态调整其优先级，让 OS 适应不同类型的任务

## 例题

**例1：** 为什么 MLFQ 比单纯的 RR 或 FCFS 更好地平衡了响应时间和吞吐量？

**解答：** MLFQ 对短作业（交互式/IO密集型）给予高优先级+短时间片，保证响应快；对长作业（CPU密集型）给予低优先级+长时间片，减少切换开销提高吞吐量。它根据进程实际行为动态调整优先级，实现了"对短作业好，对长作业不差"的效果。
