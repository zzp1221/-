## 核心定义

**读者-写者问题（Reader-Writer Problem）：** 数据文件可被多个读者进程同时读取，但写者进程必须独占访问。需保证：(1) 多个读者可同时读；(2) 写者在写时，读者和其他写者都不能访问；(3) 解决策略分为"读者优先"（读者到来直接读，可能导致写者饿死）和"写者优先"（有写者等待时新读者需排队）、公平策略（FIFO 顺序服务）。使用一个读写信号量 rw 和读者计数器 readcount + 互斥信号量 mutex 实现基本模型。增加写者计数器可实现写者优先。

**哲学家进餐问题（Dining Philosophers）：** 五个哲学家围着圆桌，每人面前一根筷子（共五根）。哲学家要么思考、要么进餐（需同时拿起左右两根筷子）。问题是避免死锁（所有哲学家同时拿起左筷子等待右筷子）和饥饿。解决方案包括：(1) 至多允许 4 个哲学家同时进餐；(2) 奇数哲学家先拿左筷子偶数先拿右筷子；(3) 仅当两边筷子都可用时才允许拿（同时拿）。

## 关键结论

- 读者写者：读写锁（rwlock）是标准解决方案，允许多读单写
- "读者优先"容易导致写者饥饿；"写者优先"容易导致读者饥饿
- 哲学家进餐问题的核心是避免死锁（循环等待）
- 解决方法（破坏循环等待条件）：限制同时进餐人数、非对称拿筷子、同时拿两根筷子
- 信号量数组是解决哲学家问题的标准工具
- 这些经典问题抽象了现实中数据库读写、资源分配等场景

## 易错点

1. 读者写者中的计数器 readcount 需要互斥保护：多个读者同时修改 readcount 会产生竞争条件
2. 哲学家进餐中的"同时拿两根筷子"实现：需要额外的 mutex 信号量保证"判断是否可拿->拿两根筷子"的原子性
3. 读者写者中 V(rw) 应该在 readcount == 0 时由最后一个读者执行，而非每个读者都执行

## 例题

**例1：** 为什么哲学家进餐问题中，让所有奇数号哲学家先拿左筷子、偶数号先拿右筷子可以避免死锁？

**解答：** 此方案打破了循环等待（Circular Wait）。奇数哲学家(1,3,5)拿左->右，偶数(2,4)拿右->左。比如 1 号和 2 号竞争 1 号右/2 号左这根筷子——1 号把它当右筷，2 号把它当左筷，不会形成所有哲学家都拿着左筷等待右筷的循环等待链。

## 代码示例

```cpp
// 读者-写者问题（读者优先）
sem_t rw, mutex;
int readcount = 0;

void* reader(void* arg) {
    sem_wait(&mutex);
    if (++readcount == 1)
        sem_wait(&rw);  // 第一个读者锁住写者
    sem_post(&mutex);
    // 进行读操作
    sem_wait(&mutex);
    if (--readcount == 0)
        sem_post(&rw);  // 最后一个读者释放写者
    sem_post(&mutex);
}

void* writer(void* arg) {
    sem_wait(&rw);   // 写者独占
    // 进行写操作
    sem_post(&rw);
}

// 哲学家进餐（限制最多 4 人同时进餐）
sem_t chopstick[5];
sem_t limit(4);  // 限制同时进餐人数

void* philosopher(void* num) {
    int i = *(int*)num;
    while (true) {
        think();
        sem_wait(&limit);          // 限制人数
        sem_wait(&chopstick[i]);   // 拿左筷子
        sem_wait(&chopstick[(i+1)%5]); // 拿右筷子
        eat();
        sem_post(&chopstick[i]);
        sem_post(&chopstick[(i+1)%5]);
        sem_post(&limit);
    }
}
```
