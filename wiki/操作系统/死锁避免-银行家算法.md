## 核心定义

银行家算法（Banker's Algorithm）是由 Dijkstra 提出的经典死锁避免算法。其核心思想是：在每一次资源分配前，通过"安全性算法"计算分配后系统是否仍处于安全状态（Safe State）。安全状态是指存在一个进程安全序列（P1, P2, ..., Pn），使得对于 Pi 所需的资源可用量都能被当前系统剩余资源 + 前 j 个进程释放的资源总和满足。若存在这样的序列，分配后仍安全，可进行分配；否则拒绝分配（即使资源当前可用），进程需等待。银行家算法需要维护的数据结构：Available（各类资源当前可用数量）、Max（每个进程对各资源的最大需求量）、Allocation（当前分配量）、Need（Need = Max - Allocation，还需的资源量）。

银行家算法的安全性检查步骤：设 Work = Available, Finish[i] = false；(1) 找一个未完成的进程 Pi 满足 Need[i] <= Work；(2) 假设 Pi 获得资源执行完并释放：Work += Allocation[i], Finish[i] = true；(3) 重复直到所有 Finish[i] 都是 true（安全）或找不到满足条件的 Pi（不安全）。

## 关键结论

- 银行家算法通过动态预判安全状态来避免死锁
- 核心思想：分配前做"试分配"并用安全性算法检查是否仍安全
- 每一类资源有多个实例（与仅单实例不同），能处理多实例资源
- 算法需要知道进程的最大资源需求，这在实际中常难以事先预知
- 时间复杂度 O(m*n^2)（m 资源类型数，n 进程数），开销较大

## 易错点

1. 银行家算法只能避免死锁，不能解决死锁：已经死锁的系统再用银行家算法已经没有意义
2. Need[i] <= Work 要求 Need 中每一类资源都 <= Work 对应类别（不能只看总和）
3. 安全状态不等于不死锁：当前安全不代表永远不死锁；不安全状态也不等于已经死锁（只是有死锁风险）

## 例题

**例1：** 假设系统中有 A、B、C 三类资源，当前 Available = (3,3,2)。5 个进程的 Allocation 和 Max 如下，问系统是否安全？

Allocation: P0(0,1,0), P1(2,0,0), P2(3,0,2), P3(2,1,1), P4(0,0,2)
Max: P0(7,5,3), P1(3,2,2), P2(9,0,2), P3(2,2,2), P4(4,3,3)

**解答：** Need = Max-Allocation: P0(7,4,3), P1(1,2,2), P2(6,0,0), P3(0,1,1), P4(4,3,1)。找安全序列：Work=(3,3,2)。P1 Need(1,2,2)<=Work，取 P1；Work=(5,3,2)。P3 Need(0,1,1)<=Work，Work=(7,4,3)。P4 Need(4,3,1)<=Work，Work=(7,4,5)。P2 Need(6,0,0)<=Work，Work=(10,4,7)。P0 Need(7,4,3)<=Work，Work=(10,5,7)。全部完成，安全序列为 P1,P3,P4,P2,P0。
