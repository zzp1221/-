## 核心定义

死锁检测与恢复是处理死锁的另一策略，允许系统进入死锁状态，然后检测并恢复。检测方法分为两类：(1) 每种资源类型只有一个实例——用等待图（Wait-For Graph，节点是进程，边 P_i -> P_j 表示 P_i 等待 P_j 释放资源）。周期性地调用环检测算法（DFS 检测图中是否有环），若发现环则存在死锁。(2) 每种资源类型有多个实例——类似银行家算法，使用 Available、Allocation、Request 矩阵，找一个可以完成的进程序列直到所有进程都能完成或有进程无法完成。(3) 检测时机：可以周期检测（定时器）或在资源利用率下降/进程阻塞过多时触发。

恢复方法有三种：(1) 进程终止——终止一个或多个死锁进程，打破循环等待链。选择"代价最小"的进程（如优先级最低、运行时间最短等）；(2) 资源抢占——从进程中抢占资源分配给死锁进程，直到打破死锁。需要考虑：选择哪个进程（通常选拥有最少资源/容易恢复的）、如何回滚（需保存检查点 Checkpoint）、防止饥饿；(3) 进程回滚——将进程回滚到死锁发生前的安全检查点。

## 关键结论

- 等待图法适用于各类资源仅有一个实例：有环则死锁
- 银行家算法式检测法适用于多实例：类似安全性算法
- 进程终止是最简单直接的恢复方式
- 检测频率越高则死锁持续时间越短，但 CPU 开销越大
- 实际 OS（如 Linux、Windows）大多采用"鸵鸟策略"（忽略死锁），由应用程序自行处理

## 易错点

1. 等待图中有环 != 每种资源多实例时有死锁：多实例资源系统中，资源的 Request 图中有环不一定死锁（如果某个资源可以其他方式释放）
2. 检测频率的权衡：检测太频繁浪费 CPU；检测太不频繁会导致大量进程处于死锁状态
3. 进程终止后资源的重新分配可能导致另一个死锁：终止死锁进程后需要重新检查安全状态

## 例题

**例1：** 为什么大多数通用操作系统（Windows、Linux）不实现死锁检测和恢复机制？

**解答：** 检测和恢复开销大，会影响系统性能。大多数应用层的死锁可以通过开发者遵守编程规范来避免。OS 把死锁处理的责任交给应用开发者，"鸵鸟策略"最为经济实用。

## 代码示例

```cpp
// 等待图死锁检测（单实例资源/DFS 检测环）
#include <vector>
using namespace std;

bool hasCycle(vector<vector<int>>& graph, vector<int>& state, int u) {
    state[u] = 1;  // 访问中
    for (int v : graph[u]) {
        if (state[v] == 1) return true;  // 发现环
        if (state[v] == 0 && hasCycle(graph, state, v))
            return true;
    }
    state[u] = 2;  // 已处理
    return false;
}

bool deadlockDetected(vector<vector<int>>& waitGraph, int n) {
    vector<int> state(n, 0);  // 0未访 1访问中 2完成
    for (int i = 0; i < n; i++)
        if (state[i] == 0 && hasCycle(waitGraph, state, i))
            return true;
    return false;
}
```
