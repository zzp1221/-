## 核心定义

流水线冒险（Pipeline Hazards）是指阻碍下一条指令在下一个时钟周期顺利执行的各种情况，导致流水线必须停顿（Stall）或产生气泡（Bubble）。冒险分为三种类型：结构冒险（Structural Hazard）——硬件资源冲突，当两条指令同时需要同一硬件资源而该资源无法被同时使用时发生，例如单端口存储器同时面对 IF 的取指和 MEM 的数据访问；数据冒险（Data Hazard）——指令间的数据依赖关系导致流水线重叠执行出错，分三种子类：RAW（Read After Write，真依赖，后一条需读前一条写的结果）、WAR（Write After Read，反依赖，后一条需写但前一条还需读旧值）、WAW（Write After Write，输出依赖，两条指令写同一寄存器）；控制冒险（Control Hazard）——分支/跳转指令使 PC 无法在 IF 阶段确定下一条指令地址，因为分支条件要等到 EX 阶段才计算出，期间已取入的指令可能作废。

## 关键结论

- RAW 依赖是最常见的数据冒险，必须通过停顿或转发（Forwarding）解决
- WAR 和 WAW 在简单按序流水线中不会发生，仅在乱序执行中出现（需寄存器重命名解决）
- 控制冒险的解决策略：分支预测（动态/静态）、延迟槽（Delay Slot）、尽早计算分支条件
- 结构冒险在现代 CPU 中已基本消除（I-Cache 和 D-Cache 分离、寄存器堆多端口）
- 流水线互锁（Pipeline Interlock）：硬件自动检测冒险并插入停顿（气泡），保证正确性

## 易错点

1. RAW/WAR/WAW 的读写方向别搞反：RAW = 写后读（先写后读），WAR = 读后写（先读后写），WAW = 写后写。
2. 数据冒险的"距离"决定是否可转发解决：若两条指令相距 1 条（EX->EX 转发），可在不损失性能下解决；相距 > 1 可能需要停顿或多重转发。
3. 分支冒险中"冲刷"（Flush）指令的代价被严重低估：将错误路径上已取入流水线的指令变为 NOP（气泡），对于深度流水线代价很大。

## 例题

**例题1：** 以下指令序列是否有数据冒险？如有，是何类型？如何解决？
```
ADD R1, R2, R3
SUB R4, R1, R5
```
**解答：** SUB 的源操作数 R1 依赖于 ADD 的结果。RAW 冒险：ADD 在 WB 阶段才写回 R1，但 SUB 在 ID 阶段就需要 R1。解决：通过转发（Forwarding），ADD 的 EX/MEM 阶段结果（ALU 输出）直接旁路送给 SUB 的 EX 阶段输入，无需等待 WB，零停顿。

**例题2：** 结构冒险示例：单端口存储器的指令和数据访问冲突。如何解决？

**解答：** IF 阶段（取指令）与 MEM 阶段（LW/SW 指令的访存）可能在同一周期访问存储器。解决方案：(1) 分离 I-Cache 和 D-Cache（哈佛架构思路）。(2) 若不得已共享，插入停顿（Stall）让 IF 等待 MEM 完成，产生气泡。
